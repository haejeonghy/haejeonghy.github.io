→ RBAC은 쿠버네티스에서 인증된 사용자의 **자원 접근 권한을 역할(Role) 기반으로 제어**하며, 멀티테넌시 환경에서 각 테넌트(조직/네임스페이스)별 **안전하고 분리된 권한 관리**를 가능하게 하는 핵심 보안 메커니즘

# 🔐 인증, 인가, 멀티테넌시, RBAC 요약 정리

## 1. 기본 개념 정리

| 개념 | 설명 |
| --- | --- |
| **인증 (Authentication)** | 사용자가 누구인지 확인하는 절차 (예: 로그인) |
| **인가 (Authorization)** | 인증된 사용자가 어떤 권한을 가졌는지 확인하는 절차 |
| **멀티테넌시 (Multi-tenancy)** | 하나의 소프트웨어 인스턴스가 여러 사용자 또는 조직을 각각 분리된 고객처럼 동시에 서비스하는 아키텍처 구조 |

---

## 2. RBAC (Role-Based Access Control)

### ✅ 구성 요소

| 구성 요소 | 설명 |
| --- | --- |
| **Role** | 특정 네임스페이스의 리소스에 대한 권한 정의 |
| **ClusterRole** | 네임스페이스 범위를 초월한 리소스 권한 정의 (예: 모든 네임스페이스의 파드 조회, 노드 접근 등) |
| **RoleBinding** | Role을 특정 사용자, 그룹, 또는 ServiceAccount에 연결 |
| **ClusterRoleBinding** | ClusterRole을 네임스페이스 상관없이 사용자에게 부여 |
| **ServiceAccount** | 파드가 API 서버에 접근할 때 사용하는 계정 |
| **컨텍스트(Context)** | 현재 사용할 네임스페이스, 클러스터, 사용자 정보를 정의 |

---

## 3. 실습 흐름 요약

### 1. wordpress 네임스페이스에 전체 권한을 주는 Role 생성

```yaml
kind: Role
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: wordpress-namespace-full-access
  namespace: wordpress
rules:
- apiGroups: ["", "extensions", "apps"]
  resources: ["*"]
  verbs: ["*"]
- apiGroups: ["batch"]
  resources:
    - jobs
    - cronjobs
  verbs: ["*"]

```

---

### 2. RoleBinding 생성

```yaml
kind: RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: wordpress-namespace-full-access-rb
  namespace: wordpress
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: wordpress-namespace-full-access
subjects:
- kind: ServiceAccount
  name: wordpress-sa
  namespace: wordpress

```

---

### 3. ServiceAccount 생성

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: wordpress-sa
  namespace: wordpress

```

---

### 4. 확인 절차

- `kubectl get rolebinding -n wordpress`
    
    → RoleBinding이 제대로 생성되었는지 확인
    
- `kubectl get sa -n wordpress`
    
    → ServiceAccount 존재 여부 확인
    
- `kubectl describe rolebinding wordpress-namespace-full-access-rb -n wordpress`
    
    → Role과 ServiceAccount가 올바르게 연결되었는지 확인
    
- 서비스어카운트 토큰 수동 생성 후 `kubeconfig`에 사용자로 등록하여 테스트
- default 네임스페이스에서 `권한 없음` 확인 → 인가(RBAC)가 정확히 작동하는지 검증

## 실습

1. wordpress 네임스페이스의 전체 권한을 가지는 Role 생성
    
    ```yaml
    # wordpress 네임스페이스에 대한 전체 리소스 접근 권한을 부여하는 Role 정의
    kind: Role  # 네임스페이스 범위 내 권한을 정의하는 리소스
    apiVersion: rbac.authorization.k8s.io/v1
    metadata:
      name: wordpress-namespace-full-access  # 이 Role의 이름
      namespace: wordpress  # 이 Role이 적용될 네임스페이스 (필수)
    
    rules:
      # 첫 번째 규칙: 기본 API 그룹("", 즉 core)과 extensions/apps 그룹의 모든 리소스에 대해 모든 작업 허용
      - apiGroups: ["", "extensions", "apps"]  # 대상 API 그룹들
        resources: ["*"]  # 모든 리소스 유형 (pods, services, deployments 등)
        verbs: ["*"]  # 모든 작업 허용 (get, list, watch, create, delete 등)
    
      # 두 번째 규칙: batch API 그룹의 jobs, cronjobs 리소스에 대해 모든 작업 허용
      - apiGroups: ["batch"]
        resources:
          - jobs
          - cronjobs
        verbs: ["*"]  # 모든 작업 허용
    
    ```
    
2. full namespace 파일 apply 해서 role 생성
3. role을 적용하려면 RoleBinding 필요
    
    ```yaml
    # wordpress 네임스페이스 내 Role을 특정 ServiceAccount에 바인딩하는 RoleBinding 정의
    kind: RoleBinding  # 네임스페이스 범위 내에서 Role을 바인딩하는 리소스
    apiVersion: rbac.authorization.k8s.io/v1
    metadata:
      name: wordpress-namespace-full-access-rb  # RoleBinding의 이름
      namespace: wordpress                      # 적용할 네임스페이스
    
    roleRef:  # 어떤 Role을 바인딩할 것인지 지정
      apiGroup: rbac.authorization.k8s.io       # Role 리소스가 속한 API 그룹
      kind: Role                                # 바인딩 대상 리소스 종류 (Role)
      name: wordpress-namespace-full-access     # 연결할 Role 이름
    
    subjects:  # 실제 권한을 부여할 주체 목록
    - kind: ServiceAccount                      # 주체의 종류 (사용자, 그룹, SA 중 SA 선택)
      name: wordpress-sa                        # 권한을 부여할 ServiceAccount 이름
      namespace: wordpress                      # ServiceAccount가 속한 네임스페이스
    
    ```
    
4. RoleBinding 생성
5. 서비스 어카운트 생성
    
    ```yaml
    apiVersion: v1
    kind: ServiceAccount
    metadata:
      name: wordpress-sa
      namespace: wordpress
    
    ```
    
6. 롤 바인딩과 연결된 서비스 어카운트 리스트 확인하기
7. 연결하면 서비스 어카운트 토큰 생성 → kubeconfig 설정 파일의 사용자 토큰 정보에 등록하여 사용자와 서비스 어카운트 연결
    1. 자동으로 생성되지 않아 토큰 생성
8. 허용한 namespace가 아닌 default에선 권한 제한된 거 확인
