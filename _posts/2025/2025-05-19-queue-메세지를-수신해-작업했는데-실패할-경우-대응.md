# queue 메세지를 수신해 작업했는데 실패할 경우 대응
Created: 2025년 5월 19일 오후 1:32
Tags: 설계, 아키텍처
보관소: No
최종 편집 일시: 2026년 3월 14일 오전 1:08

# 배경

1. 사용자가 기관 자산 삭제 요청을 했을 경우 해당 기관의 자산, 내역 데이터를 전부 삭제해야 함
2. mydata에서 queue로 특정 종류의 자산을 삭제하라는 메세지를 발행
3. 자산 서버에서 메세지를 수신했고 삭제 작업 실행
4. 이때 메세지를 소모했는데 삭제가 실패함 
5. 다른 서버는 성공함
6. retry를 어떻게 할 것이냐가 중요한 이슈

# 해결 방안

1. 삭제 대상 DB, redis에 넣고 관리, 주기적으로 배치를 돌려 다시 삭제 시도 가능
    1. retry count, asset pk id, 
    2. 장점
        1. retry가 쉬움 
        2. 삭제 주체(서비스 오너)가 에러 핸들링, 재시도 배치 관리 쉬움 
        3. 중복 삭제 시도 하지 않을 수 있음
        4. 실패한 원인 저장 가능
    3. 단점
        1. 별도의 DB 관리 운영 리소스 필요
        2. 배치 구현 필요 

# 피드백

1. 큐를 사용해서 넣기? 재시도 시점? 
    1. retry 큐로 하면 관리가 힘듬 
    2. 실행하는 시점이 필요함 
    3. 관리의 용이성이 떨어짐
2. rabbitMQ 같은건 터질 수 있음 
3. SQS도 싱글이기때문에 멀티로 받는건 불안 
4. DL queue
