## **📌 Backpressure 개념 정리**

### **1. Backpressure란?**

> **Backpressure(역압)**는 **생산자(Producer)**가 데이터를 너무 빠르게 보내고, **소비자(Consumer)**가 이를 제때 처리하지 못할 경우, 생산자의 전송 속도를
> 
> 
> **자동으로 조절하거나 멈추게 하는 흐름 제어(flow control) 메커니즘**
> 

---

### **2. 왜 필요한가?**

- 소비자가 감당 못하면 시스템 전체에 **병목, 메모리 폭주(OOM), 데이터 유실** 등이 발생할 수 있음
- 안정성, 성능, 신뢰성을 보장하기 위해 필요

---

### **3. 실제 예시 (비유)**

### **🛒 마트 계산대 비유**

- 고객(Producer)이 빠르게 물건을 가져오고, 계산대 직원(Consumer)이 천천히 계산할 경우 줄이 길어짐
- 계산대에서 “입장 제한”하거나 “멈춰주세요”라고 하면 → Backpressure 작동

---

### **4. Go에서의 Backpressure 예시**

```
ch := make(chan int, 2) // 버퍼 크기 2

ch <- 1
ch <- 2
ch <- 3 // ❗ 여기서 블로킹 발생 (소비자가 꺼낼 때까지 대기)
```

- ch <- 3은 **버퍼가 가득 찼기 때문에** 블로킹 상태에 빠짐
- 에러가 나는 것이 아니라 **생산자 고루틴이 멈춰서 기다림**
- 이 자체가 Backpressure 작용의 예

---

### **5. Backpressure가 자주 사용되는 곳**

| **분야** | **설명** |
| --- | --- |
| TCP | 수신 버퍼가 꽉 차면 ACK 신호로 송신 속도 제어 |
| gRPC Streaming | 서버/클라이언트 간 메시지 흐름 조절 |
| Kafka | Consumer가 느리면 메시지 처리 지연 발생 가능 |
| Go Channel | 버퍼가 다 차면 send가 블로킹됨 |
| Reactive Streams | 소비자가 요청한 만큼만 생산자가 전송 (pull 방식) |

---

### **6. 구현 방식**

| **방식** | **설명** |
| --- | --- |
| Pull 기반 | 소비자가 필요할 때 요청함 (ex. Reactive Streams) |
| Push + 제어신호 | 생산자가 푸시하다가 멈추라는 신호를 받음 |
| 버퍼 + 블로킹 | Go 채널처럼 버퍼가 다 차면 자동으로 멈춤 |

---

## **✅ 요약**

- **정의**: 소비자가 느릴 때 생산자의 전송 속도를 조절하는 메커니즘
- **목적**: 데이터 유실/과부하 방지, 시스템 안정성 유지
- **Go에서**: 채널의 버퍼가 다 차면 생산자가 자동으로 멈춰 대기함
- **특징**: 에러는 발생하지 않으며, 자연스럽게 흐름을 제어함
