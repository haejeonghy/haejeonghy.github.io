## **📘 도메인 주도 설계(DDD)에서의 Aggregator (정확한 용어: Aggregate)**

### **✅ 개념 정의**

> Aggregate
> 
> 
> **일관성 있는 상태를 유지하는 객체들의 집합**
> 

> 그 집합의 **중심이 되는 객체(Aggregate Root)**를 통해서만 외부에서 접근 가능하게 설계됩니다.
> 

---

### **🧱 핵심 구성 요소**

| **구성 요소** | **설명** |
| --- | --- |
| **Aggregate Root** | 외부에서 접근 가능한 **최상위 도메인 객체** |
| **Entities** | Aggregate 내부에서 루트가 관리하는 **하위 객체들** |
| **Boundary** | Aggregate는 **트랜잭션 경계 단위**로 사용됨 (ACID 보장 단위) |

---

### **📌 왜 필요한가?**

- 하나의 트랜잭션에서 다뤄야 할 **일관성 있는 도메인 단위**를 정의
- 외부에서 내부 객체를 직접 건드리지 못하게 하여 **불변성과 일관성 유지**
- 복잡한 도메인 로직을 **모듈화**하고 유지보수를 용이하게 함

---

### **✅ 예시: 주문(Order) 시스템**

```
type Order struct {
  ID     string
  Items  []OrderItem
  Status string
}

type OrderItem struct {
  ProductID string
  Quantity  int
}
```

| **역할** | **설명** |
| --- | --- |
| Order | Aggregate Root |
| OrderItem | 내부 엔티티 (직접 접근 불가) |

→ Order를 통해서만 OrderItem을 추가, 삭제, 수정 가능하게 만들어야 함

→ 주문 상태(Status)를 바꿀 때, **항목 전체와 함께 처리**해야 일관성 유지됨

---

### **💡 Aggregate가 수행하는 “Aggregator” 역할**

- **여러 객체의 상태를 모아서**
- **하나의 일관된 도메인 상태로 관리**
- 외부에는 오직 **Aggregate Root만 노출**
- 내부의 복잡한 관계를 캡슐화하고, 도메인 규칙을 중앙 집중적으로 처리

---

### **🚨 실무에서의 주의사항**

| **주의점** | **이유** |
| --- | --- |
| Aggregate 크기를 너무 크게 잡지 말 것 | 하나의 트랜잭션 단위이므로 성능, 복잡성에 영향 |
| 외부에서 내부 엔티티 직접 조작 금지 | 일관성 깨짐 위험 |
| 가능한 한 Root를 통해서만 상태 변경 | 도메인 규칙 적용 일관화 |

---

### **✅ 요약**

| **항목** | **설명** |
| --- | --- |
| 목적 | 도메인 객체들을 묶어 일관된 상태 관리 |
| 중심 개념 | Aggregate Root + 내부 엔티티 |
| 특징 | 캡슐화, 트랜잭션 경계, 도메인 규칙 통제 |
| 사용 예시 | 주문(Order) + 주문 항목들, 게시글(Post) + 댓글들 |
