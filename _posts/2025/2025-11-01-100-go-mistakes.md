# 2.3 init 함수를 잘못 사용하지 마라.

## 2.3.1 개념

init함수는 패키지가 초기화 될 때 실행된다. 한 패키지에 다수의 init() 함수가 있으면 파일의 알파벳 순서대로 실행된다. 

init 함수는 직접 호출할 수 없다. 

```go
func main() {
	init() //호출 불가 선언되지 않은 컴파일 에러 ㅈ
}
```

## 2.3.2 init 함수를 사용해야 하는 경우

- init 함수는 에러 관리 기능이 부족하다. 에러를 리턴하지 않기 떄문에 에러 발생을 알리는 유일한 방법은 panic 뿐이다.
- 테스트 하기 힘들다. init 함수 때문에 단위 테스트 코드를 작성하기가 힘들다.
- 정적 함수 설정을 할 수 있다. 에러가 발생할 일이 없는 경우에 사용한다.

```go
func init() {
	redirect := func(w http.ResponseWriter, r *http.Request) {
		http.Redirect(w, r, "/", http.StatusFound)
	}
	http.HandleFunc("/blog", redirect)
}
```

# 2.5. 인터페이스 오염을 조심하라

→ 불필요한 추상화로 코드가 복잡해져서 이해하기 힘들게 되는 현상

## 2.5.1 개념

인터페이스는 오브젝트의 동작을 표현하는 수단을 제공한다. 인터페이스는 오브젝트가 공통으로 구현할 추상화를 정의한다. 

io.Reader와 io.Writer의 예시가 있는데, io.Reader는 데이터 소스에서 데이터를 읽고 io.Writer는 타깃에 데이터를 쓴다. 

인터페이스의 구현을 DB에 write하도록 커스텀 하더라도 사용하기 쉽고 테스트 코드를 작성하기도 쉽다. 

인터페이스에 메서드가 늘어날 수록 재사용성은 떨어질 수 있다. 

인터페이스와 조합하여 상위 수준의 추상화를 정의할 수도 있다. io.ReadWriter의 예시

## 2.5.2 인터페이스를 사용해야 하는 경우

인터페이스 유스케이스

- 공통 동작
    - 공통 동작을 하는 인터페이스를 뽑아내서 사용
- 결합 분리
    - 구체적인 구현 대신 추상화된 코드를 작성하면 나중에 구현을 바꾸더라도 앞서 작성한 코드를 고치지 않아도 된다.
    - 단위 테스트와 연계할 수 있다.
    
    ```go
    type CustomerService struct {
    	store mysql.Store // 이 타입을 mysql.Store로 선언하면 통합 테스트를 작성하여 mysql을 띄워야 하나 인터페이스로 생성하면 mock을 사용 가능
    }
    
    func (cs CustomerService) CreateNewCustomer(id string) error {
    	customer := Customer{id: id}
    	return cs.store.StoreCustomer(customer)
    }
    ```
    
- 동작 제한

구문을 강제하는 등 여러 이유에 따라 타입에 대한 동작을 제한하는데 사용할 수 있다. 

```go
type IntConfig struct {}

func (c *IntConfig) Get() int {
	// 설정 가져오기
}

func (c *IntConfig) Set(value int) {
	// 설정 업데이트 하기
}

// 설정 업데이트는 하면 안 될때 
type intConfigGetter interface {
	// get 만 명시하면 set은 못함
	Get() int
}

type Foo Struct {
	threshold intConfigGetter
}

func NewFoo(threshold intConfigGetter) Foo {
	return Foo{threshold: threshold}
}

func (f Foo) Bar() {
	threshold := f.threshold.Get()
}
```

## 2.5.3 인터페이스 오염

인터페이스는 추상화를 정의하기 위한 것이다. 추상화는 찾아내야지 창조하면 안 된다. 특별한 이유 없이 추상화 코드를 새로 만들면 안 된다. 

인터페이스 남용으로 발생하는 가장 대표적인 문제는 코드 흐름이 매우 복잡해진다는 것이다. 

# 2.6 제공자 측에 인터페이스를 두지 마라

- 제공자측 구현 (producer side): 구체적인 구현이 있는 패키지에 인터페이스 정의
- 사용자측 구현 (consumer side): 인터페이스를 사용하는 외부 패키지에 정의

고 언어에서 인터페이스는 암묵적으로 적용되기 때문에 추상화를 찾아야지 창조하면 안된다는 접근 방법을 따라야 한다. 추상화가 필요하다고 판단하고 필요에 맞는 추상화 수준을 결정하는 것은 사용자이다. 

```go
// 요건 제공자측 구현
// store/order_impl.go
package store

// store 패키지 안에 있는 구체 구현체

// InMemoryOrderStore는 OrderSaver, OrderReader 인터페이스를 암묵적으로 구현
type InMemoryOrderStore struct {
    orders map[string]Order
}

func NewInMemoryOrderStore() *InMemoryOrderStore {
    return &InMemoryOrderStore{
        orders: make(map[string]Order),
    }
}

// Save는 OrderSaver 인터페이스의 Save 메서드를 구현
func (s *InMemoryOrderStore) Save(o Order) error {
    if _, exists := s.orders[o.ID]; exists {
        return fmt.Errorf("order %s already exists", o.ID)
    }
    s.orders[o.ID] = o
    return nil
}

// Get은 OrderReader 인터페이스의 Get 메서드를 구현
func (s *InMemoryOrderStore) Get(id string) (Order, error) {
    o, ok := s.orders[id]
    if !ok {
        return Order{}, fmt.Errorf("order %s not found", id)
    }
    return o, nil
}

// 사용자측 구현

// api/order.go
package api

import "cafe/store"

// 소비자 쪽에서 필요한 기능만 골라 interface 정의
type OrderSaver interface {
    Save(store.Order) error
}

type OrderReader interface {
    Get(id string) (store.Order, error)
}

// 핸들러에서 인터페이스로 의존
type OrderService struct {
    saver  OrderSaver
    reader OrderReader
}

func NewOrderService(saver OrderSaver, reader OrderReader) *OrderService {
    return &OrderService{saver: saver, reader: reader}
}

func (s *OrderService) Create(o store.Order) error {
    return s.saver.Save(o)
}

func (s *OrderService) Fetch(id string) (store.Order, error) {
    return s.reader.Get(id)
}
```

- 여러 소비자에서 거의 동일한 동작을 요구한다면, 소비자 패키지마다 인터페이스를 중복 정의하기보다는 공통의 인터페이스를 정의할 수 있는 별도의 “포트(port)” 혹은 “어댑터 레이어”를 만드는 것을 고려해볼 수 있습니다.
- 처음부터 모든 중복을 제거하려고 하기보다는, 실제 사용 패턴을 보면서 추상화를 발전시키는 방식이 실용적입니다.

# 2.7 인터페이스를 리턴하지 마라

일반적으로 인터페이스를 리턴하면 유연성이 떨어진다. 모든 클라이언트가 특정한 추상화만 사용하기 때문이다. 

> 자신이 하는 일에는 엄격하고 다른 사람으로부터 받을 때는 관대하라. TCP
> 

인터페이스보다는 구조체로 리턴한다. 

가능하면 인터페이스로 받는다.

```go
    return &dbOrderStore{ /* db connection */ }

// cafe/storage/order.go
```

# 2.8 any는 아무것도 알려주지 않는다

interface{}를 모두 any로 사용 가능 (alias) any는 모든 값 타입을 담을 수 있다. 

```go
package store

type Customer struct {}

type Contract struct {}

type Store struct {}

func (s *Store) Get(id string) (any, error) {}
```

any 타입으로 리턴하거나 인수를 받으면 사용자가 얻을 수 있는 정보는 없다. 엉뚱한 데이터 타입으로 호출해도 컴파일 단계에서 걸러낼 수 없다. 

any를 사용하면 좋을 때

- encoding/json 패키지 → 모든 타입으로 마샬링 할 수 있으므로 any 인수를 받는다.
- database/sql 패키지 → 매개변수화 된 질의 파라미터는 어떤 타입으로든 나올 수 있으므로 any 인수를 받는다.
    
    ```go
    func (c *Conn) QueryContext(ctx context.Context, query string, args ...any) (*Rows, error) {}
    ```
    

any는 정말 모든 타입을 받거나 리턴해야 할 경우에 사용하고, 그렇지 않을 경우 약간의 중복이 발생하더라도 표현력 측면에서 나을 수 있다. 

# 2.9 제네릭이 필요한 시점을 파악하라

제네릭은 나중에 구체적인 타입이 필요한 시점에 지정해서 인스턴스로 만들 때 사용한다. 

## 2.9.1 개념

```go
// 제네릭을 명시해 다양한 케이스의 맵의 키를 추출할 수 있도록 함
// getKeys를 호출할 때 타입 인수를 전달하는 작업이 컴파일 시간에 철됨
func getKeys[K comparable, V any](m map[K]V) []K {
	var keyes []K
	for k := range m {
		keys = append(keys, k)
	}
	return keys 
}

type customContraint interface {
	~int | ~string // ~를 사용하면 내부가 int, string인 모든 타입으로 제약 iota 등
}

// k의 타입을 int나 string으로 제한할 수 있다 
func getKeys[k customContraint, V any](m map[K]V) []K {}

// 제네릭으로 데이터 구조에도 사용 가능
type Node[T any] struct {
	Val T
	next *Node[T]
}

func (n *Node[T]) Add(next *Node[T]) {
	n.next = next
}
```

## 2.9.2 주요 사용 사례와 잘못 사용하는 경우

- 사용하면 좋은 경우
    - 데이터 구조: 바이너리 트리, 링크드 리스트, 힙 등 데이터 구조를 구현할 때 원소 타입을 추출하는데 제네릭을 이용하면 좋다
    - 슬라이스, 맵, 채널을 다루는 함수
        
        ```go
        func merge[T any](ch1, ch2 <- chan T) <-chan T {}
        ```
        
    - 타입 대신 동작을 추출할 때
        
        ```go
        type Interface interface {
        	Len() int
        	Less(i, j int) bool
        	Swap(i, j, int) 
        }
        
        type SliceFn[T any] struct {
        	S []T
        	Compare func(T, T) bool
        ```

# 2.10 타입 임베딩을 제대로 하라

고 언어의 구조체에서 이름 없이 선언한 필드를 임베디드 필드 embedded field 라고 표현

인터페이스 안에서 다른 대상에 대한 인터페이스를 만드는데 사용함. 

```go
type Foo struct {
	Bar
}

type Bar struct {
	Baz int
}

foo := Foo{}
foo.Baz = 42 // Foo의 필드로 격상됨 (올라감) 
```

- 외부 클라이언트가 격상된 필드를 사용하게 임베딩 되는 경우 잘못 사용하는 거임

```go
package main

import "sync"

type InMem struct {
	sync.Mutex
	m map[string]int
}

func New() *InMem {
	return &InMem{m: make(map[string]int)}
}

func (i *InMem) Get(key string) (int, bool) {
	i.Lock()
	v, contains := i.m[key]
	i.Unlock()
	return v, contains
}

func main() {
	m := New()
	m.Lock() // sync.Mutex가 격상되어서 외부에서 접근할 수 있게 됨
}

// 적절한 예
type InMem struct {
	mu sync.Mutex // 이렇게 하면 m.mu.Lock으로 접근해야 하는데 export 하지 않아서 Lock() 호출 불가, 타입 임베딩 안됨
	m map[string]int
}
```

- 타입 임베딩 하기 좋은 예 → 단순히 전달만 하는 메서드를 사용할 때

```go
type Logger struct {
	io.WriterCloser // 타입 임베딩으로 필듯 ㅓㄴ언
}

func main() {
	l := logger{WriterCloser: os.Stdout}
	// io.WriterCloser의 Write(), Close()에 접근 가능
	_, _ = l.Write([]byte("foo"))
	_ = l.Close()
}
```

- 타입 임베딩은 필수가 아님
    - 사용한다면
        - 필드를 간단히 접근하기 위한 편의 구문을 위해 사용하면 안됨
        - 외부에서 볼 수 없게 만들 필드나 메서드는 격상시키면 안됨
        - 포워딩 메서드가 추가된 보일러 플레이트 코드를 방지하는데 도움이 됨

## 2.11.2 빌더 패턴

빌더 패턴 → 다양한 오브젝트 생성 문제를 유연하게 처리하도록 고안한 것

```go
type Config struct {
	Port int
}

type ConfigBuilder struct {
	port *int
}

func (b *ConfigBuilder) Port(port int) *ConfigBuilder {
	b.port = &port
	return b
}

func (b *ConfigBuilder) Build() (Config, error) {
	cfg := Config{}
	
	if b.port == nil {
		cfg.Port = defaultHTTPPort
	} else {
		if *b.port == 0 {
			cfg.Port = randomPort()
		} else if *b.port < 0 {
			return Config(), errors.New("port should be positive")
		} else {
			cfg.Port = *b.port
		}
	}
	
	return cfg, nil
}

builder := httplib.ConfigBuilder{}
builder.Port(8080)
cfg, err := builder.Build()
// 지정한 포트가 잘못되었다면 여기서 에러 핸들링을 할 수 있어 포트 관리가 쉽다
if err != nil {
	return err
}
// 하지만 사용자가 default 설정을 사용하고 싶다면 nil을 전달해서 부자연스럽다. 
server, err := httplib.NewServer("localhost", cfg)
if err != nil {
	return err
}
```

## 2.11.3 함수형 옵션 패턴

- 익스포트 하지 않은 구조체에 설정을 담는다
- 각 옵션은 동일한 타입을 리턴하는 함수로 구성된다.
- 디폴트 설정을 사용하려고 해도 nil을 전달하지 않아도 되고, 사용하기 편리하다

```go
type options struct {
	port *int
}

type Option func(options *options) error

func withPort(port int) Option {
	// 클로저를 리턴 -> 익명함수이고 본문 밖 변수 참조 가능
	return func(options *options) error {
		if port < 0 {
			return errors.New("포트는 양수로 지정해야 함")
		}
		options.port = &port
		return nil
	}
}

// 가변인수 옵션 받음
func NewServer(addr string, opts ...Option)(*http.Server, error) {
// 빈 구조체 option을 선언해서
	var options options
	// opt가 있으면 세팅
	for _, opt := range opts {
		err := opt(&options)
		if err != nil {
			return nil, err
		}
	}
	
	var port int
	if options.port == nil {
		port = defaultHTTPPort
	} else {
		if *options.port == 0 {
			port = randomPort()
		} else {
			port = *options.port
		}
	}
}

server, err := httlib.NewServer("localhost", 
	httplib.WithPort(8080),
	httplib.WithTimeout(time.Second))
```

# 3.2 정수 오버플로를 무시하지 마라

## 3.2.1 개념

산술 연산 과정에서 지정된 바이트 수로 표현할 수 있는 범위를 벗어날 때 발생한다. 

타입의 최댓값을 넘어설 경우 → 비트가 음수로 바뀌게 된다. 

실행 시에는 정수 오버 플로나 언더 플로가 발생하더라도 아무런 경고 메시지가 나오지 않고 겉으로 드러나지 않게 조용히 실행된다. 

## 3.2.2 값 증가 연산에서 발생하는 정수 오버플로 잡아내기

크기가 정해진 타입으로 증가 연산을 수행하는 과정에서 발생하는 정수 오버플로를 잡아내려면 math 상수를 이용하여 값을 검사하면 된다. 

```go
func Int32(counter int32) int32 {
	if counter == math.MaxInt32 {
		panic("int32 overflow")
	}
	return counter + 1
}
```

## 3.2.3 덧셈 연산에서 발생하는 정수 오버플로 잡아내기

```go
func AddInt(a, b int) int {
	if a > math.MaxInt-b {
		panic("int overflow")
	}
}
```

## 3.2.4 곱셈 연산에서 발생하는 정수 오버플로 잡아내기

- 최솟값(max.MinInt)에 대해 검사해야 함

```go
func MultiplyInt(a, b int) int {
	if a == 0 || b == 0 {
		return 0
	}
	result := a * b
	if a == 1 || b == 1 {
		return result
	}
	if a == math.MinInt || b == math.MinInt {
		panic("integer overflow")
	}
	if result / b != a {
		panic("integer overflow")
	}
	
	return result
}
```

# 3.3 부동 소수점을 정확하게 이해하라

- float32, float64가 존재하는데 실제 산수 연산에 대해 근사치를 구하기 위해 사용
    - 실제 산수 연산에선 존재하지 않는 bit 제한이 있음
- 소수를 표현하는 비트와 지수를 표현하는 비트로 구성됨
    - 소수부는 기본값
    - 지수부는 소수부에 적용할 배수
        
        ```go
        부호 x 2^지수 x 소수
        ```
        
- 낮은 정밀도는 저장된 값의 정확도에 영향을 미친다.
    - 부동 소수점 수 두 개를 == 연산자로 비교한 결과는 엄밀히 말해 정확하지 않다.
    - 오차가 일정 수준 이하임을 확인하려면 두 값을 뺀 결과를 확인해야 한다.
- 부동 소수점 연산의 결과는 프로세서마다 달라질 수 있다.
- 10진수를 부동 소수점 수로 변환하면 정확도가 떨어진다. 여러 부동 소수점 연산을 연속해서 수행하면 오차가 발생한다.
- 속도보다 정확도가 중요하다면 덧셈, 뺄셈, 곱셈, 나눗셈 중에서 곱셈과 나눗셈부터 수행한다.
