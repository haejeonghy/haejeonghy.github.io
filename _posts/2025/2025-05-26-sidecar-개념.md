## **📌 Sidecar 개념 정리**

### **1. Sidecar란?**

> **Sidecar(사이드카)**는
> 
> 
> **주 애플리케이션 컨테이너와 함께 배포되어 보조 역할을 수행하는 독립적인 프로세스 또는 컨테이너**
> 

> 마치 오토바이에 달린 **사이드카(보조석)**처럼,
> 
> 
> **기능 보완이나 인프라 처리를 분리하여 담당**
> 

---

### **2. 왜 사용하는가?**

- **주 애플리케이션을 변경하지 않고도** 기능을 확장하거나 보완할 수 있음
- 코드 재사용, 운영 일관성, 책임 분리
- 인프라 기능을 **공통된 Sidecar로 분리**하면 각 서비스는 본연의 비즈니스 로직에 집중할 수 있음

---

### **3. 동작 구조 예시**

```
+--------------------------+
|   Application Container  |     ← 주 애플리케이션
+--------------------------+
|   Sidecar Container      |     ← 보조 역할 수행 (ex. 로깅, 프록시 등)
+--------------------------+
      (같은 Pod 내에서 함께 실행됨)
```

---

### **4. 주요 사용 사례**

| **용도** | **예시** |
| --- | --- |
| **서비스 메시** | Istio의 Envoy Sidecar → 네트워크 제어 (트래픽 라우팅, 인증, 암호화 등) |
| **로깅/모니터링** | Fluentd Sidecar → 로그 수집 및 전송 |
| **프록시/보안** | 인증 토큰 갱신, 접근 제어, TLS 암호화 |
| **스토리지/캐시** | Redis Sidecar → 캐시 또는 세션 관리 |

---

### **5. 쿠버네티스와 Sidecar**

- **Pod 단위로 실행**됨
- 하나의 Pod 안에 **주 컨테이너 + 사이드카 컨테이너**가 함께 들어감
- 서로 **localhost 네트워크 공유**, **볼륨 공유** 가능

---

### **6. 예시: Istio에서의 Sidecar (Envoy)**

```
+-------------------------+
|      My Service         |
|    (app container)      |
+-------------------------+
|     Envoy Proxy         | ← Sidecar로 붙음
|   (트래픽 암호화 등)    |
+-------------------------+
```

- 이 구조 덕분에 서비스 코드를 수정하지 않고도 TLS 암호화, 트래픽 제어 등을 적용할 수 있음

---

### **7. 장단점**

| **장점** | **단점** |
| --- | --- |
| 코드 수정 없이 기능 추가 | 리소스 사용량 증가 |
| 책임 분리 및 재사용성 | Sidecar 복잡도 증가 |
| 통일된 인프라 관리 가능 | Pod 내 컨테이너 간 연동 이슈 발생 가능 |

---

## **✅ 요약**

| **항목** | **설명** |
| --- | --- |
| 정의 | 주 컨테이너와 같은 Pod에 배포되어 보조 기능을 수행하는 컨테이너 |
| 목적 | 기능 분리, 인프라 보완, 재사용 |
| 예시 | Envoy(프록시), Fluentd(로깅), 인증/보안 모듈 등 |
| 환경 | Kubernetes Pod 내에서 함께 실행됨 |
