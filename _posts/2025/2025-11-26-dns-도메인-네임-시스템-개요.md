# DNS(도메인 네임 시스템) 개요
Created: 2025년 11월 26일 오후 10:52
Tags: Network
보관소: No
최종 편집 일시: 2026년 3월 14일 오전 1:08

## **1. 개념**

- DNS는 사람이 이해하기 쉬운 도메인 이름(예: www.example.com)을
    
    컴퓨터가 사용하는 IP 주소(예: 192.0.2.1)로 변환하는 시스템입니다.
    
    → 인터넷의 “전화번호부” 역할.
    
- DNS는 **계층적·분산형 구조**로 되어 있어 중앙 서버 하나가 모든 정보를 갖고 있지 않습니다.
- 참고:
    - Cloudflare – https://www.cloudflare.com/learning/dns/what-is-dns/
    - Wikipedia – https://en.wikipedia.org/wiki/Domain_Name_System

---

## **2. 왜 중요한가**

- DNS가 없으면 사용자는 IP 주소를 직접 기억해야 하므로
    
    인터넷 사용이 사실상 불가능해집니다.
    
- 대부분의 웹사이트 접근, API 호출, 이메일 송수신 등이
    
    **DNS 조회 → IP 주소 획득 → 접속** 흐름을 거칩니다.
    
- DNS 문제는 곧바로 서비스 지연, 접속 불가 문제로 이어집니다.

---

## **3. 동작 흐름(단계별)**

브라우저에서 www.example.com을 입력했을 때의 전체 절차입니다.

카페 비유를 함께 넣었습니다.

1. **로컬 캐시 확인**
    - 브라우저/OS가 기존에 저장한 IP가 있는지 확인
    - (비유) 안내데스크에 “이 지점 주소 아세요?”라고 먼저 물어봄
    - 참고: https://www.freecodecamp.org/news/how-dns-works-the-internets-address-book/
2. **DNS 리졸버(Resolver) 질의**
    - 캐시에 없으면 ISP 또는 회사 내부 DNS로 질의
    - 이 리졸버가 루트 → TLD → 권한 서버까지 대신 조회해 줌
    - (비유) 안내데스크가 본사에 연락해서 지점 위치를 확인
3. **루트 네임서버 → TLD → 권한 네임서버 순회**
    - 루트 서버: 최상위 .com, .org 같은 TLD 정보를 안내
    - TLD 서버: 해당 도메인 담당 서버 정보 안내
    - 권한 네임서버: 최종 실제 IP 주소 반환
    - 참고: https://aws.amazon.com/route53/what-is-dns/
4. **최종 IP 주소 응답 및 접속**
    - 리졸버가 얻은 IP를 클라이언트에 전달
    - 클라이언트는 해당 IP로 TCP/HTTPS 등 연결을 시도
    - (비유) 고객이 지점 주소를 받아 이동

---

## **4. 주요 용어 정리**

| **용어** | **설명** |
| --- | --- |
| **A 레코드** | 도메인 → IPv4 주소 매핑 |
| **AAAA 레코드** | 도메인 → IPv6 주소 매핑 |
| **CNAME** | 도메인의 별칭 설정 |
| **NS 레코드** | 도메인을 담당하는 네임서버 설정 |
| **TTL** | 캐시 유지 시간(Time To Live) |
| **재귀(Recursive) 서버** | 루트 → TLD → 권한 서버까지 직접 조회해주는 서버 |
| **권한(Authoritative) 서버** | 특정 도메인의 실제 IP 레코드를 보유한 최종 서버 |

---

## **5. 실무에서 자주 겪는 문제**

- DNS 캐시가 오래된 값을 들고 있어 **잘못된 IP로 접속하려고 할 때**
- 사내 와이파이/내부망에서 DNS 프록시나 필터링이 걸려 있을 때
- CDN 도메인이나 API 엔드포인트가 변경되었는데 **TTL이 길어 캐시 갱신이 늦을 때**
- Kubernetes/컨테이너 환경에서 DNS 지연 → 서비스 응답 지연 발생 가능
- MSA 구조에서 각 마이크로서비스가 도메인 이름 기반으로 통신할 경우
    
    **DNS 장애 = 전체 서비스 장애**로 이어질 위험 있음
    

---

## **6. 왜 와이파이를 껐다 켜면 해결되는가?**

- 로컬 DNS 캐시가 초기화됨
- DHCP 재할당으로 **새 DNS 서버 목록을 다시 받음**
- 사내 AP(DNS 프록시)의 캐시를 초기화함
- VPN DNS override가 재적용됨
- 결국 “잘못된 DNS 상태가 다 리셋되고 정상 상태로 돌아오기 때문”

---

## **7. 요약**

> DNS는 도메인 이름을 IP 주소로 바꾸는 인터넷의 핵심 인프라이며,
> 

> 모든 네트워크 통신의 출발점이다.
> 

> 캐시 문제, 내부망 DNS 정책, TTL 지연 등으로 인해 장애가 쉽게 발생하므로
> 

> MSA·내부망 운영 환경에서는 DNS 관리를 매우 신중하게 해야 한다.
>
