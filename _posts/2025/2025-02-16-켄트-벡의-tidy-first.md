# 코드 정리와 리팩토링 원칙

## 📌 1. 코드 정리는 작은 단계로 진행하세요

- 항상, 그리고 반드시 **작은 단계를 거쳐** 코드를 정리하세요.

## 📌 2. 코드 패턴을 일관되게 유지하세요

- 한 가지 패턴을 사용하면 익숙해지지만, **두 가지 이상의 패턴을 섞으면 혼란스럽습니다.**
- 코드의 동일한 동작이 **다른 방식으로 작성되면 의미가 흐려질 수 있습니다.**
- 한 가지 방식을 선택하고 **일관성 유지**

---

## 📌 3. 인터페이스를 명확히 하세요

- 루틴을 호출해야 하는데 기존 인터페이스가 **어렵거나 복잡하면 새로운 인터페이스를 구현**
- **새 인터페이스 → 기존 인터페이스를 호출** → 기존 인터페이스 제거 후 직접 구현

---

## 📌 4. 코드 정리 원칙

### ✅ 코드의 가독성과 유지보수성을 높이는 방법

1. **독자의 입장에서 코드 읽기**
2. **전체 파일의 순서 변경보다 읽는 순서에 영향을 주는 요소부터 변경**
3. **변경할 요소를 가까이 배치**
4. **복잡한 표현식은 일부를 변수로 추출하고 의미를 명확히**

### ✅ 입력을 명확하게 만드는 방법

- 루틴에서 **명시적으로 전달되지 않는 데이터가 있다면?**
    - 루틴을 나누고, 앞부분에서 **매개변수 값을 채운 후** 뒷부분에서 명시적으로 전달

### ✅ 코드 정리의 필요성

- **코드 작성보다 코드 읽고 이해하는 비용이 더 크다**
- 코드 정리를 하면 **작은 단위로 분리하여 응집도를 높일 수 있음**
- 한 번에 머릿속에 기억해야 하는 코드의 양을 줄일 수 있음

---

## 📌 5. 코드 정리 프로세스

### ✅ PR (Pull Request) 관리

- 코드 정리는 **별도의 PR로 분리**
- **PR당 몇 개의 코드 정리만 포함**하여 변경 사항을 명확히

### ✅ 연쇄적인 코드 정리 과정

1. **보호 구문 정리**
2. **사용하지 않는 코드 삭제**
3. **대칭적인 코드 구조 맞추기**
4. **새로운 인터페이스 도입 후 기존 루틴 변경**
5. **읽기 순서 최적화**
6. **응집도를 높이는 배치**
7. **설명을 위한 변수 도입**
8. **설명을 위한 상수 도입**
9. **명시적인 매개변수 사용**
10. **비슷한 코드끼리 묶기**
11. **도우미 함수 추출**
12. **더미 코드 한 곳으로 모으기**
13. **설명하는 주석 추가**
14. **불필요한 주석 삭제**

---

## 📌 6. 코드 정리와 동작 변경의 우선순위

### ❌ **코드 정리를 하지 말아야 할 경우**

- **앞으로 다시는 코드 변경이 없을 때**
- **설계를 개선해도 배울 것이 없을 때**

### ⏳ **코드 정리를 나중으로 미뤄야 할 경우**

- **정리할 코드가 너무 많지만 보상이 불확실할 때**
- **작은 단위로 나누어 정리가 가능할 때**

### 🔄 **동작 변경 후에 코드 정리해야 할 경우**

- **코드 정리를 미루면 변경 비용이 더 커질 때**
- **코드 정리를 안 하면 일이 끝나지 않은 느낌이 들 때**

### 🎯 **코드 정리 후에 동작 변경해야 할 경우**

- **코드 정리를 통해 코드 이해가 쉬워지거나 동작 변경이 쉬워지는 경우**
- **어떤 코드를 어떻게 정리해야 할지 명확한 경우**

---

## 📌 7. 코드 정리 여부를 결정하는 4가지 기준

### 1️⃣ 비용

- **코드를 정리하면 비용이 줄어들까요?**
- **나중에 하는 게 더 나을까요?**

### 2️⃣ 수익

- **코드를 정리하면 수익이 더 커질까요?**
- **더 빨리 이익을 얻을 수 있을까요?**

### 3️⃣ 결합도

- **코드를 변경하면 의존성이 줄어드나요?**
- **변경 시 필요한 요소가 줄어드나요?**

### 4️⃣ 응집도

- **코드를 정리하면 변경 범위를 좁힐 수 있을까요?**
- **더 작은 단위로 집중해서 변경할 수 있을까요?**
