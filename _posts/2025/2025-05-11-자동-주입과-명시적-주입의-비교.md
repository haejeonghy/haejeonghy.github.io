## 📚 의존성 주입 (Dependency Injection)이란?

**의존성 주입**은 **객체**나 **함수**가 필요한 **의존성(Dependency)**을 외부에서 주입받는 방법입니다. 이는 **코드의 결합도를 낮추고**, **테스트 용이성**을 높이며, **유연한 아키텍처**를 만들기 위한 핵심 설계 패턴입니다.

---

## 🔗 의존성 주입의 두 가지 방법

### 1. 명시적 주입 (Explicit Injection)

- **명시적으로 함수나 객체의 생성자에서 직접 필요한 의존성을 전달하는 방법**
- **간결하고 직관적**
- **코드 변경 시 영향 범위가 작음**

### 📝 예시 (Golang)

```go
// 인터페이스 정의
type Database interface {
    Query(query string) string
}

type MySQL struct{}

func (db *MySQL) Query(query string) string {
    return "Result from MySQL: " + query
}

// 명시적 주입
func GetUserData(db Database, userID int) string {
    return db.Query(fmt.Sprintf("SELECT * FROM users WHERE id = %d", userID))
}

func main() {
    db := &MySQL{}
    fmt.Println(GetUserData(db, 123)) // Result from MySQL: SELECT * FROM users WHERE id = 123
}

```

### **장점**

- **코드의 명확성**: 의존성을 추적하기 쉽고, 사이드 이펙트가 적음
- **테스트 용이성**: Mock이나 Stub을 쉽게 주입할 수 있어 단위 테스트가 간편
- **유연한 코드 구조**: 다양한 구현체를 주입하여 쉽게 교체 가능
- **변경 범위의 최소화**: 코드의 결합도를 낮춰 작은 변경이 전체 시스템에 미치는 영향을 줄임

### **단점**

- **코드가 장황해질 수 있음**: 매번 의존성을 직접 주입해야 함
- **복잡한 의존성 체계를 관리하기 어려움**

---

### 2. 자동 주입 (Automatic Injection)

- **의존성을 자동으로 주입하여 복잡성을 줄이는 방법**
- **초기 설정이 복잡할 수 있음**
- **코드의 제어 흐름이 숨겨져 있어 직관성이 떨어질 수 있음**

### 📝 예시 (Golang + Uber Fx)

```go
package main

import (
    "context"
    "fmt"
    "go.uber.org/fx"
)

type Database interface {
    Query(query string) string
}

type MySQL struct{}

func (db *MySQL) Query(query string) string {
    return "Result from MySQL: " + query
}
}

func NewDatabase() Database {
    return &MySQL{}
}

func GetUserData(db Database) {
    fmt.Println(db.Query("SELECT * FROM users"))
}

func main() {
    app := fx.New(
        fx.Provide(NewDatabase),
        fx.Invoke(GetUserData),
    )
    app.Run()
}

```

### **장점**

- **복잡한 초기화의 단순화**: 의존성의 초기화와 주입이 자동으로 처리됨
- **모듈화된 구성**: 각 컴포넌트를 독립적으로 관리 가능
- **생명주기 관리**: 시작과 종료 시점의 로직을 명확히 제어할 수 있음
- **중복 제거**: 동일한 의존성을 여러 곳에서 쉽게 공유 가능

### **단점**

- **코드의 추적성 부족**: 의존성의 흐름이 숨겨져 있어 디버깅이 어려움
- **테스트 복잡성**: 의존성 그래프가 복잡해질 경우 Mocking이 어려울 수 있음
- **초기 학습 곡선**: 새로운 팀원이나 유지보수 담당자가 적응하기 어려움

---

## 📝 결론

- **명시적 주입**은 **코드의 명확성**과 **테스트 용이성**이 뛰어나며, 작은 규모의 프로젝트나 단일 모듈에 적합합니다.
- **자동 주입**은 복잡한 초기화가 필요한 대규모 시스템에서 더 유리하며, 코드의 일관성을 높일 수 있습니다.
- 두 접근 방식은 **프로젝트의 규모**와 **복잡도**에 따라 적절히 선택하는 것이 중요합니다.
