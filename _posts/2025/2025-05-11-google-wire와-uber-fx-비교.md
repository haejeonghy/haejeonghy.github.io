# 🛠️ Google Wire와 Uber Fx 비교

---

## 📚 Google Wire란?

**Google Wire**는 **컴파일 타임 의존성 주입 (Compile-Time Dependency Injection)** 도구로, **간단하고 타입 안전한 의존성 관리**를 목적으로 합니다. 이는 런타임 주입이 아닌 **컴파일 시점**에 의존성을 결정하는 점에서 **Uber Fx**와 차이가 있습니다.

---

### ✅ Wire의 주요 특징

- **컴파일 타임 의존성 주입**
- **타입 안정성 (Type Safety)**
- **빠른 성능** (Reflection 없이 동작)
- **간결하고 명확한 코드 구조**
- **낮은 런타임 오버헤드**

---

### 🛠️ 기본 동작 방식

Wire는 **의존성 그래프**를 컴파일 시점에 생성하여, 런타임에 필요한 객체를 빠르게 주입할 수 있도록 합니다. **Go Generate**와 함께 사용하여, 의존성 트리를 미리 생성하는 방식입니다.

### **📝 기본 예제 (Wire)**

```go
// main.go
package main

import (
    "fmt"
)

type Database struct {
    DSN string
}

type UserService struct {
    DB *Database
}

func NewDatabase() *Database {
    return &Database{DSN: "user:password@tcp(localhost:3306)/mydb"}
}

func NewUserService(db *Database) *UserService {
    return &UserService{DB: db}
}

func main() {
    userService := InitializeUserService()
    fmt.Println(userService.DB.DSN) // user:password@tcp(localhost:3306)/mydb
}

```

---

### **📝 Wire 설정 파일 (wire.go)**

```go
// wire.go
// +build wireinject

package main

import "github.com/google/wire"

func InitializeUserService() *UserService {
    wire.Build(NewDatabase, NewUserService)
    return &UserService{}
}

```

---

### **📝 Wire 파일 생성**

```bash
wire

```

이 명령어를 실행하면 Wire가 의존성 트리를 생성하여 **wire_gen.go** 파일을 만듭니다.

---

### **📝 Wire가 생성한 코드 (wire_gen.go)**

```go
// Code generated by Wire. DO NOT EDIT.

package main

func InitializeUserService() *UserService {
    database := NewDatabase()
    userService := NewUserService(database)
    return userService
}

```

---

### ✅ Wire의 장점

- **타입 안정성:** 컴파일 시점에 타입을 검사하므로 런타임 오류가 적음
- **퍼포먼스 최적화:** 리플렉션 없이 컴파일된 코드로 작동하여 빠름
- **간결한 코드:** 의존성 생성과 주입이 분리되어 코드가 더 읽기 쉬움
- **테스트 편의성:** 의존성 그래프가 명시적이어서 Mocking이 간단함

### ⚠️ Wire의 단점

- **복잡성 증가:** 대규모 프로젝트에서는 의존성 트리가 복잡해질 수 있음
- **코드 생성 필요:** 매번 Wire를 실행하여 코드를 생성해야 함
- **유연성 부족:** 런타임에서 의존성 변경이 어려움
- **생명주기 관리 부족:** 자원 해제나 종료 로직이 포함되지 않음

---

## 🚀 Uber Fx와의 비교

| **특징** | **Google Wire** | **Uber Fx** |
| --- | --- | --- |
| **의존성 주입 시점** | 컴파일 타임 | 런타임 |
| **생명주기 관리** | 수동 (직접 구현) | 자동 (Lifecycle Hook) |
| **복잡한 초기화** | 어려움 | 쉬움 |
| **의존성 트리 관리** | 고정적 | 동적 |
| **유연성** | 낮음 | 높음 |
| **테스트 편의성** | 높음 (명시적) | 낮음 (숨겨진 흐름) |

---

### 📝 같은 예제를 Fx로 작성하면

```go
// main.go
package main

import (
    "context"
    "fmt"
    "go.uber.org/fx"
)

type Database struct {}

type UserService struct {
    DB *Database
}

func NewDatabase(lc fx.Lifecycle) *Database {
    db := &Database{}

    lc.Append(fx.Hook{
        OnStart: func(ctx context.Context) error {
            fmt.Println("Connecting to Database...")
            return nil
        },
        OnStop: func(ctx context.Context) error {
            fmt.Println("Disconnecting from Database...")
            return nil
        },
    })

    return db
}

func NewUserService(db *Database) *UserService {
    return &UserService{DB: db}
}

func main() {
    app := fx.New(
        fx.Provide(NewDatabase, NewUserService),
    )
    app.Run()
}

```

---

### ✅ Fx의 장점

- **자원 누수 방지:** 리소스를 확실하게 해제할 수 있음
- **의존성 순서 보장:** 의존성 간의 초기화 순서를 명확히 정의 가능
- **안정적인 종료:** Graceful Shutdown으로 데이터 손실 방지
- **일관성:** 초기화와 종료 로직이 명확하게 분리

### ⚠️ Fx의 단점

- **초기 설정 비용:** 모든 컴포넌트의 초기화와 종료를 명확히 정의해야 함
- **디버깅 어려움:** 자동 주입으로 인해 의존성 흐름이 숨겨질 수 있음

---

## 📝 결론

- **Wire**는 간단하고 빠른 컴파일 타임 의존성 주입에 적합하지만, 생명주기 관리가 부족하여 복잡한 애플리케이션에는 한계가 있음.
- **Fx**는 의존성 초기화, 종료를 자동으로 관리하여 대규모 시스템에 더 적합하며, 복잡한 생명주기 관리가 필요한 경우 더 나은 선택임.
