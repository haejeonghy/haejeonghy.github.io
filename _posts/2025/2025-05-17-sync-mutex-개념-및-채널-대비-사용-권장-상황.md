# 🔒 `sync.Mutex` 개념 및 채널 대비 사용 권장 상황

---

## 📌 `sync.Mutex`란?

`sync.Mutex`는 Go에서 제공하는 **뮤텍스(Mutual Exclusion, 상호 배제)** 락 객체로,

**하나의 고루틴만 특정 코드 블록 또는 데이터에 접근할 수 있도록 제어**하는 동시성 도구이다.

### ✨ 기본 구조

```go
var mu sync.Mutex

mu.Lock()
// 임계 구역 (critical section)
mu.Unlock()

```

- `Lock()` 호출 → 뮤텍스 획득
- `Unlock()` 호출 → 뮤텍스 해제
- 동시에 하나의 고루틴만 락을 잡을 수 있음

---

## 🧾 예시

```go
var (
    mu   sync.Mutex
    data int
)

func increment() {
    mu.Lock()
    data++
    mu.Unlock()
}

```

- 여러 고루틴이 동시에 `data++` 해도 **경쟁 상태 없이 안전하게 동작**

---

## 🧠 채널 vs 뮤텍스 비교

| 항목 | 채널 (channel) | 뮤텍스 (mutex) |
| --- | --- | --- |
| 개념 | 데이터 전달 수단 | 접근 제어 수단 |
| 통신 방식 | 데이터를 **보내고 받음** | 공유 자원에 **접근을 잠금(lock)** |
| 상태 공유 | 공유하지 않고 메시지를 통해 상태 전달 | **공유된 상태**를 잠그고 접근 |
| 동시성 제어 | 명시적인 통신을 통한 제어 | **암묵적인 락 해제**로 제어 |
| 디버깅 난이도 | 상대적으로 쉬움 (흐름이 명확) | 복잡함 (데드락 가능성) |

---

## ✅ 뮤텍스가 채널보다 나은 상황

| 상황 | 설명 |
| --- | --- |
| **단순한 카운터, 변수 보호** | `int`, `map`, `slice` 등의 간단한 자료구조 접근을 보호할 때 |
| **데이터를 공유하고 있지만, 통신이 필요 없는 경우** | 값을 주고받을 필요 없이, 단지 하나의 고루틴만 접근하게 하면 될 때 |
| **성능이 중요한 경우** | 채널은 내부적으로 더 많은 연산을 포함하므로 **락이 더 빠름** |
| **구조체 필드 일부만 보호할 때** | 예: 구조체 내부 일부 필드만 락을 걸고 싶을 경우 |
| **공유 리소스 접근이 빠르게 끝나는 경우** | 예: 짧은 read/write가 반복되는 상황에서는 락이 더 단순하고 효율적임 |

---

## ⚠️ 주의사항

- **Lock()과 Unlock()은 반드시 짝지어야 하며**, `defer mu.Unlock()` 형태로 쓰는 것이 안전
- **데드락**에 주의: 락을 여러 개 사용할 경우 순서에 유의해야 함
- **panic이 발생하면 Unlock이 호출되지 않아 영구 락이 걸릴 수 있음**

---

## 🎯 결론

- **채널은 상태 전달 중심**, **뮤텍스는 상태 보호 중심**
- **공유 상태를 안전하게 보호하고 싶을 때는 뮤텍스를,명시적인 통신 흐름을 만들고 싶을 때는 채널을 선택**
