## 6. OSI 모델

### 6.1 OSI (Open Systems Interconnection) 모델이란?

- 개방형 시스템 상호작용 모델

### 6.2 왜 통신 모델이 필요한가?

1. 존재 이유를 알아야 이해할 수 있음
2. **표준과 규칙이 없으면 클라이언트와 통신할 수 없음**
3. **표준 모델이 없으면 네트워크 장비를 업그레이드하기 어려워짐**
    - 장비들이 서로 다른 표준을 가지고 있으면 발전할 수 없음
    - 서로 다른 모델들과 장비들이 서로 통신해야 함
4. **어떤 매체를 사용하든 실제 매체와 분리되어 있어서 업그레이드 가능**
5. **대화하는 방법만 알고 있으면 장비를 정상적으로 업그레이드할 수 있음**
6. **각 계층은 독립적으로 개선 및 변경 가능하도록 구축됨**

## 7. OSI 모델 7 계층

### 7.1 **7계층 - 애플리케이션 (Application Layer)**

- 대표 프로토콜: `HTTP`, `FTP`, `gRPC`
- **네트워크 엔지니어 입장:** 데이터가 들어오는 계층
- **백엔드 엔지니어 입장:** 사실상 HTTP가 애플리케이션

### 7.2 **6계층 - 표현 (Presentation Layer)**

- 역할: `Encoding`, `Serialization`
- **예시:**
    - 구조를 문자열로 직렬화할 때
- **특징:**
    - 애플리케이션에서 한 계층 내려옴
    - OSI 모델에서 이런 기능을 꼭 분리해야 함

### 7.3 **5계층 - 세션 (Session Layer)**

- 역할: `Connection Establishment`, `TLS`
- **특징:**
    - 연결이 설정되고 상태가 실질적으로 관리됨
    - 클라이언트와 서버 간 상태 저장
    - 몇몇 프로토콜을 `Stateful`, `Stateless`로 구분
    - 세션이 제거되면 연결을 다시 시작하거나 무효화할 수 있음
    - 암호화 기능 포함

### 7.4 **4계층 - 전송 (Transport Layer)**

- 대표 프로토콜: `UDP`, `TCP`
- **중요성:**
    - 백엔드 엔지니어에게 **4계층과 7계층이 가장 중요함**
- **특징:**
    - `TCP`, `UDP`는 4계층에서만 존재
    - 모든 것은 `TCP` 또는 `UDP` 위에 구축됨

### 7.5 **3계층 - 네트워크 (Network Layer)**

- **특징:**
    - 포트에 대한 개념이 없음
    - `IP 주소`만 알고 있음
    - `IP`는 **라우팅, IP 주소 개념을 포함**

### 7.6 **2계층 - 데이터 링크 (Data Link Layer)**

- **개념:** `프레임`, `MAC 주소`, `이더넷`

### 7.7 **1계층 - 물리 (Physical Layer)**

- **개념:** `광섬유`, `빛의 파동`, `전기 신호`
- **특징:**
    - 데이터를 **2계층 프레임으로 변환해야 함**

## 📂 1️⃣ 데이터 흐름 개요

📌 **리시버(Receiver)에서의 데이터 처리 과정**

1. **물리 계층(1계층) → 디지털 비트 변환 및 전송**
2. **데이터 링크 계층(2계층) → 데이터 프레임으로 변환하여 전달**
3. **네트워크 계층(3계층) → IP 패킷으로 변환**
4. **전송 계층(4계층) → TCP 세그먼트로 변환**
5. **세션 계층(5계층) → 세션 생성 및 관리**
6. **표현 계층(6계층) → JSON 데이터 직렬화 및 변환**
7. **애플리케이션 계층(7계층) → 클라이언트 요청 처리 및 응답 반환**

📌 **데이터의 변환 관계**

- **세그먼트 → IP 패킷으로 변환 → 프레임 안에 포함되어 전송됨**

---

## 🔄 2️⃣ OSI 계층별 데이터 처리 과정

### **1️⃣ 물리 계층 (Physical Layer)**

- 데이터를 **디지털 비트 스트림(0과 1)**으로 변환하여 전송
- 광섬유, 전기 신호, 무선 신호 등의 형태로 데이터를 전송

### **2️⃣ 데이터 링크 계층 (Data Link Layer)**

- 수신된 데이터를 **프레임(Frame)** 형태로 변환
- **MAC 주소 기반으로 통신**하여 데이터를 올바른 노드로 전달

### **3️⃣ 네트워크 계층 (Network Layer)**

- *IP 패킷(IP Packet)**으로 변환하여 송수신
- **IP 주소 기반으로 경로(Routing) 결정**

### **4️⃣ 전송 계층 (Transport Layer)**

- **TCP 또는 UDP 프로토콜을 사용하여 세그먼트(Segment)로 변환**
- **연결 지향적(TCP) 또는 비연결형(UDP) 방식 선택**

📌 **세그먼트 → IP 패킷 → 프레임 안에 포함되어 전송됨**

---

## 🔑 3️⃣ 연결 과정과 세션 처리

📌 **HTTPS POST 요청 시 데이터 흐름**

1. **POST 요청을 새로 시작할 때, 세션 생성 과정 발생 (SYN 요청 전송)**
2. **서버는 Application 계층까지 가지 않고, 세션 계층(5계층)에서 멈추고 세션을 생성**
3. **SYN/ACK 응답을 보낸 후 연결이 완료됨**
4. **세션이 만들어졌으므로 POST JSON 요청을 다시 시작**
5. **JSON 데이터를 변환하여 애플리케이션 계층에서 처리**
6. **이벤트 트리거, 데이터베이스 저장, 클라이언트 응답 반환**

📌 **항상 모든 계층을 거칠 필요는 없음**

- 예를 들어, 세션을 만든 후에는 바로 **JSON 데이터를 받아서 6계층으로 전달** 가능

---

## 🛠️ 4️⃣ 표현 계층(6계층)과 애플리케이션 계층(7계층)의 역할

📌 **6계층 (표현 계층, Presentation Layer)**

- JSON 데이터를 직렬화(Serialization) 및 변환
- 암호화, 압축, 인코딩 등의 역할 수행
- **일부에서는 6계층이 실제 애플리케이션이라고 주장하기도 함**

📌 **7계층 (애플리케이션 계층, Application Layer)**

- 클라이언트 요청을 처리하고, 필요한 데이터베이스 작업 수행
- 최종적으로 데이터를 변환하여 응답 반환

📌 **애플리케이션 계층의 데이터 흐름 예시**

- JSON 데이터를 파싱하여 내부 서비스로 전달
- 이벤트 트리거 또는 데이터베이스에 저장
- 처리 완료 후 클라이언트에게 응답 반환
