### 동시성 프로그래밍

- 비동기적 : 여러개의 루틴이 선행작업의 순서나 완료 여부 상관없이 실행되는 방식
    - RxJava, Reactive 등 서드파티 라이브러리에서 제공
- 코루틴 : 여러 개의 개별적인 작업 (루틴) → 협력 (코) 한다는 의미
- 병행 수행 : 여러 개의 코어가 태스크가 동시에 수행되는 것
- 태스크 : 큰 실행 단위인 프로세스나 좀 더 작은 실행 단위인 스레드
    - 프로세스 : 실행되는 메모리, 스택, 열린 파일 등 모두 포함하기 때문에 프로세스 간 문맥 교환 할 때 비용이 많이 듬
    - 스레드 : 자신의 스택만 독립적으로 가지고 나머지는 대부분 스레드끼리 공유하므로 문맥 교환 비용이 낮아 프로그래밍에서 많이 사용됨
    - 코루틴은 문맥교환이 없고 최적화된 비동기 함수를 통해 비선점형으로 작동하는 특징이 있어 협력형 멀티태스킹 구현 가능하게 함

### 코루틴의 개념과 사용 방법

- 코루틴 : 비용이 많이 드는 문맥 교환 없이 해당 루틴을 일시 중단해서 이러한 비용을 줄일 수 있음
    
    → 운영체제가 스케줄링에 개입하는 과정이 필요하지 않음
    
    → 일시 중단은 사용자가 제어할 수 있음
    
- 코루틴 주요 패키지
    - launch /async : 코루틴 빌더
    - Job / Deferred : cancellation 지원
    - Dispatchers : Default는 백그라운드 코루틴을 위한 것이고 Main은 Android나 Swing, JavaFx 위해 사용
    - delay / yield : 상위 레벨 지연 함수
    - Channel / Mutex : 통신과 동기화를 위한 기능
    - coroutineScope / supervisorScope : 범위 빌더
    - select : 표현식 지원
- core 패키지 주요 기능
    - CommonPool : 코루틴 문맥
    - produce / actor : 코루틴 빌더
- launch & async
    - launch :  코루틴 블록 만들어내는 것 -> 새로운 코루틴을 실행할 수 있게 하며 Job 객체 반환
    - async : 새로운 코루틴을 실행할 수 있는데 launch와는 달리 Deferred<T>를 통해 결괏값 반환

```kotlin
fun main() { // 메인 스레드 문맥
	GlobalScope.launch { // 새로운 코루틴 백그라운드 실행 
		delay(1000L) // 1초 넌블로킹 지연
		println("World")
	}
	println("hello") // 메인 스레드의 코루틴 지연되는 동안 계속 실행
	Thread.sleep(2000L) // 바로 종료 안되게 2초 기다림 
}

val job = GlobalScope.launch { // GlobalScope : 실행 범위 결정 
	delay(1000L)
	println("")
}

// await()을 사용해서 지연된 결괏값을 받기 위해 사용

val one = GlobalScope.async {
	}
val two = GlobalScope.async {
	}

GlobalScope.launch {
	val combined = one.await() + "_" + two.await()
}
```

```kotlin
public suspend fun delay(timeMillis: kotlin.Long): kotlin.Unit { } 

// 사용자 함수 추가 
suspend fun doSomething() {
}
```

- 코루틴 문맥 : CoroutineContext에 의해 정의됨
    - launch{ ...} 등 인자가 없는 경우에는 CoroutineScope에서 상위의 문맥이 상속되어 결정되고 launch(Dispatchers.Default) { .. } 와 같이 사용되면 GlobalScope에서 실행되는 문맥과 동일하게 사용됨 (GlobalScope는 메인 스레드의 생명주기가 끝나면 같이 종료됨)
    - 내부적으로 보통 CommonPool이 지정되어 코루틴이 사용할 스레드의 공동 풀을 사용하게 됨
- CoroutineStart : 시작 방법 정의
    - DEFAULT : 즉시 시작
    - LAZY : 코루틴을 느리게 시작(처음에는 중단된 상태이며 start(), await() 등으로 시작)
    - ATOMIC : 최적화된 방법으로 시작
    - UNDISPATCHED : 분산 처리 방법으로 시작
    
    ```kotlin
    public fun launch (
    	context: CoroutineContext,
    	start: CoroutineStart,
    	parent: Job?,
    	onCompletion: CompletionHandler?,
    	block: suspend CoroutineScope.() -> Unit): Job {
    }
    ```
    
- runBlocking() : 새로운 코루틴을 실행하고 완료되기 전까지 현재 스레드를 블로킹함
    - 지연함수 사용가능 → 블록을 2초정도 붙잡음

```kotlin
runBlocking {
	delay(2000)
}

// main으로 블로킹 모드로 동작시키기
fun main() = runBlocking<Unit> { // main() 함수가 코루틴 환경에서 실행
	launch { // 백그라운드로 코루틴 실행
		delay(1000L)
		println("World")
	}
	//delay(2000L) 사용하지 않아도 코루틴을 기다림 
}

// suspend 지정 가능
suspend fun main() = coroutineScope {} 

// 클래스 내의 멤버 메서드에서도 사용 가능 
fun mySuspendMethod() = runBlocking<Unit> {
	}
```

- join() : 코루틴의 작업이 완료되는 것을 기다리게 하려면 Job 객체의 join() 함수를 사용하면 됨
    - launch에서 반환되는 값은 Job 객체이므로 main() 함수에서 join() 함수를 호출

```kotlin
fun main() = runBlocking<Unit> {
	// Job의 객체를 반환
	val job = launch {
		delay(1000L)
		println("")
	}
	job.join() // 명시적으로 코루틴이 완료되길 기다림 - 취소할 경우 job.cancel() 함수를 사용
}
```

- sequence() : 아주 많은 값을 만들어내는 코드로부터 특정 값의 범위를 가져올 수 있음
    - Sequence<T> 반환

```kotlin
val fibonacciSeq = sequence {
	var a = 0
	var b = 1
	// 지연함수 
	yield(1)

	while (true) {
		yield(a + b)
		val tmp = a + b
		a = b
		b = tmp
	}
}

fun main() {
	println(fibonacciSeq.take(8).toList()) // 8개 피보나치 수열 리스트로 출력됨
}
```

- yieldAll() : 무한한 시퀀스를 만들어내는 generateSequence() 함수를 사용해서도 요소값 산출 가능

```kotlin
var saved = fibonacciSeq.iterator()
println("${saved.next()}. ${saved.next()}, ${saved.next()}"}
```

### 코루틴 동작 제어

- 코루틴 문맥 제어

```kotlin
fun main() = runBlocking<Unit> {
	val jobs = arrayListOf<Job>()
	// Dispatcher.Unconfined는 호출자 스레드에서 코루틴을 시작하지만 첫 번째 지연점까지만 실행
	// 특정 스레드나 풀에 가두지 않고 첫 번째 일시 중단 후 호출된 지연 함수에 의해 재개됨 권장 X
	jobs += launch(Dispatchers.Unconfined) {
	}
	jobs += launch(coroutineContext) {
	}
	// 기본 문맥인 CommonPool에서 실행되고 GlobalScope로 표현됨 
	// 공유된 백그라운드 스레드의 CommonPool에서 코루틴을 실행하도록 함 - 연산 중심 코드에 적합
	jobs += launch(Dispatchers.Default) {
	}
	// 입출력 위주의 동작을 하는 코드에 적합된 공유된 풀 - 
	// 블로킹 동작이 많은 파일이나 소켓 I/O 처리에 적합
	jobs += launch(Disptachers.IO) {
	} 
	jobs += launch {
	}
	// newSingleThreadContext는 사용자가 새 스레드 풀을 만들 수 있음 
	// 비용이 많이 들고 더 이상 필요하지 않으면 해제하거나 종료시켜야 함
	jobs += launch(newSingleThreadContext("MyThread")) {
	}
	jobs.forEach{it.join(){
}
```

- 실행 상태의 판단 : 연산이 마무리되기 전까지 조건식에 의해 루틴이 중단되지 않음

```kotlin
fun main() = runBlocking<Unit> {
	val startTime = System.currentTimeMillis()
	val job = GlobalScope.launch {
		var nextPrintTime = startTime
		var i = 0
		// while(isActive) 로 변경하면 의도한 시간에 루프가 취소되어 중단됨 
			while (i < 5) {// 루프가 완료될때까지 루틴이 끝나지 않음 
			if(System.currentTimeMillis() >= nextPrintTime) {
				nextPrintTime += 500L
			}
		}
	}
	delay(1300L)
	job.cancelAndJoin()
}
```

- 채널 : 서로 자료를 주고 받기 위해 약속된 일종의 통로 역할 - 넌블로킹 전송 개념

```kotlin
fun main() = runBlocking<Unit> {
	val channel = Channel<Int>()
	launch {
		// 다량의 CPU 연산 작업이나 비동기 로직 둘 수 있음
		for (x in 1..5) channel.send(x * x)
	}
	// 5개의 값을 채널로부터 받음 
	repeat(5) { println(channel.receive())}
	println()
}

// 채널 닫기
fun main() = runBlocking<Unit> {
	val channel = Channel<Int>()
	launch {
		for (x in 1..5) channel.send(x * x)
		channel.close() // 모두 보내고 닫기 명시 
	}
	// for 문을 통해 끝까지 읽기 
	for (element in channel) println(element)
	println()
}

SendChannel.offer(element: E): Boolean // 가능하면 요소를 채널에 추가 꽉 찬 경우 false 반환
ReceiveChannel.poll(): E? // 요소를 반환, 채널이 비어있으면 null 반환 
```

- produce 생산자 소비자 패턴

```kotlin
// 생산자를 위한 함수 생성 
fun CoroutineScope.producer(): ReceiveChannel<Int> = produce {
	var total: Int = 0

		total += x
		send(total)
```

- 버퍼를 가진 채널 : send() 함수가 먼저 호출되면 receive() 함수가 호출되기 전까지 send() 함수는 일시 지연됨 → 반대의 경우에도 receive() 함수가 호출되면 send() 함수가 호출되기 전까지 receive() 함수는 지연됨

```kotlin
fun main() = runBlocking<Unit> {
	// 버퍼 capacity 값을 줌 
	val channel = Channel<Int>(3)
	val sender = launch(coroutineContext) { // 송신자 측
		repeat(10) {
			println()
			channel.send(it) // 지속적으로 보내다가 꽉 차면 일시 지연
		}
	}
	delay(1000) // 아무것도 받지 않고 1초 기다린 후
	sender.cancel() // 송신자의 작업 취소
}
```

- select : 다양한 채널에서 무언가 응답해야 한다면 각 채널에 따라 결과가 달라질 수 있는데 이때 select를 사용하면 표현식을 통해 결과를 받을 수 있음

```kotlin
fun main() = runBlocking {
	val routine1 = GlobalScope.produce {
		delay()
		send("A")
	}
	val routine2 = GlobalScope.produce {
		delay()
		send("B")
	}
	val result = select<String> {
		// onReceive를 통해 값을 받아 완성된 결과를 가져옴
		routine1.onReceive { result -> result }
		routine2.onReceive { result -> result }
	}
} 
```

### 공유 데이터 문제 알아보기

- 병행 프로그래밍에서는 전역 변수 같은 변경 가능한 공유 자원에 접근할 때 값의 무결성을 보장할 수 있는 방법이 필요함 → 코틀린에서는 추가로 공유 자원의 보호와 스레드 안전을 구현하기 위해 원자 변수, 스레드 가두기, 상호 배제 등을 사용할 수 있음
- 동기화 기법
    - synchronized 메서드와 블록 : 스레드 간 서로 공유하는 데이터가 있을 때 동기화해서 데이터의 안전성 보장함 → 특정 스레드가 이미 자원을 사용하는 중이면 나머지 스레드의 접근을 막음

```kotlin
@Synchronized fun synchronizedMethod() {
}
```

- 원자 변수 : 특정 변수의 증가나 감소, 더하기나 빼기가 단일 기계어 명령으로 수행되는 것 → 해당 연산이 수행되는 도중에는 누구도 방해하지 못하기 때문에 값의 무결성을 보장할 수 있게됨
- 스레드 가두기 : 특정 문맥에서 작동하도록 단일 스레드에 가두는 방법이 있음
    - newSingleThreadContext 사용

```kotlin
fun main() = runBlocking<Unit> {
	// massiveRun() 스레드 가두기를 직접 적용
	massiveRun(counterContext) {
		counter++
	}
}
```

- 상호 배제 : 코드가 임계 구역에 있는 경우 절대로 동시성이 일어나지 않게 하고 하나의 루틴만 접근하는 것 보장
- Mutex의 lock(), unlock() 사용하여 임계 구역을 만들 수 있음
    - isLocked 사용하여 mutext가 잠금 상태일 때 true를 반환함

```kotlin
val mutex = Mutext()
var counter = 0

fun main() = runBlocking<Unit> {
	massiveRun {
		mutex.withLock {
			counter++ // 임계 구역 코드 
		}
	}
}
```

- actor 코루틴 빌더
    - actor : 코루틴과 채널에서 통신하거나 상태를 관리함 : 들어오는 메일 기능만 함

```kotlin
data class Task (val desc: String)
// actor는 한 번에 1개의 메시지만 처리하는 것 보장
val me = actor<Task> {
	// 닫힌 상태가 아니라면 receive() 사용해서 desc를 반복 출력
	while(!isClosedForReceive) {
		println(receive().desc.repeat(5))
	}
}
```

- 이벤트 루프
    - Kovenant, Quasar, libuv와 같은 라이브러리 통해 이벤트 처리를 위한 프로그래밍 모델을 만들기 위해 이벤트 루프를 사용함
    - 이벤트 루프 : 웹 애플리케이션을 만들 때 사용자와 상호작용하기 위해서는 항상 이벤트를 기다리며 감시하는 주체가 필요하며, 이벤트가 발생하면 이것을 처리하기 위해 특정 루틴을 동작시켜야 함
        - 각 이벤트 요청에 대한 이벤트 큐를 가지며 이벤트 큐는 이벤트 루프에 의해 처리할 핸들러인 작업자 스레드가 결정되어 실행됨 → 이벤트 큐의 실행이 끝나면 이벤트 루프에 의해 다시 이벤트 큐의 위치로 돌아감
