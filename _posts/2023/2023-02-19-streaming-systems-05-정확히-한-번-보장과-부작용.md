# Streaming Systems: 05 ‘정확히 한 번’ 보장과 부작용
Created: 2023년 2월 19일 오후 5:17
보관소: No
최종 편집 일시: 2026년 3월 14일 오전 1:08

- 시스템이 정확한 결과를 얻기 위해 빔 모델을 올바르게 구현하는 방법에 대해 다룬다.
- ‘정확히 한 번 exactly-once’ 보장
    - 모든 데이터가 정확히 한 번 처리되도록 하는 것

# 왜 ‘정확히 한 번’ 보장이 중요한가?

- 많은 범용 스트리밍 시스템은 모든 데이터를 처리한다고 보장할 수 없었다.
- ‘최소한 한 번 at-least-once’ 보장을 통해 누락 없이 적어도 한 번은 데이터가 처리됨을 보장해줬으나
    - 중복 처리의 가능성이 있어 집계 결과는 부정확할 수 있다.
    - 이 시스템은 메모리상에서 집계 작업을 수행해 시스템에 문제가 발생하면 결과에 손실이 발생할 수 있다.
    - 짧은 지연 시간으로 예측 결과만 제공할 수 있었지만 결과의 정확성에 대해서는 아무것도 보장해주지 못했다.
- 람다 아키텍처 Lambda Architecture의 등장
    - 스트리밍 시스템을 빠르지만 부정확한 결과를 얻기 위해 사용하고 배치 시스템을 통해 정확한 결과를 얻도록 해준다.
    - 데이터스트림이 재현 가능한 경우에만 가능한 방법이지만 충분히 많은 데이터가 이 조건을 만족해 쓸만했다.
- 람다 아키텍처의 문제
    - 부정확성 inaccuracy
        - 운이 없으면 10% 혹은 그 이상의 데이터가 영향받을 수 있다.
        - 오직 절반의 보장만을 약속한다고 본다.
    - 불일치 inconsistency
        - 최종 계산에 사용되는 배치 시스템은 종종 스트리밍 시스템과 다른 방식으로 데이터를 다룬다.
        - 두 파이프라인이 비슷한 결과를 내도록 하는 것은 어렵다.
    - 복잡성 complexity
        - 람다 아키텍처는 정의에 따라 서로 다른 코드 둘을 작성하고 유지해야 한다.
        - 각각 다른 실패 모드를 가진 2개의 복잡한 분산 시스템을 실행하고 유지해야 한다.
        - 매우 단순한 형태의 파이프라인이 아니라면 부담이 생각보다 크다.
    - 예측 불가능성 unpredictability
        - 사용자는 매번 다른 비율로 스트리밍 결과와 배치 결과가 다르다는 것을 확인하게 된다.
        - 그렇게 되면 사용자는 스트리밍 데이터를 불신하고 배치 결과를 기다리게 된다.
    - 지연 latency
        - 일부 경우에는 지연 시간이 짧으면서 정확한 결과가 필요하다.
        - 람다 아키텍처는 이를 애초에 제공하지 않는다.

# 정확성 대 완결성

- 빔 파이프라인이 데이터를 처리할 때 데이터가 누락되거나 중복 처리되지 않도록 해줘야 한다.
    - 하지만 스트리밍 파이프라인 특성상 시간 윈도우 집계가 완료된 후 데이터가 늦게 나타날 수 있다.
    - 빔 SDK는 시스템이 이러한 지연 데이터를 얼마나 기다릴지 설정하게 해준다.
        - 이 시간보다 늦게 도착된 데이터는 모두 무시된다.
        - 완결성 completeness에 대한 것으로 볼 수 있다.
        - 처리 시간 동안 나타난 모든 데이터는 정확히 한 번 처리되고, 늦게 도착된 데이터는 명시적으로 무시된다.
- 배치 시스템도 같은 완결성의 문제를 갖고 있다.
    - 배치에서 다루도록 설정된 시간대에 대한 데이터가 배치가 돌아간 이후에 수집된다면 누락될 것이다.

## 부작용

- 빔과 데이터플로우의 특징 중 하나는 사용자가 파이프라인 그래프의 일부분으로 실행되는 코드를 직접 작성해 삽입할 수 있게 해준다는 것이다.
    - 데이터 플로우는 이 코드가 데이터당 한 번만 실행됨을 보장해주지 않는다.
        - 하나의 데이터에 사용자 코드를 여러 번 실행할 수도 있고, 여러 워커에서 같은 데이터를 대상으로 동시에 실행될 수도 있다.
        - 시스템 장애 시에도 ‘최소한 한 번’ 보장을 제공하기 위해 필요한 부분
        - 중복 계산 결과 중 하나만 선택돼 파이프라인으로 흘러간다.
- 멱등성 idempotence 이 없는 부작용은 정확히 한 번만 실행됨이 보장되지 않는다.
    - 외부 서버로 데이터를 전송하는 등 파이프라인 외부에 영향을 주는 부작용을 포함하는 코드를 작성한다면 단일 데이터에 대해 두 번 이상 실행되는 결과가 나올 수 있다.
        - 외부 서비스에 영향을 주는 작업을 데이터가 원자적으로 다룰 수 있는 방법이 없기에 이런 상황이 불가피하다.
        - 보통 파이프라인은 최종 결과를 파이프라인 밖으로 내보내고, 그러한 작업은 멱등성을 갖지 않는다.
            - 이런 경우 싱크 sink 에 멱등성이 없는 작업을 멱등성을 보장하는 형태로 재구성하는 추가적인 단계를 넣어 문제를 회피할 수 있다.

        - 첫 번째는 1분 동안 각 개별 사용자로부터 발생한 이벤트 수를 계산하고

        - GroupByKey 후에 동일한 키를 갖는 모든 데이터가 셔플 이라는 절차 이후 동일한 머신에서 처리 된다.
        - 데이터플로우 워커는 RPC를 사용해 데이터를 셔플해 한 키에 대한 모든 데이터가 같은 머신에 배정되도록 한다.
    - Count.perKey는 각 사용자의 모든 데이터를 지정된 워커로 셔플하고,
    - Count.global은 이러한 모든 부분 결과를 단일 워커로 셔플해 총 합계를 계산한다.
    - 데이터플로우가 데이터를 정확히 처리하려면 이 셔플에서 모든 데이터가 정확히 한 번만 셔플되도록 해줘야 한다.
        - 서플의 분산된 특성으로 인해 이 문제 해결이 쉽지 않다.
    - 파이프라인은 파이프라인 외부와 데이터를 읽고 쓴다.
        - 데이터플로우는 이 과정에서 부정확한 결과가 초래하지 않도록 보장해줘야 한다.
        - 데이터플로우는 기술적으로 가능할 때마다 소스 및 싱크에 대해 아파치 스파크와 아파치 플링크가 종단간 ‘정확히 한 번’ 보장이라고 부르는 특성을 지원해준다.

# 구글 데이터플로우 셔플에서의 ‘정확히 한 번’ 보장

- 셔플
    - 데이터플로우가 매 데이터가 셔플될 때 정확히 한 번 보장을 제공하는 방법
- 데이터플로우의 스트리밍 셔플은 RPC를 사용한다.
    - RPC 통신이 개입하는 상황이라면 데이터 무결성에 대해 심사숙고할 필요가 있다.
- RPC가 실패할 수 있는 이유
    - 네트워크가 중단되거나
    - RPC가 완료 전에 타임아웃 되거나
    - 수신 서버가 어떤 이유로 응답을 실패한 것으로 줄 수 있다.
- 데이터플로우는 셔플 도중 데이터 손실이 발생하지 않도록 업스트림 백업 upstream backup을 사용한다.
    - 발신하는 쪽에서 수신 확인을 받을 때까지 RPC를 반복해 시도함을 뜻한다.
    - 데이터플로우는 발신 서버에 크래시 crash 가 발생해도 RPC를 계속 재시도하도록 보장해준다.
    - 이렇게 하면 모든 데이터가 적어도 한 번은 전달된다고 보장할 수 있다.
- 이러한 재시도는 중복 데이터를 만들 수 있다.
    - 데이터플로우를 포함해 대부분의 RPC 프레임워크는 발신 서버에 성공 또는 실패를 표시하는 상태를 준다.
    - 분산 시스템에서는 RPC가 실패한 것처럼 보여도 종종 실제로는 성공할 수 있다.
        - RPC 경쟁 상황 race condition 이 발생해 타임아웃이 나는 경우
        - RPC가 성공했음에도 수신 서버가 확인 응답에 실패하는 경우  등
    - 발신 서버가 실제로 신뢰할 수 있는 유일한 응답은 성공에 대한 응답 뿐
- 상태로 실패를 반환하는 RPC는 일반적으로 호출이 성공했는지 여부만을 나타낸다.
    - 특정 오류 코드로 명확한 실패 이유를 전달할 수는 있지만
    - 데드라인 초과 같은 다수의 일반적인 RPC 오류는 의미가 모호하다.
        - 스트리밍 셔플 시에 실제로 성공한 RPC를 다시 시도하면 데이터가 두 번 전달될 수 있기에
        - 데이터플로우에서는 이러한 중복을 감지하고 제거하는 방법이 필요하다.
- 알고리즘
    - 전송되는 모든 메시지는 고유한 식별자가 붙는다.
    - 각 수신 서버는 이미 처리한 모든 식별자의 목록을 저장해둔다.
    - 데이터를 받을 때마다 이 목록에서 식별자를 찾고 발견되면 중복으로 무시된다. 데이터플로우는 확장성을 갖는 키/값 저장소 기반 위에 구축되므로 이 저장소를 중복 제거 목록을 보관하는 용도로 사용한다.

# 결정론 다루기

- 현실에서 실제 동작하게 하려면 많은 고민이 필요하다.
    - 빔 모델이 사용자 코드가 비결정적인 출력을 생성하도록 허용한다.
        - ParDo는 동일한 입력 데이터에 대해 (재시도 때문에) 두 번 실행되면서 각 시도마다 다른 출력을 생성할 수 있다.
            - 이 중 하나만 파이프라인에 투입되는 것이 기대되는 동작이지만
            - 비결정적인 결과 때문에 두 출력이 동일한 ID를 갖는다는 것을 보장하기 어렵다.
        - ParDo가 여러 데이터를 결과로 낼 수 있다는 것도 까다로운 점이다.
            - 재시도가 이뤄질 때마다 다른 수의 결과가 생성될 수 있다.
- 모든 사용자 코드가 결정적인 결과만을 생성해야 한다고 요구한다면?
    - 쉽지 않다.
    - 많은 파이프라인에 비결정적 변환이 필요하다.
    - 파이프라인 작성자는 자신이 작성한 코드가 비결정적이라는 사실을 깨닫지 못하는 경우가 매우 빈번하다.
        - 입력 데이터를 보충하기 위해 클라우드 빅테이블에 추가적인 데이터를 조회하는 변환의 경우
        - 변환 재시도 사이에 빅테이블 내의 값이 변경될 수 있으므로 이는 비결정적인 작업에 해당한다.
        - 현재 시간이나 난수 생성기에 의존하는 코드 역시 비결정적이다.
        - 사용자 코드가 순전히 결정적이라고 할지라도 늦은 데이터를 허용하는 이벤트 시간 집계에는 비결정적인 입력이 있을 수 있다.
- 데이터 플로우는 체크 포인트를 사용해 비결정적 처리를 효과적으로 결정적으로 바꾸어 문제를 해결한다.
    - 변환에서 나오는 각 출력은 고유 ID와 함께 다음 단계로 전달되기 전에 안정적인 저장소로 체크포인트 된다.
    - 셔플 전달 시 재시도가 일어나면 체크포인트 된 출력을 전달해 재시도 과정에서 사용자의 비결정적 코드가 실행되지 않는다.
        - 사용자의 코드는 여러 번 실행될 수 있어도 이러한 실행 중 하나만이 선택되는 것이다.
        - 데이터플로우는 안정적인 저장소에 중복 결과가 쓰이는 것을 방지하기 위해 일관적 저장소를 사용한다.

# 성능

- ‘정확히 한 번’ 보장을 제공하는 셔플 전달을 구현하기 위해 데이터 ID 목록이 각 수신자 키마다 저장된다.
    - 도착하는 모든 데이터에 대해 데이터플로우는 수신 서버 키에 저장된 데이터 ID를 확인해 중복 여부를 확인한다.
    - 단계별로 발생하는 모든 출력은 생성된 데이터 ID를 중복 감지에 안정적으로 쓸 수 있도록 저장소로 체크포인트 된다.
- 이 과정을 신중하게 구현하지 않으면 읽기/쓰기가 크게 증가해 파이프라인의 전반적인 성능이 저하된다.
    - 따라서 ‘정확히 한 번’ 보장을 위해 I/O를, 매 데이터마다 발생하는 I/O를 줄일 필요가 있다.
- 데이터플로우는 그래프 최적화 및 블룸 필터 Bloom Filter라는 두 가지 핵심 기술을 통해 이 목표를 달성하고 있다.

## 블룸 필터

- ‘정확히 한 번’ 보장 자체를 최적화하는 것을 목표로 도입되었다.
- 정상적인 파이프라인의 경우
    - 대부분의 데이터는 중복되지 않는다.
    - 블룸 필터를 사용해 성능을 크게 향상시킬 수 있다.
        - 블룸필터
            - 집합 내 요소 여부를 빠르게 검사할 수 있게 해주는 간결한 데이터 구조이다.
            - 거짓 양성 false positive는 반환할 수 있지만 절대 거짓 음성 false negative 을 반환하지는 않는다.
            - 필터가 “요소가 집합에 있다’ 라고 알려주면 요소가 실제 집합 내에 있을 가능성이 있다는 것을 의미한다.
            - 필터가 집합 내에 요소가 없다고 하면 확실히 존재하지 않는다.
- 블룸필터의 특성을 통해 다음과 같은 접근이 가능하다.
    - 데이터플로우 구현의 작동
        - 각 워커는 자신이 본 모든 ID의 블룸 필터를 유지한다.
        - 새 데이터 ID를 만날 때마다 이를 필터에서 조회한다.
            - 필터가 결과로 음성을 반환하면 이 데이터는 중복되지 않은 것이며
                - 워커 입장에서는 비용이 큰 저장소 조회를 건너뛸 수 있다.
            - 필터가 결과로 영상을 반환하는 경우에만 이 두 번째 조회를 수행할 필요가 생긴다.
                - 블룸 필터의 거짓 양성율이 낮게 유지되는 한 비용이 큰 두 번째 조회가 발생할 필요가 생긴다.
                - 블룸 필터의 거짓 양성율이 낮게 유지되는 한 비용이 큰 두 번째 조회가 발생할 가능성은 낮다.
- 블룸 필터는 시간이 지남에 따라 필터가 채워지는 경향이 있다.
    - 그럴 수록 양성율이 증가한다.
    - 워커가 재시작할 때마다 저장된 ID 목록을 스캔해 블룸 필터를 새로 구성해야 한다.
    - 데이터플로우는 각 데이터마다 시스템 타임스탬프를 붙여주고 있다.
        - 블룸 필터 하나를 만들어 사용하는 대신 10분 간격으로 개별 필터를 만들어 쓴다.
        - 데이터가 도착하면 데이터플로우는 시스템 타임스탬프를 바탕으로 적절한 필터를 쿼리하게 된다.
        - 필터가 시간이 지남에 따라 가비지 컬렉션에 의해 처리되도록 할 수 있기에 블룸 필터가 포화 상태가 되는 것을 방지하고, 시작 시 스캔돼야 하는 데이터의 양을 제한해주는 역할을 한다.
- 데이터가 시스템에 도착하고 도착 시간을 기준으로 블룸 필터에 배정된다.

    - 필터에 처음 도착하는 데이터는 중복되지 않기에 모든 목록 조회 전에 필터링된다.
        - r1은 중복해 전달되므로 실제로 중복인지 확인하기 위해 목록 조회가 필요하다.
        - r4, r6도 마찬가지이다.
        - r8은 중복되지 않는다.

## 가비지 컬렉션

- 모든 데이터플로우 워커는 자신이 본 고유 데이터 ID를 저장해둔다.
    - 데이터플로우의 상태 및 일관성 모델은 키 단위이므로
    - 실제 구현에서 각 키는 해당 키로 전달된 데이터의 목록을 저장해둔다.
    - 저장 장치의 제약으로 정보를 영구적으로 보관할 수는 없기에
        - 처리가 확인된 데이터 ID 정보를 가비지 컬렉션을 통해 정리할 필요가 있다.
- 전송 서버가 엄격하게 증가하는 일련 번호를 데이터마다 달아서 아직 전달 중인 (수신이 확인되지 않은) 가장 이른 일련 번호를 추적하는 방법
    - 그보다 앞선 모든 데이터는 이미 수신이 확인됐기 때문에 가비지 컬렉션의 대상이 될 수 있다.
- 데이터플로우는 이미 블룸 필터 버킷을 구분하기 위해 시스템 타임스탬프로 각 데이터에 표시를 해두고 있다.
    - 일련 번호를 사용해 가비지 컬렉션을 수행하는 대신 시스템 타임스탬프를 기반흐로 가비지 컬렉션 워터마크를 계산하는 것이 가능하다.
    - 이 접근 방식에서 부수적으로 얻는 이점은 이 워터마크가 특정 처리 단계에서 대기하는 데 소요된 시간을 기반으로 하기에
    - 파이프라인의 어떤 부분에 지연이 발생하는지를 직관적으로 알 수 있게 해준다.
    - 메타데이터는 데이터플로우 WebUI에 표시되는 시스템 지연 지표의 밑바탕이 된다.
- 데이터가 이미 가비지 컬렉션의 대상이 된 타임스탬프를 가지고 늦게 도착하는 경우는?
    - 이런 현상은 네트워크 파편 network remnant 으로 인해 발생하는 것으로
    - 이전에 전송한 데이터가 네트워크 상에 일정 시간 정체돼 있다가 갑자기 나타나는 현상.
    - 가비지 컬렉션을 트리거하는 낮은 워터마크는 데이터 전달이 확인될 때까지는 진행되지 않는다.
        - 뒤늦게 나타나는 데이터는 이미 성공적으로 처리됐음이 보장된다. → 무시!

# 소스에서의 ‘정확히 한 번’ 보장

- 빔은 데이터를 데이터플로우 파이프라인으로 읽어들이기 위한 소스 API를 제공한다.
    - 데이터플로우가 처리 중 실패가 발생하면 소스에서 데이터를 다시 읽어올 수 있고
    - 소스에 의해 생성된 고유한 데이터가 정확히 한 번만 처리되도록 보장해줄 필요가 있다.
- 대부분의 결정성 소스 deterministic source 에 대해 데이터플로우는 이 과정을 투명하게 처리한다.
    - 파일에서 데이터를 읽어오는 경우
        - 파일 내 데이터는 파일을 몇 번을 읽든 항상 결정된 수서로 결정된 바이트 위치에 존재한다.
        - 파일 이름과 바이트 위치만 있으면 각 데이터를 정확히 구분할 수 있다.
        - 이를 바탕으로 개별 데이터에 대한 고유한 ID를 자동으로 생성할 수 있다.
- 유사한 결정성 보장을 제공하는 또 다른 소스로는 아파치 카프카가 있다.
    - 각각의 카프카 토픽은 정적인 분할 partition 으로 나뉘며, 분할 내 데이터 순서는 항상 결정적이다.
        - 이러한 결정성 소스는 중복 없이 데이터플로우에서 원활하게 동작한다.
- 모든 소스가 그리 간단하지는 않다.
    - 데이터플로우 파이프라인에 쓰이는 일반적인 소스 중 하나는 구글 클라우드 Pub/Sub 이다.
    - Pub/Sub 는 비결정성 소스에 해당한다.
        - 여러 구독자가 Pub/Sub 토픽에서 데이터를 가져갈 수 있지만 어떤 구독자가 특정 메시지를 받을지를 예측할 수는 없다.
        - 처리에 실패할 경우 Pub/Sub 는 메시지를 다시 전달하지만 메시지가 원래 처리 방식과 다르게 다른 순서와 다른 워커에 전달될 수 있다.
        - 비결정적인 동작으로 인해 재시도 시에 안정적인 데이터 ID를 결정적으로 할당할 방법이 없기 때문에 데이터플로우가 중복을 감지하기 위해 추가적인 도움이 필요하다는 것을 의미한다.
- 데이터플로우는 데이터에 ID를 자동으로 할당할 수 없기 때문에 비결정성 소스는 데이터의 ID가 무엇이 돼야 하는지 시스템에 알려줄 필요가 있다.
    - 빔은 소스 API로 Unbounded Reader.getCurrentRecordID 메소드를 제공한다.
    - 소스에서 각 데이터에 고유 ID를 제공하고 데이터플로우에 중복 제거가 필요하다고 통보하면 동일한 ID의 데이터는 필터링된다.

# 싱크에서의 ‘정확히 한 번’ 보장

- 어느 시점에서 모든 파이프라인은 데이터를 외부로 출력해야 하며 싱크는 단순히 이러한 출력 작업을 수행하는 변형일 뿐이다.
    - 데이터를 외부로 전달하는 것은 일종의 부작용이며
    - 데이터플로우는 부작용에 대해서는 ‘정확히 한 번’ 보장을 제공하지 않는다고 이야기했다.
- 싱크가 어떻게 출력이 정확히 한 번만 일어나도록 보장할 수 있을까?
    - 빔 SDK 중 일부로 제공되는 여러 종류의 싱크를 사용하는 것이 가장 간단한 답이다.
        - 이 싱크는 여러 번 실행되더라도 중복이 생성되지 않도록 신중하게 설계돼 있다.
        - 가능하면 파이프라인을 작성할 때 이러한 기본 제공 싱크 중 하나를 사용하는 것이 좋다.
    - 기본 제공되는 싱크로 충분하지 않아 직접 작성해야 하는 경우
        - 가장 좋은 접근 방식은 작성한 부작용이 멱등성을 유지하도록 해서 반복 실행해도 문제가 없도록 하는 것이다.
        - 종종 DoFn의 일부 구성 요소는 비결정적이라 반복 실행될 때 변화가 있을 수 있다.
            - 윈도우 집계에서 한 윈도우 내의 어떤 데이터가 있는지는 비결정적일 수 있다.
    - 비결정적인 특성이 생길 수 있는 경우
        - 위험을 해결하는 표준적인 방법은 현재 데이터플로우가 DoFn 출력 중 하나만을 셔플 경계로 통과시켜 주는 보장을 활용하는 것
        - 이 보장을 이용하는 간단한 방법은 내장된 Reshuffle 변환을 통하는 것

# 사용 사례

## 소스 예시: 클라우드 Pub

- 클라우드 Pub/Sub 은 게시자의 메시지를 구독자에게 전달해주는 확장 가능하고 안정적이며 짧은 지연 시간을 자랑하는 완전 관리형 시스템
    - 게시자는 이름이 갖는 주제에 데이터를 게시하고
    - 구독자는 해당 이름으로 주제를 구독해 데이터를 가져온다.
    - 단일 주제에 대해 여러 구독을 생성할 수 있으며
    - 이 경우는 각 구독자는 구독 생성 시점부터 주제에 게시된 모든 데이터의 전체 복사본을 받는다.
    - Pub/Sub은 데이터 수신이 확인될 때까지 계속 데이터를 전달하도록 보장한다.
        - 같은 데이터가 여러 번 전달될 수 있다.
    - Pub/Sub은 분산 환경을 고려해 많은 게시 프로세스가 동일한 주제에 게시할 수 있고
        - 많은 구독 프로세스가 동일 주제를 구독해 메시지를 가져올 수 있다.
        - 데이터를 가져온 후 구독자는 일정 시간 내에 수신을 확인해줘야 하며
        - 그렇지 않으면 해당 절차는 만료돼 Pub/Sub는 같은 데이터를 다른 구독 프로세스에 다시 전송한다.
- Pub/Sub은 높은 확장성을 보이지만 데이터플로우 같은 시스템에서 사용하기에는 어려운 소스 형태
    - 어떤 데이터가 어떤 작업자에게 어떤 순서로 전달될지 알 수 없으며
    - 실패한 경우 발생하는 재전송은 서로 다른 순서로 이전과 다른 작업자에게 데이터를 보낼 수도 있다.
- Pub/Sub은 각 메시지에 메시지 ID를 배정하며 이 ID는 재전송시에도 동일하게 유지된다.
    - 데이터 플로우 Pub/Sub 소스는 기본적으로 이 ID를 사용해 Pub/Sub에서 중복 항목을 제거한다.
        - 사용자 프로스세스가 게시를 다시 시도할 수도 있으며
        - 그 결과 Pub/Sub에 중복 항목이 발생할 수 있다.
    - 서비스의 관점에서는 이 둘이 같은 데이터이기 때문에 같은 데이터 ID를 가질 것이다.
        - 데이터플로우의 Pub/Sub 소스는 사용자가 이러한 데이터 ID를 사용자 정의 속성으로 제공할 수 있도록 해준다.
        - 게시자가 재시도 시 동일한 ID를 전송해주면 데이터플로우는 중복을 감지해낼 수 있다.
- 빔도 Pub/Sub에 대한 참조 구현을 소스 형태로 제공한다.
    - 이는 데이터플로우 내에서 사용하는 것이 아니라 데이터플로우 이외의 구현에서 사용하기 위한 것
    - 다양한 이유로 인해 데이터플로우는 내부적으로 Pub/Sub를 처리하고 이 공개된 Pub/Sub 소스를 사용하지는 않는다.

## 싱크 예시: 파일

- 파일에 지속적으로 데이터를 출력하기 위해 빔의 파일 싱크를 사용할 수 있다.

## 싱크 예시: 구글 빅쿼리

- 구글 빅쿼리 Google BigQuery 는 완전 관리형의 클라우드 네이티브 데이터 웨어하우스로
    - 빔은 빅쿼리를 싱크로 제공하고 빅쿼리는 매우 짧은 지연 시간을 보장하는 스트리밍 삽입 API를 제공해준다.
    - 스트리밍 삽입 API를 사용해 삽입되는 데이터에 고유 ID를 지정할 수 있으며 빅쿼리는 동일 ID를 통해 중복된 삽입을 필터링한다.
        - 이 기능을 사용하려면 빅쿼리 싱크가 각 데이터에 대해 통계적으로 고유한 ID를 생성해줘야 한다.
        - 이를 위해 고유한 128비트 ID를 생성하는 java.util.UUID 패키지를 사용한다.
        - 임의의 UUID를 생성하는 과정은 비결정적인 작업이므로 빅쿼리에 삽입하기 전에 Reshuffle을 추가해줘야 한다.
        - 이렇게 하면 데이터플로우에서 일어나는 모든 재시도는 항상 동일한 UUID를 사용하게 된다.
        - 빅쿼리에 중복 데이터를 삽입하게 되면 이 동일한 삽입 ID를 통해 빅쿼리는 필터링을 수행한다.

# 다른 시스템

## 아파치 스파크 스트리밍

- 스파크 스트리밍은 지속적인 데이터 처리를 위해 마이크로배치 아키텍처를 사용하고 사용자는 논리적인 관점에서 스트림 개체를 처리하게 해준다.
    - 내부적으로 스파크는 이 스트림을 연속적인 일련의 RDD로 표현해낸다.
    - RDD는 배치 형태로 처리되며 스파크는 정확성을 보장하기 위해 배치 처리가 갖는 ‘정확히 한 번’ 보장에 의존한다.
    - 이 접근 방식은 딥 파이프라인과 대규모 입력의 경우 출력이 지연되는 결과가 발생할 수 있으며 원하는만큼 짧은 지연 시간을 달성하기 위해 종종 별도의 튜닝이 필요하다.
- 스파크는 모든 작업이 멱등성을 보장한다고 가정하고 있으며, 작업 그래프의 현재 지점까지 작업 체인을 반복해 작동시킬 수 있다.
    - 체크포인트 기능을 통해 RDD가 구체화되도록 해 해당 RDD 이전의 작업이 다시 작동되지 않도록 보장해준다.
    - 체크포인트 기능은 성능상의 이유를 위한 것이지만 이를 이용해 비멱등성을 갖는 부작용을 구현할 때도 사용할 수 있다.

## 아파치 플링크

- 스트리밍 파이프라인에서 ‘정확히 한 번’ 보장을 제공하지만 데이터플로우나 스파크와는 다른 방식을 사용한다.
- 플링크 스트리밍 파이프라인은 주기적으로 전체 파이프라인의 안정적인 스냅샷을 계산한다.
    - 플링크 스냅샷은 점진적으로 계산되므로 스냅샷을 계산하는 동안 모든 연산이 일시 중단될 필요가 없다.
    - 스냅샷 생성 동안 데이터가 시스템을 계속 흘러가도록 해 스파크 스트리밍 접근 방식에서 보이는 일부 지연 문제를 완화할 수 있다.
- 플링크는 소스에서 진행되는 데이터스트림에 특수한 번호가 지정된 스냅샷 마커를 삽입해 스냅샷을 구현한다.
    - 각 연산에서 스냅샷 마커를 수신하면 특정 알고리즘을 실행해 상태를 외부 위치에 복사하고
    - 스냅샷 마커를 다운스트림 연산에 전파한다.
    - 모든 과정에서 스냅샷 알고리즘을 실행하면 전체 스냅샷을 사용할 수 있게 된다.
    - 워커에 오류가 발생하면 전체 파이프라인이 마지막으로 저장된 전체 스냅샷으로 상태를 롤백한다.
        - 이때 전송 중인 메시지는 스냅샷에 포함될 필요가 없다.
        - 플링크의 모든 메시지 전달은 순서가 보장되는 TCP 기반 채널을 통해 수행된다.
        - 연결 실패는 마지막으로 정상 처리된 일련번호에서 연결을 다시 시작해 처리할 수 있다.
        - 데이터플로우와 달리 플링크에서 작업은 워커에 정적으로 할당되기 때문에 연결이 동일한 송신 서버에서 다시 시작되고 같은 데이터가 재실행된다고 가정할 수 있다.
- 플링크는 언제든지 이전 스냅샷으로 롤백할 수 있으므로 스냅샷에 아직 반영되지 않은 상태 수정은 임시적인 것으로 간주해야 한다.
    - 플링크 파이프라인 외부로 데이터를 전송하는 싱크는 스냅샷이 완료될 때까지 기다린 후 해당 스냅샷에 포함된 데이터만 전송해야 한다.
    - 플링크는 notifySnapshotComplete 콜백을 제공해 싱크가 각 스냅샷이 완료되는 시기를 알 수 있도록 하고 데이터를 계속 전송할 수 있도록 해준다.
        - 플링크 파이프라인에 출력 지연이 발생할 수 있지만 이 지연 시간은 싱크에서만 발생한다.
- 실제로 스파크는 파이프라인의 각 단계에서 배치 단위로 시간이 지연되기 때문에 플링크가 딥 파이프라인에 대해 스파크보다 종단간 지연 시간을 낮출 수 있다.
- 플링크의 분산 스냅샷은 스트리밍 파이프라인에서 일관성을 다루는 세련된 방법이다.
    - 파이프라인에 대해 여러 가지 가정을 깔아야 한다.
    - 실패 시 이전 스냅샷으로 롤백함으로 인해 발생하는 영향이 크기 때문에 실패가 드물다고 가정해야 한다.
    - 출력에서 지연 시간을 짧게 유지하기 위해 스냅샷이 빠르게 완료될 수 있다고 가정한다.
    - 초대형 클러스터에서 스냅샷 완성에 걸리는 시간이나 실패율이 증가해 문제가 되지 않는지 좀 더 살펴볼 필요가 있다.
- 작업이 워커에 정적으로 할당된다고 가정함으로써 구현도 단순하게 되어 있다.
    - 이 가정덕에 플링크는 연결 실패 시 같은 데이터를 같은 워커로부터 순서대로 가져올 수 있음을 알기 때문에 워커 간의 ‘정확히 한 번’ 보장을 제공하는 전송 계층을 간단히 제공할 수 있다.
    - 반대로 데이터플로우의 경우 작업이 워커 간에 지속적으로 분산되기 때문에 이런 가정을 만족하기 어렵다.
        - 데이터플로우는 ‘정확히 한 번’ 보장을 제공하기 위해 더 복잡한 전송 계층을 구현해야 했다.
