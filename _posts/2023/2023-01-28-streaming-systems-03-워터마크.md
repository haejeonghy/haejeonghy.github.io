# 워터마크

## 정의

- 데이터를 수집하며 결과를 지속적으로 출력하는 파이프라인에서 언제 이벤트 시간 윈도우를 닫아도 안전한지를 결정해야 한다면?
    - 파이프라인이 무한한 입력을 처리한다고 가정할 때
    1. 이벤트 시간 윈도우를 현재 처리 시간을 기반으로 둔다.
        - 단순한 접근 방법
        - 데이터 처리와 전송이 즉각 이뤄지지 않기 때문에 처리 시간과 이벤트 시간은 같을 수 없다.
        - 파이프라인 문제로 데이터가 엉뚱한 윈도우에 할당될 수 있다.
        - 할당되는 윈도우에 대해 보장을 해줄 수 없기 때문에 실패한다.
    2. 파이프라인의 메시지 처리 속도를 고려해 반영한다.
        - 직관적인 접근 방법
        - 입력의 변화, 기대하는 결과의 다양성, 처리에 쓰이는 시스템 자원 등에 영향을 받아 임의로 달라질 수 있는 지표에 의존하게 된다.
        - 진행 정도만으로는 완결성(특히 일정 시간 간격 동안 모든 입력을 봤는지 여부)에 답을 해주지 못한다.
        - 실제 시스템의 경우 시스템이 입력을 제대로 처리하지 못하는 다양한 상황이 발생할 수 있다.
            - 크래시, 네트워크 오류, 시스템 중단, 프로그램 논리상의 문제 등등
    3. 각 데이터가 논리적인 이벤트 시간 타임스탬프를 포함하고 있다고 가정한다.
        
        [스트리밍 파이프라인 내에서 처리 중 및 완료된 데이터 이벤트 시간 분포. 새로운 데이터는 입력으로 도착하고 처리될 때까지 처리 중(in-flight) 상태로 유지된다. 처리 중 분포의 가장 왼쪽 가장자리가 가장 오래된 처리 되지 않은 요소에 해당한다. ](/resource/post-format/streaming-systems-03-%EC%9B%8C%ED%84%B0%EB%A7%88%ED%81%AC/stsy_0301.mp4)
        
        스트리밍 파이프라인 내에서 처리 중 및 완료된 데이터 이벤트 시간 분포. 새로운 데이터는 입력으로 도착하고 처리될 때까지 처리 중(in-flight) 상태로 유지된다. 처리 중 분포의 가장 왼쪽 가장자리가 가장 오래된 처리 되지 않은 요소에 해당한다. 
        
        - 입력 데이터가 쉬지 않고 생성되는 상황이라 합리적이다.
        - 이벤트 시간을 포함하는 데이터를 통해 파이프라인 내에서 이벤트 시간 타임스탬프의 분포를 확인할 수 있다.
            - 파이프라인은 여러 기기를 통해 병렬로 입력 데이터를 소비한다.
            - 개별 기기 간 순서에 대한 보장은 없다.
            - 시간이 지날 수록 더 많은 데이터가 오른쪽 in-flight 상태에 놓일 것이고, in-flight 상태에 속한 데이터의 더 많은 수가 완료되면서 completed 상태에 놓일 것이다.
        - in-flight 부분의 가장 왼쪽 끝, oldest unprocessed element가 워터마크 정의에 사용된다.
- 정의
    - 워터마크는 아직 완료되지 않은 가장 오래된 작업이 갖는 단조 증가(monotinically increasing) 타임스탬프이다.
- 특징
    - 완결성 completeness
        - 워터마크가 특정 타임스탬프 T를 지나 진전이 된 경우 T를 기준으로 정시 또는 그 전에 발생한 이벤트는 만나지 않았음을 보장받는다.
        - T에서는 그 이전의 결과를 생성해도 올바르다.
        - 윈도우를 닫을 수 있는 시점을 알 수 있다.
    - 가시성 visibility
        - 메시지가 어떤 이유로든 파이프 라인 내에서 처리되지 않고 있으면 워터마크가 진행되지 않는다.
        - 워터마크 진행을 방해하는 데이터를 검토해 문제의 원인을 찾을 수 있다.

## 소스 워터마크 생성

[완벽한 워터마크와 휴리스틱 워터마크](/resource/post-format/streaming-systems-03-%EC%9B%8C%ED%84%B0%EB%A7%88%ED%81%AC/stsy_0302.mp4)

완벽한 워터마크와 휴리스틱 워터마크

- 완벽한 워터마크는 워터마크가 모든 데이터를 고려했음을 보장한다.
- 휴리스틱 워터마크는 지연 데이터가 발생할 수 있음을 인정한다.

### 완벽한 워터마크 생성

- 생성 조건
    - 워터마크가 그보다 작은 이벤트 시간을 갖는 데이터가 입력으로 들어오지 않도록 엄격하게 보장해주는 방식으로 입력 데이터에 타임 스탬프를 할당해야 한다.
    - 지연 데이터를 고려할 필요가 없다.
    - 완벽한 워터마크를 생성하려면 입력에 대한 완벽한 지식이 필요하므로 실제 대부분의 분산 입력 소스에는 적용이 어렵다.
- 완벽한 워터마크를 생성할 수 있는 사례
    1. 인입 타임스탬프 ingress timestamp
        - 시스템에 입력되는 데이터의 인입 시간을 이벤트 시간으로 설정하는 경우 완벽한 워터마크를 생성할 수 있다.
            - 이 경우 소스 워터마크는 파이프라인에서 관찰되는 처리 시간을 추적한다.
        - 2016년 이전 윈도우 지원 스트리밍 시스템이 사용했던 방법
        - 장점
            - 이벤트 시간이 단일 단조 증가 형태로 할당되기 때문에 시스템이 스트림 안에서 다음에 올 타임스탬프의 특성을 완벽하게 추정할 수 있다.
                - 이벤트 시간 진행과 윈도우 처리가 매우 쉽다.
        - 단점
            - 워터마크가 실제 데이터 자체 이벤트 시간과 무관하다.
                - 시스템이 실제 데이터 이벤트 시간을 무시하고 인입 시간만 데이터 처리의 지표로 삼기 때문이다.
    2. 시간 순으로 정렬된 정적 로그 데이터
        - 시간순 로그 데이터
            - 아파치 카프카 토픽 등
            - 각 분할에 단조 증가하는 이벤트 시간이 포함되어 있는 정적 분할에 속하는 정적 크기를 갖는 데이터
            - 워터마크 생성을 위해 입력 소스로 알려진 정적 분할 중 처리되지 않은 데이터의 가장 작은 이벤트 시간(가장 최근에 읽은 데이터의 최소 이벤트 시간)을 추적한다.
        - 시스템은 정적 분할 내 이벤트 시간이 단조 증가한다는 사실을 알고 있기 때문에 다음 타임스탬프에 대해 문제없이 추정할 수 있다.
            - 비순서 처리의 제한된 형태로 볼 수 있다.
            - 일정한 분할 내에서 순서가 뒤바뀔 수 있는 정도가 해당 분할 내에서 관찰한 최소 이벤트 시간으로 제한되는 셈
            - 보통 분할 내에서 단조 증가 타임스탬프를 보장할 수 있는 방법은 해당 분할로 데이터가 쓰여지는 시점에 타임스탬프가 할당되는 경우이다.

### 휴리스틱 워터마크 생성

- 워터마크보다 이른 이벤트 시간을 갖는 데이터를 만날 가능성이 낮다는 추정에 대한 워터마크를 생성한다.
- 휴리스틱 워터마크를 생성해 사용하는 파이프라인은 지연 데이터 처리를 고려해야 한다.
    - 지연 데이터는 휴리스틱 워터마크를 사용할 때만 발생한다.
    - 휴리스틱이 충분히 좋은 경우에는 거의 발생하지 않는다.
        - 이 때 워터마크는 완결성을 추정할 수 있는 도구 역할을 한다.
- 다수의 실질적인 분산 입력 소스에서 완벽한 워터마크를 사용하기는 불가능하다.
    - 고도로 정확한 휴리스틱 워터마크를 생성하는 건 가능하다.
- 휴리스틱 워터마크가 가능한 경우
    1. 시간 순으로 정렬된 동적 로그 데이터
        - 구조화된 동적 로그 확인
            - 개별 파일 안에서는 이벤트 시간이 단조 증가하지만 파일 간 이벤트 시간 관계는 고정되지 않는 경우
        - 예상되는 로그 파일의 전체는 실행 시점에는 알 수 없다.
        - 여러 독립적인 팀이 입력을 생성하고 관리하는 글로벌 규모의 서비스에서 흔히 볼 수 있다.
        - 입력에 대한 완벽한 워터마크를 생성하는 것은 어렵지만 정확한 휴리스틱 워터마크를 생성하는 것은 가능하다.
            - 기존 로그 파일에서 처리되지 않은 데이터의 최소 이벤트 시간을 추적하고
            - 증가율을 모니터링하고
            - 네트워크 토폴로지 및 대역폭과 같은 외부 정보를 활용해 모든 입력에 대한 완벽한 지식 없이도 정확한 워터마크를 만들 수 있다.
        - 이런 유형의 입력 데이터는 구글에서 찾을 수 있는 가장 흔한 유형의 무한 데이터셋이다.
    2. 구글 클라우드 Pub/Sub 
        - Pub/Sub은 순서를 보장하며 전달되지 않는다.
        - 단일 게시자가 2개의 메시지를 순서대로 게시하더라도 가능성은 낮지만 순서가 뒤바뀔 수 있다.
            - 사용자 개입 없이 처리량을 높이기 위해 확장성을 갖추려는 아키텍처의 동적인 특성 연유
            - 클라우드는 Pub/Sub에서 완벽한 워터마크를 보장할 방법이 없다.
            - 하지만 정확한 휴리스틱 워터마크를 구축한 바 있다.
- 소스에 대해 더 많이 알슈록 휴리스틱이 향상되고 더 적은 지연 데이터를 만나게 된다.
- 소스의 유형, 이벤트 분산 및 사용 패턴이 다양하다는 점을 고려하자면 모든 경우에서 사용할 수 있는 해결책은 없다.
    - 완벽한 워터마크든 휴리스틱이든 입력 소스에서 워터마크가 생성된 후 시스템은 파이프라인을 통해 워터마크를 문제 없이 전파할 수 있다.
    - 완벽한 워터마크는 이후 처리 시스템에서도 완벽한 상태를 유지한다.
    - 휴리스틱 워터마크는 설정했을 때와 마찬가지로 휴리스틱으로 유지된다.
- 파이프라인 전체에서 완결성을 추적해야 하는 복잡성을 입력 소스에서 워터마크를 생성하는 문제로 좁혀서 해결할 수 있게 된다.

## 워터마크 전파

- 대부분의 실제 파이프라인은 여러 단계로 구성된다.
    - 파이프라인 단계
        - 파이프라인이 어떤 새로운 기준으로 데이터를 집계 하려고 할 때마다 파이프라인에 단계를 도입해야 한다.
- 워터마크는 입력 소스에서 생성된다.
- 개념적으로 워터마크도 시스템을 따라 흐른다.
    - 다양한 수준으로 세분화해 워터마크를 추적할 수 있다.
        - 여러 단계로 구성된 파이프라인의 경우 각 단계마다 입력과 이전 단계에 의존하는 자신만의 워터마크를 추적한다.
            - 따라서 파이프라인에서 뒤에 나오는 단계일수록 앞에 놓인 단계보다 입력의 적은 부분을 볼 수 밖에 없기 때문에 과거의 워터마크를 사용하게 된다.
- 파이프라인의 단일 작업, 즉 단계의 경계에서 워터마크를 정의할 수 있다.
    - 이를 통해 파이프라인의 각 단계의 진행 정도를 파악할 수 있다.
    - 개별 단계마다 가능한 빠르게 독립적인 결과를 낼 수 있다.
    - 단계의 정의에서 다음과 같은 워터마크를 정의할 수 있다.
        - 입력 워터마크 input watermark
            - 해당 단계의 모든 업스트림 진행 상황을 담고 있다.
            - 소스의 경우 입력 워터마크는 입력 데이터의 워터마크를 생성하는 소스 고유의 함수로 볼 수 있다.
            - 소스가 아닌 단계의 경우 입력 워터마크는 모든 업스트림 소스와 해당 단계의 모든 샤드/분할/인스턴스의 출력 워터마크 중 최솟값으로 정의된다.
        - 출력 워터마크 output watermark
            - 해당 단계의 진행 정도를 담는다.
            - 해당 단계의 입력 워터마크와 단계 내 처리 중인 모든 비지연 데이터의 이벤트 시간 중 최솟값으로 정의된다.
                - ‘처리 중’의 의미는 주어진 단계가 실제 수행하는 작업과 스트림 처리 시스템의 구현에 따라 달라진다.
                    - 보통 집계를 위해 버퍼링되어 있으나 아직 다운스트림으로 구체화되지 않은 데이터와 다운스트림 단계로 전달 중인 출력 데이터 등을 포함한다.
- 특정 단계의 입출력 워터마크를 정의함에 있어 유용한 특징
    - 해당 단계로 인해 발생하는 이벤트 시간 지연을 계산할 수 있다.
        - 한 단계의 입력 워터마크 값에서 출력 워터마크 값을 빼면 해당 단계로 인해 발생하는 이벤트 시간 지연을 알 수 있다.
            - 지연은 각 단계의 출력이 실시간으로 얼마나 늦어지는지 보여준다.
- 입출력 워터마크의 정의는 파이프라인 전체에서 워터마크의 순환적인 관계를 보여준다.
- 파이프라인 내 연이은 단계는 해당 단계의 이벤트 시간 지연에 따라 필요한만큼 워터마크를 지연시킨다.
    - 각 단계 안에서 처리 과정 역시 단일 요소로 구성되어 있지 않다.
    - 한 단계 내 처리 과정을 여러 개념적 요소를 갖는 흐름으로 분할할 수 있다.
    - 각각의 요소가 출력 워터마크에 영향을 준다.
    - 구성 요소의 정확한 특성은 단계가 수행하는 작업과 구현 방법에 따라 달라진다.
    - 개념적으로 이런 구성 요소는 작업이 완료될 때까지 진행 중인 작업이 머무르는 버퍼 역할을 한다.
        - 데이터가 도착해 처리를 위해 버퍼링 될 때 이는 나중에 지연된 집계 작업을 위해 기록된다.
        - 집계 작업이 시작되면 다운스트림 단계에서 소비할 때까지 대기하기 위해 출력 버프로 결과가 쓰여진다.
        - 각 버퍼마다 자체 워터마크를 갖도록 해 버퍼의 진행 정도를 추적할 수 있다.
        - 각 단계의 버퍼가 갖는 워터마크의 최소가 해당 단계의 출력 워터마크를 만들게 된다.
            - 출력 워터마크는 다음 워터마크의 최소가 된다.
                - 데이터를 보내는 단계에서는 소스별 워터마크
                - 파이프라인 외부 소스에 대해서는 외부 입력별 워터마크
                - 기록될 수 있는 각 상태에 대해서는 상태별 구성 요소 워터마크
                - 데이터를 수신하는 단계에 대해서는 출력 버퍼별 워터마크
            - 이렇게 세분화해 워터마크를 사용할 수 있도록 하면 시스템 동작을 가시적으로 파악하기 쉬워진다.
                - 워터마크를 통해 시스템 내 다양한 버퍼를 거쳐가는 데이터의 위치를 추적하고, 어디서 지연이 발생하는지 쉽게 진단이 가능하다.

### 워터마크 전파 이해하기

- 사용자의 게임에 참여 시간이 게임을 얼마나 즐기는지 보여준다는 가정을 세우고
- 사옹자별 세션 길이를 계산하는 작업을 수행할 때
- 세션 길이를 계산하기 위해 “무엇, 어디서, 언제, 어떻게”라는 네 가지 질문에 맞춰 파이프라인을 구성한다.
- 고정된 시간 내 평균 세션 길이를 구하기 위해 질문에 맞춰 파이프라인을 구성한다.
- 모바일용 점수와 콘솔용 점수로 두 데이터셋으로 작업한다고 가정할 때
    - 두 독립적인 데이터셋에 병렬로 정수 합을 통한 계산을 동일하게 수행한다.
    - 한 파이프라인에서는 모바일용 점수를 계산하고, 다른 파이프라인에서는 콘솔용 점수를 계산한다.
    - 두 단계로 구성된 두 파이프라인이 동일한 작업을 수행하지만 다른 데이터에 대해 수행돼 출력 워터마크가 달라진다.

모바일 및 콘솔 게임 세션의 평균 길이

- 두 파이프라인은 모바일 및 콘솔의 개별 세션 길이를 계산한다.
    - 그리고 계산된 세션 길이가 파이프라인의 두번째 단계로 전달돼 고정 윈도우 안에서 전체 세션 길이의 평균이 계산된다.
- 모바일 세션과 콘솔 세션 단계의 출력 워터마크는 적어도 각각 대응하는 입력 워터마크만큼 늦고, 실제로는 조금 더 늦게 된다.
    - 이는 실제 시스템에서 계산 응답 시간에 시간이 걸리고
    - 주어진 입력에 대한 처리가 완료될 때까지 출력 워터마크가 진행되지 않기 때문이다.
- 평균 세션 길이 단계의 입력 워터마크는 두 단계의 출력 워터마크 중 최솟값이다.
    - 다운스트림 입력 워터마크는 업스트림 출력 워터마크 중 최소를 뜻한다.
    - 업스트림 단계가 이를 뒤따르는 단계보다 시간상 앞서기때문에 멀리 있는 다운스트림 워터마크일수록 더 과거를 다룬다.

### 워터마크 전파와 출력 타임스탬프

- 한 윈도우에 대한 유효한 타임스탬프 범위는 윈도우에서 가장 오래된 지연 없는 데이터의 타임스탬프에서 시작하고 무한대까지 확장될 수 있다고 추정할 수 있다.
- 이 때 선택 가능한 많은 방법이 있지만 대부분의 상황에서 의미 있는 선택은 다음 몇 가지로 제한된다.
1. 윈도우의 끝
    1. 출력 타임스탬프가 윈도우 경계를 나타내도록 하려면 윈도우 끝을 사용하는 것이 유일하게 안전한 방법이다.
    2. 가능한 방법 중 워터마크가 가장 매끄럽게 진행되도록 해준다.
2. 첫 번째 비지연 데이터의 타임스탬프
    1. 워터마크를 최대한 보수적으로 유지할 수 있다. 
    2. 그로 인해 워터마크 진행에 방해가 될 수 있다는 단점이 있다. 
3. 특정 데이터의 타임스탬프
    1. 특정한 경우 다른 임의 데이터의 타임스탬프가 올바른 선택이 되기도 한다. 
    2. 특정 쿼리에 대한 스트림과 해당 쿼리 결과에 발생한 클릭 이벤트의 스트림을 조인하는 경우
        1. 조인을 수행한 후 어떤 시스템은 쿼리의 타임스탬프가 더 유용하다고 판단하기도 하며
        2. 다른 시스템은 클릭 이벤트의 타임스탬프로 선호하기도 한다. 
        3. 데이터가 늦게 도착하지 않는 이상 워터마크의 정확성 관점에서든 어느 경우든 유효하다.
- 출력 타임스탬프의 선택이 전체 파이프라인에 주는 영향

### 까다로운 겹치는 윈도우의 경우

- 출력 타임스탬프와 관련된 중요한 문제는 슬라이딩 윈도우를 처리하는 방법이다.
- 출력 타임스탬프를 초기 데이터로 설정하는 단순한 접근 방식은 워터마크가 보류돼 다운스트림 지연이 발생할 가능성이 크다.
- 동일한 유형의 슬라이딩 윈도우를 사용하는 두 단계의 파이프라인을 살펴볼 때
    - 데이터가 연이은 윈도우 3개로 배치된다고 가정하는 경우
        - 입력 워터마크가 진행됨에 따라 슬라이딩 윈도우가 다음과 같이 행동할 것을 기대한다.
            - 첫 단계의 첫 윈도우가 완료되면 다운스트림으로 보낸다.
            - 두 번째 단계에서 첫 윈도우가 완료되면 다운스트림으로 보낼 수 있다.
            - 이후 첫 단계에서 두 번쨰 윈도우가 완료되는 식으로 작업이 이어질 것이다.
        - 패널의 첫 비지연 데이터의 타임스탬프를 출력 타임스탬프로 선택한다면 실제로는 다음과 같은 일이 일어난다.
            - 첫 단계의 첫 윈도우가 완료되면 다운스트림으로 보낸다.
            - 두 번째 단계의 첫 윈도우는 입력 워터마크가 업스트림 두 번째 및 세 번째 윈도우의 출력 워터마크에 의해 붙잡혀있기 때문에 완료될 수 없다.
                - 가장 초기에 있는 데이터의 타임스탬프가 해당 윈도우의 출력 타임 스탬프로 쓰이고 있기 때문에 이러한 워터마크의 행동은 올바른 것으로 볼 수 있다.
            - 첫 단계의 두 번째 윈도우가 완료되면 다운스트림으로 보낸다.
            - 두 번째 단계의 첫 번째 그리고 두 번째 윈도우는 업스트림의 세 번째 윈도우에 의해 보류돼 완료될 수 없다.
            - 첫 단계의 세 번째 윈도우가 완료되면 다운스트림으로 보낸다.
            - 두 번째 단계의 세 윈도우 모두는 완료되어 이후 단계로 보낼 수 있다.
        - 이와 같은 윈도우의 행동은 결과를 구체화시키는 시점을 불필요하게 지연시키게 한다.
            - 아파치 빔은 겹치는 윈도우에 대해서는 N+1번째 윈도우의 출력 타임스탬프가 항상 N번째 윈도우의 끝 지점보다 커지도록 하는 특별한 로직을 적용하고 있다.

## 처리 시간 워터마크

- 이벤트 시간 워터마크만 검사해서는 1시간 전의 데이터를 신속하고 지체 없이 처리하는 시스템과 실시간 데이터를 처리하다가 1시간 동안 지연이 발생한 시스템을 구분할 수 없다.
    - 이런 구분을 위해서 처리 시간 워터마크 processing-time watermark 가 필요하다.
- 스트림 처리 시스템은 단계 간의 데이터 셔플 shuffle, 데이터를 영구적인 저장 상태 persistent state 에서 읽어오거나 쓰는 행동, 또는 워터마크 진행에 맞춰 지연된 집계를 시작하는 식의 작업을 지속적으로 수행한다.
    - 이러한 모든 작업은 파이프라인의 현재 또는 업스트림 단계에서 수행된 이전 작업에 대한 응답으로 수행된다.
    - 데이터 요소가 시스템을 통해 흐르는 것처럼 이러한 요소를 처리하는 일련의 작업도 시스템을 통해 흐르게 된다.
- 처리 시간 워터마크는 이벤트 시간 워터마크를 정의한 것과 동일한 방식으로 정의한다.
    - 아직 완료되지 않은 가장 오래된 작업의 이벤트 시간 타임스탬프 대신 아직 완료되지 않은 가장 오래된 연산의 처리 시간 타임스탬프를 사용한다.
    - 처리 시간 워터마크에 발생하는 지연의 예로는 한 단계에서 다른 단계로 메시지가 전달되지 못하는 상태나, 상태나 외부 데이터를 읽기 위한 입출력이 진행되지 못하거나 처리가 완료되지 못하도록 발생하는 예외 상황 등이 있을 수 있다.
- 처리시간 워터마크는 데이터에서 발생하는 지연과는 별개로 처리가 지연되는 상황을 감지할 수 있는 방법이 있다.

## 사례 연구

### 사례 연구: 구글 클라우드 데이터플로우의 워터마크

- 데이터플로우는 데이터 처리 그래프 내에 있는 데이터 처리 단계를 여러 물리적인 워커로 분산하며, 각 워커는 일정 키 범위가 배정된다. GroupByKey 연산이 수행될 때마다 데이터는 해당 키로 셔플되어야 한다.

![GroupByKey 단계는 다른 DoFn로부터 데이터를 소비한다. ](/resource/post-format/streaming-systems-03-%EC%9B%8C%ED%84%B0%EB%A7%88%ED%81%AC/stsy_0316.png)

GroupByKey 단계는 다른 DoFn로부터 데이터를 소비한다. 

![두 단계의 키 범위는 작업에 참여하는 워커에 분산돼 할당된다.](/resource/post-format/streaming-systems-03-%EC%9B%8C%ED%84%B0%EB%A7%88%ED%81%AC/stsy_0317.png)

두 단계의 키 범위는 작업에 참여하는 워커에 분산돼 할당된다.

- 데이터플로우는 각 단계의 요소마다 범위별 워터마크를 추적한다.
    - 특정 조건을 충족되도록 하며 워터마크 취합 watermark aggregation 과정을 통해 전 범위에 대한 워터마크의 최소를 계산한다.
        - 모든 범위는 반드시 워터마크를 보고한다. 워터마크가 없으면 해당 범위는 알 수 없는 것으로 취급돼 워터마크가 진행될 수 없다.
        - 워터마크가 단조 증가하도록 한다. 지연 데이터로 인해 워터마크가 반대로 이동하게 되는 상황이라면 워터마크를 업데이트하지 않는다.
- 데이터플로우는 중앙화된 요소를 통해 워터마크 취합을 진행한다.
    - 효율을 위해 분산될 수 있다.
    - 정확성의 관점에서 워터마크 취합은 워터마크에 대한 source of truth 역할을 한다.
- 분산된 환경에서 워터마크 취합을 정확하게 하려면 해결해야 할 문제가 있다.
    - 워터마크가 조기에 진행되면 제시간에 도착한 데이터가 지연 데이터로 처리되므로 워터마크가 조기에 진행되지 않도록 하는 것이 중요하다.
        - 워커마다 작업이 물리적으로 분산 할당되기에 단일 워커만이 한 키의 영구적인 저장 상태를 수정할 수 있도록
            - 특정 키 범위에 대한 영구적인 상태는 해당 키 범위를 담당하는 워커에게만 일임돼야 한다.
        - 워터마크의 정확성을 위해 워커에서 유래한 워터마크 갱신 내용은 워커가 자신에게 주어진 영구적인 저장 상태를 여전히 담당하고 있을 때에만
            - 워터마크를 취합하는 쪽으로 전송돼야 한다.
        - 워터마크 갱신 프로토콜은 상태 소유권에 대한 유효성 확인도 반드시 고려해야 한다.

### 사례 연구: 아파치 플링크의 워터마크

- 아파츠 플링크는 고성능의 항상 가용하고 정확한 분산 데이터 스트리밍 앱을 위한 오픈 소스 스트림 처리 프레임워크이다.
    - 플링크 러너를 사용해 빔 프로그램을 실행할 수 있다.
    - 플링크상에서 실행될 떄 빔은 스트림 처리 개념의 플링크 내 구현에 의존한다.
    - 워터마크는 데이터와 같은 흐름 내에서 추적하고 취합한다.
        
        ![두 소스를 갖는 플링크 파이프라인](/resource/post-format/streaming-systems-03-%EC%9B%8C%ED%84%B0%EB%A7%88%ED%81%AC/stsy_0318.png)
        
        두 소스를 갖는 플링크 파이프라인
        
- 데이터는 두 소스에서 생성된다.
    - 데이터 스트림과 같은 경로상에 동기적으로 전송되는 워터마크 checkpoint를 함께 생성한다.
        - A에서 타임스탬프 53에 대한 워터마크 체크포인트가 생성되면 A에서는 이전의 타임스탬프를 갖는 비지연 데이터는 발생하지 않았음을 보장한다.
- 다운스트림의 KeyBy 연산은 입력 데이터와 워터마크 체크포인트를 소비한다.
    - 새로운 워터마크 체크포인트가 소비됨에 따라 다운스트림 연산의 워터마크도 진행된다.
    - 다운스트림 연산들을 위한 새로운 워터마크 체크포인트가 차례로 생성될 수 있다.
- 데이터스트림과 함께 내보내는 워터마크의 장점
    - 줄어든 워터마크 전파 지연 및 현저히 낮은 워터마크 지연
        - 워터마크가 네트워크 상에서 여러 번 전송된 후에 중앙에서 취합되는 것을 기다릴 필요가 없다.
            - 워터마크 지연을 현저히 낮출 수 있다.
    - 워터마크 집계에 대한 단일 실패 지점 없음
        - 한 곳에서 워터마크 취합을 하는 경우 문제가 발생하면 전체 파이프라인에서 워터마크 지연이 일어난다.
            - 데이터와 같은 경로를 사용하면 파이프라인 일부를 사용할 수 없어도 전체 파이프라인에서 워터마크 지연이 일어나지는 않는다.
    - 타고난 확장성
        - 중앙집중식으로 워터마크를 취합하면서 확장성을 유지하는 것이 데이터와 같은 흐름으로 워터마크를 보내며 얻는 확장성에 비하면 더 복잡하다.
- 데이터 흐름과 별도로 워터마크를 취합할 때의 장점
    - 단일 진실 공급원
        - 디버깅이나 모니터링, 파이프라인 진행에 맞춰 입력을 조절하고자 하는 경우
            - 등에 워터마크를 따로 다루는 것이 데이터스트림 안에 워터마크를 포함시키는 것보다 유리하다.
                - 이 경우 시스템의 각 부분이 부분적인 정보만을 가지고 있게 되기 때문이다.
    - 소스 워터마크 생성
        - 일부의 경우 소스 워터마크 생성에는 시스템 전반에 걸친 정보가 필요하다.
        - 이 경우 별도로 구성된 중앙 집중식 요소를 통하면 훨씬 다루기가 쉽다.

### 사례 연구: 구글 클라우드 Pub/Sub의 소스 워터마크

- 클라우드 Pub/Sub을 통해 파이프라인으로 전송되는 데이터에 대해 합리적인 휴리스틱 워터마크를 생성하는 방법
- 메시지는 Pub/Sub 토픽에 게시되고 여러 곳에서 특정 토픽을 구독할 수 있다.
    - 토픽으로 메시지를 보내면 해당 토픽을 구동 중인 모든 곳으로 같은 메시지가 전달된다.
    - 클라이언트는 메시지를 당겨서 가져가고
    - 제공된 ID를 통해 특정 메시지를 받았음을 확인해주게 된다.
    - 어떤 메시지를 가지고 갈지는 클라이언트가 정하지 못한다.
- 워터마크를 위한 휴리스틱을 만들기 위해 Pub/Sub 으로 보내지는 데이터에 가정을 할 때
    - 원본 데이터의 타임스탬프가 정상 동작한다고 가정할 필요가 있다.
        - 정상 동작은 Pub/Sub 으로 보내지기 전에 소스 데이터의 타임 스탬프가 순서를 벗어남에 있어 제한이 있다는 의미
        - 허용된 제한 범위 밖으로 순서가 뒤바뀐 타임스탬프를 갖는 모든 데이터는 지연 데이터로 간주된다.
        - 현재 이 제한 범위는 10초
            - Pub/Sub 으로 보내기 전 10초 내의 타임스탬프를 갖는 데이터는 지연 데이터가 되지 않는다.
            - 추정 대역 estimation band
            - 파이프라인이 완벽하게 입력을 따라잡고 있다고 할 때 워터마크는 데이터의 순서가 바뀔 경우를 고려해 실시간보다 10초 늦게 쫓아가고 있다는 뜻
            - 파이프라인 백로그를 사용하는 경우 워터마크 추정을 위해 추정 대역인 10초 범위뿐만 아니라 전체 백로그를 사용한다.
- Pub/Sub 은 순서를 보장하지 않으므로 백로그에 대해 충분히 알기 위해서 추가적인 메타데이터가 있어야 한다.
    - Pub/Sub 은 클라이언트로부터 수신 여부가 미확인된 가장 오래된 게시 타임 스탬프 publish timestamp를 통해 백로그를 추적할 수 있게 해준다.
    - Pub/Sub 은 애플리케이션에서 사용하는 메타데이터에는 무관심하기 때문에 전송할 메시지의 이벤트 타임스탬프와는 다르다.
        - Pub/Sub에서 메시지를 받은 시간을 표시하는 타임스탬프로 볼 수 있다.
        - Pub/Sub 메시지 전달을 위한 처리 시간 워터마크에 가깝다.
        - Pub/Sub게시 타임 스탬프는 이벤트 타임 스탬프와 다르고
        - 오래전의 데이터가 전송되는 경우 해당 메시지의 이벤트 타임스탬프와 게시 타임스탬프의 차이가 클 수 있다.
        - 메시지의 순서가 뒤바뀌는 것을 허용했기 때문에 둘의 순서 역시 다를 수 있다.
        - 하지만 게시 타임스탬프를 백로그 추적을 위해 사용해 백로그에 존재하는 이벤트 타임스탬프에 대한 충분한 정보를 알고 합리적인 워터마크를 만들 수 있다.
        - 입력 메시지를 포함하는 토픽마다 기본 구독 base subscription 과 추적 구독 traking subscription 구독 2개를 생성한다.
            - 기본 구독은 파이프라인이 실제 처리될 데이터를 읽는 목적으로 사용하며
            - 추적 구독은 워터마크 추정을 위해 메타데이터를 위한 목적으로 사용한다.
            - 기본 구독의 예
                
                ![Pub/Sub 구독에 도착한 메시지의 처리 시간 및 이벤트 시간 타임스탬프](/resource/post-format/streaming-systems-03-%EC%9B%8C%ED%84%B0%EB%A7%88%ED%81%AC/stsy_0319.png)
                
                Pub/Sub 구독에 도착한 메시지의 처리 시간 및 이벤트 시간 타임스탬프
                
                - pt, et 이벤트 기간 타임스탬프는 서로 전혀 무관하다.
                - 기본 구독의 일부 메시지는 클라이언트로부터 수신이 확인 되지 않아 백로그를 구성하게 된다.
                    - 이는 해당 메시지가 전달되지 않았거나 전달됐지만 미처리 상태임을 의미한다.
                    - 이 구독에서 메시지를 가져간느 것은 여러 샤드로 분산될 수 있다.
                    - 기본 구독만 봐서는 워터마크가 무엇이 돼야 할지 알 수 없다.
            - 추적 구독의 예
                
                ![기본 구독과 동일한 메시지를 수신하는 추가적인 추적 구독](/resource/post-format/streaming-systems-03-%EC%9B%8C%ED%84%B0%EB%A7%88%ED%81%AC/stsy_0320.png)
                
                기본 구독과 동일한 메시지를 수신하는 추가적인 추적 구독
                
                - 추적 구독은 기본 구독의 백로그를 효율적으로 조사해 백로그 내에서 이벤트 타임스탬프의 최소를 찾는다.
                - 추적 구독이 메시지를 기본 구독보다 빠르게 따라잡도록 해 기본 구독의 가장 오래된 미확인 메시지보다 더 최근의 메시지들을 조사할 수 있다.
                - 추적 구독으로부터 데이터를 가져가는 비용을 줄여서 추적 구독이 빠르게 메시지들을 따라잡도록 할 수 있다.
                    - 만약 추적 구독이 늦어지는 상황이 발생하면 워터마크 진행을 일시중단 할 수 있다.
                    - 이를 위한 조건은
                        - 추적 구독이 기본 구독보다 충분히 빠를 때
                            - 추적 구독이 최소한 추정 대역만큼은 앞서야 한다.
                            - 추정 대역 내에서 순서가 뒤바뀐 경우를 고려할 수 있게 된다.
                        - 추적 구독이 실시간에 충분히 가까울 때
                            - 추적 구독에 대한 백로그는 없게 된다.
                - 메시지의 게시 타임스탬프와 이벤트 타임스탬프에 대한 메타 정보를 안전하게 저장한 후 최대한 빠르게 추적 구독 내 메시지를 확인한 것으로 처리한다.
                    - 이 메타데이터는 저밀도 히스토그램 sparse histogram 형식으로 저장돼 저장 공간과 안전하게 기록해야 하는 데이터의 크기를 최소화한다.
            - 기본 구독에서 가장 오래된 미확인 메시지보다 최신인 게시 타임스탬프를 갖는 메시지의 이벤트 타임스탬프 범위나 추정 대역 너비만큼을 추적 구독에서 확인한다.
                - 백로그의 모든 이벤트 타임스탬프, 백로그가 작은 경우엔 가장 최근 추정 대역을 고려해 워터마크 추정에 사용할 수 있다.
                - 워터마크 값은 앞서 취해진 범위 내 최소 이벤트 시간으로 결정된다.
                - 입력에서 10초라는 제한 범위 내 모든 타임스탬프를 워터마크 계산에 고려해 지연 데이터로 삼지 않는다는 점에서 정확히 동작한다.
                    - 하지만 지나치게 보수적인 워터마크를 생성해 너무 느려질 수 있음
            - 워터마크 진행을 원활하게 하기 위한 휴리스틱
                - 메시지가 빈번하게 도착하는 경우에 잘 동작하지만
                - 그렇지 않은 경우라면 합리적인 추정을 하기에 부족할 수 있다.
                - 2분이 넘도록 구독에 데이터가 보이지 않는다면
                    - 워터마크를 실시간 근처로 진행시킨다.
                - 메시지가 더이상 오지 않는 상황에서도 워터마크와 파이프라인은 계속 진행될 수 있다.
