# Streaming Systems: 02. 데이터 처리의 무엇을, 어디서, 언제, 어떻게
Created: 2023년 1월 14일 오후 4:47
보관소: No
최종 편집 일시: 2026년 3월 14일 오전 1:08

- 데이터 처리 패턴에 대해 좀 더 자세하고 구체적인 예

# 로드맵

- 1장 요약
    - 이벤트가 실제 발생한 이벤트 시간과 처리 중 관측된 시간인 처리 시간을 구분하는 것이 중요하다.
        - 분석 결과의 정확성과 이벤트가 발생한 문맥이 중요한 경우라면 처리 시간이 아닌 이벤트 시간을 바탕으로 데이터를 분석해야 한다.
    - 윈도우
        - 시간 경계를 기준으로 데이터를 나누는 방법
        - 무한 데이터를 다룰 때 사용하는 통상적인 방법
        - 고정 윈도우(fixed window) & 슬라이딩 윈도우(sliding window), 세션
- 2장
    - 트리거(trigger)
        - 어떤 외부 신호로 윈도우가 출력되는 시점을 선언하는 방법
        - 트리거를 통해 유연하게 결과가 생성되는 시점을 결정할 수 있다.
        - 결과가 생성되는 시기를 위한 흐름 제어 방법(flow control mechanism)의 일종
        - 결과 계산 중 실제 사진(snapshot)을 찍어야 하는 시점 정할 수 있게 함
        - 윈도우의 결과를 여러 번에 걸쳐 확인할 수 있게 해줌
        - 시간이 흐르면서 결과가 점차 정제될 수 있게 해줌
        - 예측 결과(speculative result)를 낼 수 있게 해줌
        - 시간에 따른 업스트림 데이터의 변화나 지연돼 도착되는 데이터를 다룰 수 있음
    - 워터마크(watermark)
        - 이벤트 시간을 기준으로 입력이 완료됐음을 표시하는 방법
        - 워터마크는 무한 데이터를 관찰할 때 진행 정도의 지표로 생각할 수 있음
    - 누적(accumulation)
        - 같은 윈도우 내에서 관찰되는 여러 결과 사이의 관계 명시
            - 서로 완전히 독립적일 수도 있고 겹치는 부분이 있을 수 있음
    - 무슨 결과가 계산되어야 하는가?
        - 파이프라인에서 쓰는 변형(transformation)의 종류
    - 이벤트 시간의 어디에서 결과가 계산돼야 하는가?
        - 파이프라인에서 이벤트 시간 윈도우의 사용으로 결정
            - 일반적인 윈도우 형태 포함
            - 사용자가 데이터의 인입 시간을 이벤트 시간으로 할당할 경우 처리 시간 윈도우가 될 수 있음
    - 처리 시간의 언제 결과가 구체화되는가?
        - 트리거와 워터마크
    - 결과 사이의 관계가 어떻게 되는가?
        - 누적 형태가 결정
        - 무시 모드(discarding mode)
            - 모든 결과가 독립적이며 구분되는 모드
        - 누적 모드(accumulating mode)
            - 이후 결과가 이전 결과 위에 놓이는 모드
        - 누적 및 철회 모드(accumulating and retracting mode)
            - 누적 값과 이전에 트리거 된 값에 대한 철회가 함께 출력되는 모드

# 배치 처리의 기본: 무엇과 어디서

## 무엇: 변환

- 고전 적인 배치 처리에 적용되는 변환은 “무슨 결과가 계산되는가?” 라는 질문에 답을 준다.
- 9개의 값으로 구성된 간단한 데이터에서 키를 갖는 정수 합 계산하는 예 사용
    - Score: 이 이벤트와 관련된 개별 사용자의 점수
    - EventTime: 점수의 이벤트 시간 즉, 점수가 발생한 시간
    - ProcTime: 점수의 처리 시간 즉, 파이프라인에서 점수가 관찰된 시간

        - PCollections: 대용량일수도 있는 데이터셋을 표현하며, 이 데이터셋에는 병렬 변환(parallel transformation) 적용이 가능하다.
        - PTransforms: PCollections에 적용돼 PCollections를 생성하는 연산.
            - 요소 단위 변환을 수행할 수 있다.
            - 여러 요소를 집계(aggregation)/그룹핑하거나
            - 다른 PTransforms의 조합된 형태일 수 있다.

                - Grouping(그룹핑 변환)
                - Composite(조합 변환)

                    - 별도의 윈도우 변환을 적용하고 있지 않기 때문에 이벤트 시간 전반에 대한 합을 계산한다.

## 어디서: 윈도우

- 윈도우는 데이터 소스를 시간 경계에 따라 자르는 작업이다.

# 스트리밍으로 전환: 언제와 어떻게

- 이상적인 모습은 결과가 출력될 때까지 지연은 짧되 무한 데이터 입력을 다룰 수 있는 방법
    - 이를 위해 스트리밍 시스템으로 전환하는 것이 올바르다.
    - 하지만 결과 출력 전 입력을 모두 소비할 때까지 기다리는 전략을 쓸 수 없다.

## 언제: 트리거가 좋은 이유는 트리거가 좋기 때문이다!

- 트리거는 “처리 시간의 언제 결과가 구체화되는가?” 라는 질문에 답을 준다.
- 트리거는 처리 시간에서 언제 윈도우의 결과가 생성돼야 하는지를 선언한다.
    - 트리거 자체는 결과 출력 시점 결정을 이벤트 시간에서 워터마크 처리 같은 다른 시간 영역에서 일어나는 일에 기반해 내릴 수 있다.
- 개념적으로 두 종류의 트리거가 유용하며 실제 상황에서는 거의 항상 이 둘 중 하나나 둘이 조합된 형태가 쓰인다.
    - **반복 업데이트 트리거(repeated update trigger)**
        - 값이 변함에 따라 주기적으로 윈도우를 위해 업데이트 된 패널을 생성한다.
            - 업데이트는 새로운 데이터가 들어올 때마다, 혹은 1분에 한 번처럼 일정 시간 이후에 발생할 수 있다.
        - 반복 업데이트 트리거의 동작 주기를 통해 출력의 지연 정도와 비용 사이의 조율이 가능하다.
        - 스트리밍 시스템에서 가장 빈번하게 만나는 트리거
        - 구현이 간단하고 이해가 쉽다.
        - 데이터베이스에서 구체화 뷰(materialized view)와 유사한 형태로 구체화된 데이터셋에 반복된 업데이트를 적용하는 경우에 유용하다.)
        - 구현
            - 코드
            
            ```java
            // 새로운 데이터가 도착할 때마다 동작하는 트리거
            PCollection<KV<Team, Integer>> totals = input
            	.apply(Window.into(FixedWindows.of(TWO_MINUTES))
            		.triggering(Repeatedly(AfterCount(1))))
            	.apply(Sum.integersPerKey());
            ```
            
            - 동작하는 형태
                
                [스트리밍 시스템에서 데이터 단위 트리거](/resource/post-format/streaming-systems-02-%EB%8D%B0%EC%9D%B4%ED%84%B0-%EC%B2%98%EB%A6%AC%EC%9D%98-%EB%AC%B4%EC%97%87%EC%9D%84-%EC%96%B4%EB%94%94%EC%84%9C-%EC%96%B8%EC%A0%9C-%EC%96%B4%EB%96%BB%EA%B2%8C/stsy_0206.mp4)
                
                스트리밍 시스템에서 데이터 단위 트리거
                
                - 매 입력마다 결과가 생성된다.
                - 윈도우 하나에 여러 출력 결과(패널)이 나온다.
        - 이런 형태의 트리거는 결과가 일정한 테이블에 쓰이고 주기적으로 테이블을 폴링해 결과를 확인하고자 할 때 적합하다.
        - 테이블을 확인할 때마다 주어진 윈도우의 가장 최근 업데이트 된 결과가 담겨있다.
            - 이 값들은 시간이 지남에 따라 원하는 최종 결과에 수렴한다.
        - 단점
            - 너무 자주 결과가 생성된다.
                - 대용량 데이터를 처리하는 경우 합을 계산하는 집계 과정은 정보 손실 없이 스트림의 규모를 줄일 수 있는 기회가 된다.
                    - 일정 키에 속하는 데이터 규모가 큰 경우 유용하다.
        - 장점
            - 속해있는 데이터의 규모가 큰 키/윈도우의 영향을 그렇지 않은 경우와 균등하게 맞출 수 있다.
                - 결과로 나오는 스트림은 팀의 규모와 무관하게 균일한 크기로 생성될 것이다.
    - **완료 트리거(completeness trigger)**
        - 윈도우 내 일정 기준 완료 됐다고 믿는 시점 이후에 윈도우를 위한 패널을 생성한다.
        - 이 형태의 트리거는 입력이 완료되면 결과를 생성한다는 점에서 배치 처리에서 봤던 것과 유리하다.
        - 트리거를 기반으로 처리할 때 차이점은 입력 완료의 개념이 전체 입력 완료가 아닌 단일 윈도우 문맥으로 제한된다.
        - 반복 업데이트 트리거보다 덜 쓰이지만 고전 배치 처리에 더욱 가까운 형태로 스트리밍 처리를 가능하게 해준다.
        - 누락되거나 지연된 데이터에 고려가 가능하도록 해준다.
    - 트리거에 처리 시간 지연을 도입하는 방법
        - 정렬 지연(aligned delay)
            - 키와 윈도우 전반에 걸쳐 고정된 영역으로 처리 시간이 나뉜다.
            - 코드
                
                ```java
                PCollection<KV<Team, Integer>> totals = input
                	.apply(Window.into(FixedWindows.of(TWO_MINUTES))
                		.triggering(Repeatedly(AlignedDelay(TWO_MINUTES))
                	.apply(Sum.integersPerKey());
                ```
                
                [2분 단위 정렬 지연 트리거(마이크로배치)](/resource/post-format/streaming-systems-02-%EB%8D%B0%EC%9D%B4%ED%84%B0-%EC%B2%98%EB%A6%AC%EC%9D%98-%EB%AC%B4%EC%97%87%EC%9D%84-%EC%96%B4%EB%94%94%EC%84%9C-%EC%96%B8%EC%A0%9C-%EC%96%B4%EB%96%BB%EA%B2%8C/stsy_0207.mp4)
                
                2분 단위 정렬 지연 트리거(마이크로배치)
                
            - 장점
                - 예측 가능성
                - 수정이 필요한 모든 윈도우에서 동시에 정기적으로 업데이트가 발생한다.
            - 단점
                - 모든 업데이트가 한 번에 일어나기 때문에 적절한 부하처리를 위해 더 큰 최대치를 다룰 수 있도록 프로비저닝(provisioning)이 필요하다.
                    - 이 대안으로 비정렬 지연을 사용할 수 있다.
        - 비정렬 지연(unaligned delay)
            - 한 윈도우 안에서 관찰된 데이터를 기준으로 지연된다.
            - 코드
                
                ```java
                PCollection<KV<Team, Integer>> totals = input
                	.apply(Window.into(FixedWindows.of(TWO_MINUTES))
                		.triggering(Repeatedly(UnalignedDelay(TWO_MINUTES))
                	.apply(Sum.integersPerKey());
                ```
                
                [2분 단위 비정렬 지연 트리거](/resource/post-format/streaming-systems-02-%EB%8D%B0%EC%9D%B4%ED%84%B0-%EC%B2%98%EB%A6%AC%EC%9D%98-%EB%AC%B4%EC%97%87%EC%9D%84-%EC%96%B4%EB%94%94%EC%84%9C-%EC%96%B8%EC%A0%9C-%EC%96%B4%EB%96%BB%EA%B2%8C/stsy_0208.mp4)
                
                2분 단위 비정렬 지연 트리거
                
                - 한 윈도우 내 실제 지연은 둘 사이에 다소 차이가 날 수 있지만 최종 평균치는 본질적으로 동일하다.
                    - 비정렬 지연이 시스템 부하를 고르게 분산시킨다는 점에서 대규모 처리를 위해 더 좋은 선택이라고 볼 수 있다.
    - 반복 업데이트 트리거는 언제 결과의 정확성이 달성되는지에 대한 기준 없이 정확한 쪽으로 수렴한다는 사실만으로 충분하고, 결과가 주기적으로 업데이트되기를 원할 때 매우 유용하다.
        - 분산 시스템에서는 다양한 이유로 이벤트가 발생한 시간과 시스템에 관찰되는 시간 사이에 편차가 발생한다.
            - 결과가 입력 데이터에 대해 정확하고 온전한 관점을 언제부터 제시하기 시작하는지 추정하기 어렵다.

## 언제: 워터마크

- 워터마크의 역할
    - 입력 완결성이 중요하지만 막연한 추정보다는 완결성에 대해 추론할 수 있는 방법
- “처리 시간의 언제 결과가 구체화되는가?”
    - 이 질문에 대한 답을 보조하는 역할
- 워터마크는 이벤트 시간 영역에서 입력 완결성을 표현하는 시간적 개념
    - 이벤트 시간 영역에서 입력 완결성을 표현하는 시간적 개념
    - 이벤트 스트림에서 처리중인 데이터의 진행 정도와 완결 여부를 이벤트 시간과 관련해 결정하는 방법
    - 이벤트 시간과 처리 시간 사이의 편차는 대부분의 실제 분산 처리 시스템에서 계속 변하는 함수 관계
        
        ![이벤트 시간 진행 상황, 편차, 워터마크(빨간 선)](/resource/post-format/streaming-systems-02-%EB%8D%B0%EC%9D%B4%ED%84%B0-%EC%B2%98%EB%A6%AC%EC%9D%98-%EB%AC%B4%EC%97%87%EC%9D%84-%EC%96%B4%EB%94%94%EC%84%9C-%EC%96%B8%EC%A0%9C-%EC%96%B4%EB%96%BB%EA%B2%8C/Untitled%203.png)
        
        이벤트 시간 진행 상황, 편차, 워터마크(빨간 선)
        
        - 개념적으로 워터마크는 이벤트 시간의 한 지점에서 처리 시간의 한 지점으로의 함수인 F(P)→ E 형태로 이해할 수 있다.
            - 이벤트 시간의 한 지점 E가 갖는 의미는 E 이전의 이벤트 시간을 갖는 모든 입력을 이미 시스템이 봤다고 믿는다.
- 워터마크의 종류
    - 완벽한 워터마크(perfect watermark)
        - 입력 데이터를 완벽하게 이해하고 있다면 완벽한 워터마크 구축이 가능해진다.
            - 이 경우 모든 데이터는 정시에 도착하고, 당연히 지연 데이터는 존재하지 않는다.
    - 휴리스틱 워터마크(heuristic watermark)
        - 여러 곳에서 분산돼 들어오는 입력의 경우 입력 데이터를 완벽하게 이해하는 것이 현실적으로 불가능하다. → 최선의 선택은 휴리스틱 워터마크가 된다.
        - 입력에 대해 모든 가능한 정보(분할 정보, 분할 내 순서, 파일의 증가 비율 등)을 고려해 최대한 정확한 진행 상황을 추정하는 것
            - 다수의 경우 추정이 제법 정확하다.
            - 하지만 지연 데이터가 발생할 수 있다.
- 워터마크는 입력에 대한 완결성을 결정하기 때문에 완결성 트리거(completeness trigger)의 밑바탕이 된다.
    - 워터마크 완결성 트리거
    - 코드
        
        ```java
        PCollection<KV<Team, Integer>> totals = input
        	.apply(Window.into(FixedWindows.of(TWO_MINUTES))
        		.triggering(AfterWatermark()))
        	.apply(Sum.integerPerKey());
        ```
        
- 워터마크는 여러 함수의 집합이다.
    - **워터마크의 속성을 만족하는 F(P)→E 형태의 함수가 성공의 정도를 조금씩 달리하며 여럿 존재할 수 있다.**
    - 입력 데이터에 대해 완벽하게 알고 있는 상황이면 완벽한 워터마크를 구축할 수 있다.
        - 가장 이상적인 해결책이 된다.
    - 하지만 완벽한 워터마크를 계산하는 것이 비용적으로 어렵거나 해서 입력에 대해 완벽하게 알 수 없는 상황에서는 휴리스틱을 사용한다.
    - 워터마크에 쓰이는 알고리즘은 파이프라인 자체와는 독립적이다.
    - 파이프라인에 서로 다른 두 종류의 워터마크 구현을 적용한 결과
        
        [왼쪽은 완벽한 워터마크 / 오른쪽은 휴리스틱 워터마크](/resource/post-format/streaming-systems-02-%EB%8D%B0%EC%9D%B4%ED%84%B0-%EC%B2%98%EB%A6%AC%EC%9D%98-%EB%AC%B4%EC%97%87%EC%9D%84-%EC%96%B4%EB%94%94%EC%84%9C-%EC%96%B8%EC%A0%9C-%EC%96%B4%EB%96%BB%EA%B2%8C/stsy_0210.mp4)
        
        왼쪽은 완벽한 워터마크 / 오른쪽은 휴리스틱 워터마크
        
        - 워터마크가 윈도우의 끝을 통과할 때 윈도우가 구체화된다.
            - 완벽한 워터마크는 시간이 흐르면서 파이프라인의 이벤트 시간 완결성을 제대로 잡아낸다.
            - 휴리스틱 워터마크에 사용된 알고리즘은 입력값 9를 놓치레 된다.
                - 결과적으로 출력 지연이 발생하고 결과의 정확성이 떨어지면서 구체화된 출력의 모양이 크게 바뀐다.
        - 가장 큰 차이점은 워터마크 트리거가 입력의 완결 여부에 대해 판단할 수 있는 방법을 제공한다.
            - 시스템이 주어진 윈도우를 출력으로 구체화하기 전까지 시스템이 아직 입력이 완결됐다고 판단하지 않는다.
                - 입력에서 데이터 결핍을 판단할 필요가 있는 경우에 기준이 될 수 있다.
- 데이터 손실 판단이 중요한 사례 중 하나는 외부 조인이다.
    - 워터마크 같은 완결성 개념 없이는 언제까지 조인이 완결되기를 기다렸다가 포기하고 부분 조인을 출력할지 결정할 방법이 없다.
    - 워터마크 지원이 제대로 되지 않는 스트리밍 시스템에서 자주 쓰이는 처리 시간 지연을 바탕으로 결정하는 방식?
        - 이벤트 시간 편차가 가변적일 수 있기 때문에 안전한 방법이 아니다.
            - 선택된 처리 시간 지연보다 편차가 작으면 문제가 없지만, 커지면 결과가 엉망이 된다.
            - 이런 관점에서 이벤트 시간 워터마크는 외부 조인, 이상 탐지(anomaly detection)처럼 입력의 결핍에 대한 결정을 내려야 하는 스트리밍 사례에 필수적인 요소
- 워터마크의 단점
    - 너무 느림
        - 워터마크가 알려진 미처리된 데이터를 기다리느라 정상적으로 지연되는 경우, 결과를 내는 기준이 워터마크의 진행 정도에만 의존하고 있다면 출력에 직접적으로 지연이 발생한다.
        - 워터마크가 완결성에 대한 유용한 개념을 제공하지만 결과 생성을 위해 완결성에 의존하는 것이 항상 이상적이지는 않다.
    - 너무 성급함
        - 휴리스틱 워터마크가 오판으로 실제 기다려야 하는 것보다 먼저 진행된다면 워터마크 이전의 이벤트 시간을 갖는 데이터가 뒤늦게 도착할 수도 있다.
            - 지연 데이터가 발생한다.
        - 휴리스틱은 언제든지 틀릴 수 있다.
            - 출력을 구체화하는 결정에 휴리스틱을 사용하면 정확성을 고려했을 때 충분하지 않은 결과가 나올 수 있다.

## 언제: 조기/정시/지연 트리거

- 표준 워터마크 트리거 확장
    - 워터마크 트리거의 앞뒤로 반복 업데이트 트리거를 도입할 수 있도록
    - 복합된 트리거에 의해 구체화되는 패널을 세 종류로 나온다.
- 조기/정시/지연 트리거
    - 생략 가능한 조기 패널
        - 워터마크가 윈도우 끝을 지나기 전에 주기적으로 작동하는 반복 업데이트 트리거의 결과
        - 이 트리거로 생성되는 패널은 예측 결과를 담고 있지만
            - 입력 데이터가 도착하면서 시간에 따라 윈도우가 변하는 모습을 관찰할 수 있게 해준다.
            - 워터마크가 너무 느려질 수 있는 단점을 보완해준다.
    - 단일 정시 패널
        - 완결성/워터마크 트리거가 윈도우 끝을 통과한 후에 동작하는 결과물
        - **시스템은 윈도우의 입력이 완료됐다고 믿을 수 있다.**
            - 덕분에 누락된 데이터에 대한 판단이 가능해 외부 조인을 수행할 때 부분 조인을 출력하거나 하는 행동이 가능해진다.
    - 생략 가능한 지연 패널
        - 워터마크가 윈도우 끝을 통과한 후에 주기적으로 트리거 되는 반복 업데이트 트리거의 결과물
            - 이 트리거와 조기 패널에 쓰인 트리거가 동일할 필요는 없다.
        - 완벽한 워터마크의 경우 지연 패널은 의미가 없다.
        - 휴리스틱 워터마크의 경우 늦게 도착하는 데이터가 있다면 지연 패널이 발생한다.
            - 너무 성급하게 동작하는 워터마크의 단점을 보완해준다.
- 코드
    
    ```java
    PCollection<KV<Team, Integer>> totals = input
    	.apply(Window.into(FixedWindows.of(TWO_MINUTES))
    		.triggering(AfterWatermark()
    			.withEarlyFirings(AlignedDelay(ONE_MINUTE))
    				.withLateFirings(AfterCount(1))))
    	.apply(Sum.integersPerKey());
    ```
    
    [조기, 정시, 지연 트리거를 사용한 스트리밍 시스템에서의 윈도우 합](/resource/post-format/streaming-systems-02-%EB%8D%B0%EC%9D%B4%ED%84%B0-%EC%B2%98%EB%A6%AC%EC%9D%98-%EB%AC%B4%EC%97%87%EC%9D%84-%EC%96%B4%EB%94%94%EC%84%9C-%EC%96%B8%EC%A0%9C-%EC%96%B4%EB%96%BB%EA%B2%8C/stsy_0211.mp4)
    
    조기, 정시, 지연 트리거를 사용한 스트리밍 시스템에서의 윈도우 합
    
    - 두 번째 윈도우에서 워터마크가 너무 느린 경우는 주기적으로 이뤄진느 1분 단위 조기 업데이트를 통해 해결되고 있다.
        - 완벽한 워터마크의 경우에 더 분명하다.
    - 휴리스틱 워터마크가 너무 성급한 경우 늦게 도착하는 9가 바로 새롭게 수정된 패널에 반영되고 올바르게 수정된 14를 결과로 낸다.
    - 완벽한 워터마크와 휴리스틱 워터마크를 사용한 두 경우의 차이가 좁혀진다.
    - 워터마크 트리거를 사용하기 때문에 완결성에 대한 판단을 내릴 수 있다.
        - 데이터 누락을 고려할 필요가 있는 외부 조인, 이상 감지 등의 상황도 잘 다룰 수 있다.
- 윈도우의 수명
    - 완벽한 워터마크와 휴리스틱 워터마크 사이의 가장 큰 차이
    - 완벽한 워터마크에서는 워터마크가 윈도우 끝을 통과하고 나면 더이상 해당 윈도우를 위한 데이터가 남아있지 않음을 알게 되어 윈도우와 관련된 모든 상태 정보를 버릴 수 있다.
    - 휴리스틱의 경우 지연 데이터의 가능성이 있기 때문에 일정 시간 윈도우 정보를 유지해야 한다.
        - 하지만 현재로서는 시스템이 얼마나 긴 시간 각 윈도우의 상태를 유지해야 하는지 알 방법이 없다.

## 언제: 허용된 지연 범위(가비지 컬렉션)

- 휴리스틱 워터마크를 사용했을 때 각 윈도우의 상태가 마지막까지 유지된다.
    - 지연 데이터를 적절하게 처리하기 위해 필요하지만 무한 데이터를 다루고 있다면 윈도우 상태를 영구적으로 보존해야 한다.
        - 디스크 공간이 부족해진다.
- 비순서 처리 시스템은 윈도우 수명을 제한할 방법이 필요하다.
    - 시스템 내에 허용된 지연 범위를 정의해줘서 해결한다.
        - 워터마크를 기준으로 입력 데이터가 얼마나 지연될 수 있는지 한계를 둔다.
        - 이 한계 이후에 도착하는 데이터는 무시된다.
        - 개별 데이터가 얼마나 늦을 수 있는지 제한하면 윈도우 상태를 얼마나 오래 유지해야 하는지 결정할 수 있다.
            - 워터마크가 윈도우 끝마다 설정된 지연 범위를 넘어설 때까지만 윈도우 상태를 유지하게 한다.
            - 시스템이 한계를 지나 도착한 데이터를 즉각 버리도록 해 버려질 데이터를 처리하기 위해 자원을 낭비하지 않도록 해준다.
- 허용된 지연 범위를 갖는 조기/정시/지연 트리거
    - 코드
        
        ```java
        PCollection<KV<Team, Integer>> totals = input
        	.apply(Window.into(FixedWindows.of(TWO_MINUTES))
        		.triggering(
        			AfterWatermark()
        				.withEarlyFirings(AlignedDelay(ONE_MINUTE))
        				.withLateFirings(AfterCount(1)))
        		.withAllowedLateness(ONE_MINUTE))
        	.apply(Sum.integersPerKey());
        ```

## 어떻게: 누적

- 하나의 윈도우에 대해 트리거가 여러 패널을 결과로 생성하는 경우 결과 사이의 관계가 어떻게 되는가?
- 누적 모드(accumulation mode)
    - 무시(discarding)
        - 패널이 구체화 될 때마다 이전에 저장된 값을 더하거나 하지 않고 무시한다.
            - 각각의 패널이 전에 존재하던 패널과 독립적인 값을 담고 있음 의미
        - 데이터를 받아 처리하는 시스템이 누적 작업을 스스로 하고자 할 때 유용하다.
            - 예) 이후 시스템으로 차이값을 보내고 이후 시스템에서 이를 받아 합을 구하는 식
    - 누적(accumulating)
        - 패널이 구체화 될 때마다 이전에 저장된 상태가 보존되고 기존 상태와 함께 새로운 값이 결정된다.
            - 각각의 패널이 갖는 값이 이전 패널 값을 바탕으로 형성됨 의미
        - 이후의 결과가 이전 결과를 간단히 덮어쓰는 형태일 때 유용하다.
        - HBase, Bigtable 과 같이 키/값 저장소에 결과를 출력하는 경우에 해당한다.
    - 누적 및 철회(accumulating and retracting)
        - 누적 모드와 유사하지만 새 패널을 생성할 때 이전 패널과 독립적인 철회 정보도 함께 생성한다.
        - 이 철회 정보는 새로 생성된 누적 결과와 함께 이전의 결과를 철회하는 정보를 담고 있다.
            - 현재 시스템에서 생성된 데이터를 소비한 후 시스템이 결과를 다른 기준으로 다시 그룹핑할 때 새로 생성된 값이 이전 값과 다른 키로 묶여 다른 그룹에 속할 수 있다.
                - 이 경우 새 값이 단순하게 이전 값을 덮어 씌울 수는 없고 이전 값을 지우기 위한 명확한 정보가 필요하다.
            - 세션과 같은 동적 윈도우에서는 윈도우 머지가 발생하기 때문에 새 값 하나가 이전 값 여러 개를 대체할 수 있다.
                - 이 경우 새 값만으로는 어떤 이전 값이 대체되는 지에 대한 결정을 내리기가 쉽지 않다.
                - 이때 이전 값에 대한 철회 정보가 있다면 작업이 쉬워진다.
