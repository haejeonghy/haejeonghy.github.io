# Streaming Systems: 01. 스트리밍 101
Created: 2023년 1월 8일 오후 2:33
보관소: No
최종 편집 일시: 2026년 3월 14일 오전 1:08

# Intro

오늘날 빅데이터 분야에서 스트리밍 데이터 처리가 주목받고 있다. 

1. 데이터에 대한 시기적절한 통찰력을 얻을 수 있다. 
2. 스트리밍 시스템을 사용하면 대규모의 무한 데이터셋을 쉽게 처리할 수 있다. 
3. 데이터가 도착하는 시점에 처리하면 시스템의 작업 부하가 고르게 분산된다. 그로 인해 더 일관되고 예측 가능한 형태로 시스템 자원을 소비할 수 있다. 

# 용어: 스트리밍이란?

- 스트리밍 시스템
    - 무한 데이터셋을 염두에 두고 설계된 데이터 처리 엔진의 유형
        - 이 정의는 진정한 의미의 스트리밍은 물론 마이크로배치 구현도 포함된다.
            - 마이크로배치
                - 무한 데이터를 처리하기 위해 배치 처리 엔진을 반복 실행하는 스트리밍 시스템
                - 스파크 스트리밍(Spark Streaming)이 해당된다.
- 데이터 셋의 모양을 정의하는 차원
    - 기수(Cardinality)
        - 데이터셋의 크기
        - 지정된 데이터 집합이 유한인지 무한인지를 결정한다.
            - 유한 데이터(bounded data)
                - 크기가 유한한 데이터셋 유형
            - 무한 데이터(unbounded data)
                - (최소한 이론적으로) 크기가 무한한 데이터셋 유형
        - 무한 데이터셋의 무한함이라는 특성 때문에 데이터 처리 프레임워크가 부담을 갖게 된다.
    - 구성(Constitution)
        - 데이터의 물리적 표현
        - 데이터셋과 상호 작용할 수 있는 방법 정의
        - 대표적으로 두 가지 형태가 있다.
            - 테이블(table)
                - 특정 시점의 데이터셋에 대한 전체적인 뷰 의미
                - SQL 시스템은 전통적으로 테이블을 바탕으로 동작해왔다.
            - 스트림(stream)
                - 시간에 따라 변화하는 데이터셋의 요소 단위 뷰(element-by-element view).맵리듀스(map-reduce) 계통의 데이터 처리 시스템은 전통적으로 스트림을 바탕으로 동작해왔다.
    - 두 속성은 직교적(orthogonal)이다.

## 심하게 과장된 스트리밍의 한계

- 역사적으로 스트리밍 시스템은 종국엔 올바른 결과를 내는 배치 시스템과 함께 동작하며 낮은 지연 시간으로 부정확한 예측 결과를 제공하는 틈새 시장으로 밀려나 있었다.
    - 람다 아키텍처(Lambda Architecture)
    - 동일한 계산을 수행하는 배치 시스템과 스트리밍 시스템을 함께 운영하는 것
    - 스트리밍 시스템은 낮은 지연 시간으로 부정확한 결과를 제공하고, 얼마 후 배치 시스템이 최종적으로 정확한 결과를 보여주는 식
        - 스트리밍 시스템은 근사 알고리즘을 사용하거나 시스템 자체가 정확성을 보장하기 어렵기 때문이다.
    - 독립된 두 버전의 파이프라인을 구축하고 유지해야 하며, 최종적으로 두 파이프라인에서 나온 결과를 병합해야 하기 때문에 유지하기 어렵다.
- “람다 아키텍처에 대하 의문” by Jay Kreps
    - 이중으로 파이프라인을 구동할 필요가 없다.
    - 재현 가능한 시스템(replayable system)을 사용하는 상황(Kafka 등) 에서 반복 가능성(repeatability)에 대한 문제 논의
    - 주어진 작업을 위해 잘 설계된 단일 파이프라인을 적절하게 구축해 실행하는 카파 아키텍처 제안
- 필자: “배치 시스템은 기능상 잘 설계된 스트리밍 시스템의 엄격한 부분집합에 불과하다.”
    - 오늘날의 배치 시스템은 더 이상 쓸 이유가 없다고 주장.
    - 무한 데이터 처리를 위해 강력한 프레임워크와 결합된 스트리밍 시스템이 광범위하게 성숙해가는 중.
    - 람다 아키텍처는 빅데이터 역사에서 유물이 될 것이다.
- 스트리밍 시스템이 배치 시스템을 이기기 위해 필요한 것
    - 정확성(correctness)
        - 일관성을 제공할 수 있는 스토리지(consistent storage)로 귀결
            - 해당 스토리지는 장애 등의 상황에서 일관성을 유지할 수 있도록 설계되어야 한다.
        - 시간이 지남에 따라 영구 상태를 유지할 수 있는 체크포인트 방법이 필요하다.
        - 강한 일관성은 ‘정확히 한 번 처리(exactly-once-processing)’ 방식을 지원하기 위해 반드시 필요하다.
        - 정확성을 제공하기 위해서는 ‘정확히 한 번’ 방식이 필요하다.
        - 정확성은 최소한 배치 시스템의 능력 이상을 가지고자 하는 시스템이 갖추어야 하는 필수 요소이다.
            - 시스템 처리 결과의 정확성이 중요한 상황이라면 강력한 일관성을 제공하지 않는 스트리밍 시스템은 피해야 한다.
    - 시간 판단 도구(tools for reasoning about time)
        - 스트리밍 시스템이 배치의 능력을 뛰어 넘을 수 있게 해준다.
        - 다양한 이벤트 시간 왜곡이 발생하는 상황에서 무한 비순서 데이터(unbounded unordered data)를 처리할 때 반드시 필요하다.
        - 오늘 날 점점 더 많은 데이터가 이러한 특성을 보인다.
        - 기존 배치 시스템은 그런 특성으로 인한 어려움을 해결할 수 있는 도구가 부족하다.
- 배치와 스트리밍 효율성 차이
    - 현재 배치와 스트리밍 사이의 효율성 차이는 배치 시스템에서 분산 서버 간 데이터 전송 단위가 크고, 셔플 전송(shuffle transport)이 더 효율적이어서 나온 결과로 볼 수 있다.
    - 오늘날 배치 시스템은 적은 컴퓨팅 리소스로 높은 처리량을 보일 수 있도록 최적화 되어있다.
        - 이 기법을 무한 데이터 처리를 위한 시스템에 도입할 수 있다.
    - 배치 시스템(사용자에게 긴 대기 시간과 높은 효율성 제공)과 스트리밍 시스템(낮은 대기 시간과 낮은 효율성 제공) 사이에서 유연하게 선택할 수 있다.
    - Google cloud dataflow에서 통합된 단일 모델을 사용해 배치와 스트리밍이 모두 동작하도록 제공하는 것으로 확인할 수 있다.

## 이벤트 시간 대 처리 시간

- 모든 데이터 처리 시스템에는 두 가지 형태의 시간 영역이 있다.
    1. 이벤트 시간(event time)
        1. 이벤트가 실제 발생한 시간
        2. 시간에 따른 사용자 행동의 특성을 분석하거나 결제 관련 서비스, 이상 징후 감지 등과 같은 경우에 중요하다. 
    2. 처리 시간(processing time): 이벤트가 처리 시스템에서 관측된 시간
- 이상적으로는 이벤트 시간 == 처리 시간이 되는 것이지만, 다음과 같은 요소로 시간 차이가 생길 수 있다.
    - 공유된 리소스로 인한 제약
        - 네트워크의 혼잡 정도, 네트워크 파티션, 공유 환경에서 쓰이는 CPU 등 포함
    - 분산 시스템의 로직, 경쟁 상황 같은 소프트웨어 상의 원인
    - 데이터 자체의 특성
        - 키 분산, 전달되는 데이터 양이나 순서의 변화 등

![실제 시스템에서 이벤트 시간과 처리 시간이 변화하는 모습은 붉은 선과 같은 형태가 된다. ](/resource/post-format/streaming-systems-01-%EC%8A%A4%ED%8A%B8%EB%A6%AC%EB%B0%8D-101/Untitled.png)

실제 시스템에서 이벤트 시간과 처리 시간이 변화하는 모습은 붉은 선과 같은 형태가 된다. 

- 처리 시간
    - 이상적인 선과 붉은 선 사이의 수직 거리는 처리 시간 영역에서의 지연을 의미한다.
    - 이벤트가 발생한 시간과 처리되는 시간 사이에서 얼마나 지연이 발생했는지를 의미한다.
    - 두 유형의 왜곡 중 더 자연스럽고 직관적으로 이해할 수 있는 것에 해당한다.
- 이벤트 시간
    - 이상적인 선과 붉은 선 사이의 수평 거리는 해당 시점의 파이프라인에서 이벤트 시간의 왜곡을 보여준다.
    - 이벤트 시간 관점에서 현재 파이프라인이 이상적인 라인으로부터 얼마나 늦어지고 있는지 알려준다.
- 실제로 처리 시간에서 발생하는 지연과 이벤트 시간에서 발생하는 왜곡은 한 지점을 놓고 보면 동일한 크기이다.
    - 같은 현상을 바라보는 서로 다른 관점이다.
- 이벤트 시간과 처리 시간 사이의 관계가 고정되어 있지 않고 시간에 따라 변화한다.
    - 이벤트 시간이 중요한 경우라면 파이프라인 관측 시점만으로는 데이터를 분석할 수 없다.
    - 이 방식이 역사적으로 무한 데이터를 위해 설계된 많은 시스템이 동작해온 방식이다.
    - 무한 데이터셋의 무한함이라는 특성을 다루기 위해 이런 시스템은 일반적으로 인입되는 데이터를 윈도우(window) 라는 단위로 잘라 다룰 수 있는 개념을 제공한다.
        - 윈도우(window)
            - 일정 시간 경계에서 데이터셋을 유한한 크기의 조각으로 자르는 것
            - 정확성과 이벤트 시간이 주요한 경우라면 처리 시간(처리 시간 윈도우 processing-time-window)를 사용해 이 경계를 설정해서는 안 된다.
                - 그렇지 않을 경우 처리 시간과 이벤트 시간 사이의 어떤 일관적 관계 없이는 이벤트 시간 데이터 중 일부가 잘못된 처리 시간 윈도우로 들어가 정확성에 문제가 발생한다.
- 이벤트 시간으로 윈도우를 나눈다고 장밋빛 미래가 보장되지는 않는다.
    - 무한 데이터의 순서가 보장되지 않고, 가변적인 왜곡이 발생할 수 있다는 사실은 이벤트 시간 윈도우의 완결 시점(completeness)을 결정할 수 없다는 이야기가 된다.
    - 처리 시간과 이벤트 시간 사이의 예측 가능한 관계가 없다면 주어진 이벤트 시간 X에 대한 데이터가 완료됐다고 장담할 수 없다.
    - 다수의 데이터 처리 시스템은 데이터의 완결 시점에 의존해 동작하며, 무한 데이터셋에 적용될 경우 심각한 문제가 발생한다.
- 복잡한 데이터셋이 갖는 불확실성을 다룰 수 있도록 도구를 설계해야 한다.
    - 새로운 데이터가 도착하면 이전 데이터를 철회하거나 갱신할 수 있어야 한다.
    - 완결 시점이라는 개념을 시스템 전반에 적용되는 일반적인 개념 보다는 특정 경우를 위한 편리한 최적화로 간주해야 한다.

전반적인 모델 자체는 매우 단순하다. 

## 무한 데이터: 배치

배치 엔진은 무한 데이터를 염두에 두고 설계되지 않았지만 배치 시스템이 처음 고안된 이후에 무한 데이터셋 처리에도 쓰이고 있다. 

→ 접근 방식에서는 무한 데이터를 배치 처리에 적합한 유한 데이터셋의 집합으로 분할하는 것이 중요하다. 

### 고정 윈도우

![고전적인 배치 엔진에서 임시 방편의 (ad hoc) 고정 윈도우를 통한 무한 데이터 처리. 무한 데이터셋은 유한한 고정 크기의 윈도우로 미리 수집된 다음 배치 엔진의 반복 실행을 통해 처리된다. ](/resource/post-format/streaming-systems-01-%EC%8A%A4%ED%8A%B8%EB%A6%AC%EB%B0%8D-101/Untitled%202.png)

고전적인 배치 엔진에서 임시 방편의 (ad hoc) 고정 윈도우를 통한 무한 데이터 처리. 무한 데이터셋은 유한한 고정 크기의 윈도우로 미리 수집된 다음 배치 엔진의 반복 실행을 통해 처리된다. 

- 입력 데이터를 고정된 크기의 윈도우(fixed window, tumbling window)로 나눈 후 각 윈도우를 분리된 유한 데이터 소스인 것처럼 처리하는 것
    - 배치 엔진을 반복 실행해 무한 데이터셋을 처리하는 가장 일반적인 방법
    - 이벤트가 디렉터리나 파일 계층 구조로 기록되는 로그 등의 입력 소스인 경우 디렉터리나 파일 이름을 윈도우의 기준으로 삼을 수 있다.
        - 이 경우 입력 데이터가 자신이 속할 이벤트 시간 윈도우로 적절히 분류되어 있는 것과 다름이 없어 간단하다.
- 하지만 데이터 입력이 완결됐음을 알리는 완결성(completeness) 문제가 남아있다.
    - 이벤트를 전송하는 과정에서 네트워크 파티션으로 일부가 지연되는 경우에는 어떻게 하지?
    - 이벤트를 전체적으로 수집해 처리 전 공통 위치로 전송해야 하는 경우에는?
    - 이벤트가 모바일 장치에서 발생한다면?
    
    → 모든 이벤트 수집을 확인할 때까지 처리를 지연하거나, 데이터가 늦게 도착하면 소속될 윈도우에 대해 전체 배치를 재실행하는 등의 방법으로 문제의 영향을 완화할 필요가 발생한다. 
    

### 세션

![고전적인 배치 엔진에서 임시 방편의 고정 윈도를 통해 무한 데이터를 세션으로 처리하는 모습. 무한 데이터셋은 유한한 고정 크기의 윈도우로 미리 수집된 다음 배치 엔진의 연속 실행을 통해 동적인 세션으로 분리된다. ](/resource/post-format/streaming-systems-01-%EC%8A%A4%ED%8A%B8%EB%A6%AC%EB%B0%8D-101/Untitled%203.png)

고전적인 배치 엔진에서 임시 방편의 고정 윈도를 통해 무한 데이터를 세션으로 처리하는 모습. 무한 데이터셋은 유한한 고정 크기의 윈도우로 미리 수집된 다음 배치 엔진의 연속 실행을 통해 동적인 세션으로 분리된다. 

- 무한 데이터를 세션(session)처럼 더 정교한 윈도우 전략으로 분리하면 접근 방식을 더 상세하게 나눌 수 있다.
    - 세션은 보통 세션의 활동(activity) 기간으로 정의되며, 일정 크기의 세션의 비활동 간격(inactivie gap)이 따라오면 끝나는 것으로 볼 수 있다.
- 일반적인 배치 엔진을 사용해 세션을 계산하면 여러 배치로 나뉘어진 세션이 나타날 수 있다.
    - 배치 크기를 늘리면 분할되는 세션의 크기를 줄일 수 있지만 처리까지의 지연 시간이 늘어나게 된다.
    - 이전 배치에 나온 세션을 묶어주기 위한 로직을 추가하는 방법도 있지만 이 경우 복잡도가 증가한다.
- 고전적인 배치 엔진을 사용해 세션을 계산하는 것은 바람직하지 않다.

## 무한 데이터: 스트리밍

스트리밍 시스템은 무한 데이터 처리를 위해 제작됐다. 

- 스트리밍 시스템이 다루는 데이터의 특성
    - 무한 데이터
    - 이벤트 시간 기준으로 심하게 뒤섞여 이벤트 시간을 기준으로 처리하려면 파이프라인에서 시간 기반의 셔플을 수행해야 하는 데이터
    - 다양한 형태의 이벤트 타임 왜곡이 발생해 상수 시간 범위 안에 주어진 이벤트 시간 X의 대부분이 데이터를 봤다고 확신할 수 없는 데이터
- 이와 같은 특성의 데이터를 다룰 때 취할 수 있는 접근법

### 시간 무시(time-agnostic)

- 시간이 본질적으로 무관한 경우 (= 데이터 처리의 모든 결정을 데이터만 보면서 처리할 수 있을 때) 사용한다.
    - 모든 결정은 데이터가 도착하면서 이뤄지기 때문에, 스트리밍 엔진은 기본적인 데이터 전달 목적으로만 사용된다.
    - 모든 스트리밍 시스템은 시간 무시 처리를 바로 지원한다고 볼 수 있다.
        - 결과의 정확성이 필요한 경우 일관성 보장에 따른 시스템별 차이가 발생한다.
    - 배치 시스템 역시 무한 데이터를 임의의 유한 데이터 셋으로 나누어 독립적으로 처리할 수 있기 때문에 무한 데이터의 시간 무시 처리에 적합하다.

![무한 데이터의 필터링 예. 다양한 종류의 데이터가 → 방향으로 흐르며, 단일 유형의 동질 집합으로 필터링되고 있다. ](/resource/post-format/streaming-systems-01-%EC%8A%A4%ED%8A%B8%EB%A6%AC%EB%B0%8D-101/Untitled%204.png)

무한 데이터의 필터링 예. 다양한 종류의 데이터가 → 방향으로 흐르며, 단일 유형의 동질 집합으로 필터링되고 있다. 

1. 필터링 (filtering)
    1. 시간 무시 처리의 가장 기본적인 형태
    2. 웹 트래픽 로그를 처리하면서 특정 도메인에서 유래하지 않은 모든 트래픽을 필터링하는 경우
        1. 도착한 데이터 확인
        2. 도메인이 속하는지 확인
        3. 그렇지 않을 경우 삭제
        4. 이런 식의 작업은 데이터의 단일 요소에만 의존하기 때문에 데이터 소스가 무한하고 정렬되어 있지 않으며 가변적인 이벤트 시간 왜곡이 있다는 사실과 무관해진다. 

![무한 데이터의 내부 조인 수행. 조인 결과는 두 소스에서 일치하는 요소가 관찰될 때 생성된다. ](/resource/post-format/streaming-systems-01-%EC%8A%A4%ED%8A%B8%EB%A6%AC%EB%B0%8D-101/Untitled%205.png)

무한 데이터의 내부 조인 수행. 조인 결과는 두 소스에서 일치하는 요소가 관찰될 때 생성된다. 

1. 내부 조인(inner join)
    1. 2개의 무한 데이터 소스를 조인할 때 조인의 결과만 중요하다면 시간적인 요소를 고려할 필요가 없다. 
    2. 한 소스로부터 값을 확인하고, 이를 조인을 위해 버퍼링해둘 수 있다. 
    3. 다른 소스로부터 두 번째 값이 도착한 후 조인 결과를 생성한다. 
        1. 이 경우 짝을 만나지 못한 부분 조인 결과를 정리하기 위해 가비지 컬렉터 정책이 필요할 수 있다. 
        2. 얼마 후에 정리할지를 결정하기 위한 시간 개념이 들어간다. 
        3. 완성되지 못한 조인 결과가 적거나 거의 없는 경우 문제가 되지 않는다. 
    4. 내부 조인에서 외부 조인으로 전환하면 데이터의 완결성 문제가 발생한다. 
        1. 한 쪽을 보고 다른 쪽이 올지 안 올지 예측할 수 없다. 
        2. 데이터의 완결을 결정 내리기 위한 타임아웃을 도입한다. 
            1. 시간적인 개념을 포함한다는 의미가 된다. 

### 근사(approximation)

![무한 데이터에 대한 근사 계산. 데이터는 복잡한 알고리즘을 통해 계산되어 원하는 결과와 비슷한 결과 데이터를 생성한다. ](/resource/post-format/streaming-systems-01-%EC%8A%A4%ED%8A%B8%EB%A6%AC%EB%B0%8D-101/Untitled%206.png)

무한 데이터에 대한 근사 계산. 데이터는 복잡한 알고리즘을 통해 계산되어 원하는 결과와 비슷한 결과 데이터를 생성한다. 

1. 근사 알고리즘
    1. Top-N, 스트리밍 k-평균 등
    2. 근사 알고리즘은 무한 데이터를 입력으로 받아 얻고자 하는 결과와 비슷한 결과를 생성한다. 
    3. 장점: 설계상 낮은 오버헤드를 갖고 무한 데이터를 고려해 설계됐다는 점
    4. 단점: 알고리즘 종류가 많지 않고 종종 알고리즘 자체가 복잡하며 근사적인 특성으로 유용성이 떨어질 수 있다. 
2. 보통 설계 자체에 시간적인 요소를 포함한다. 
    1. 일종의 내부 감쇠(build-in decay) 등 
3. 도착하는 순서대로 데이터를 처리하기 때문에 시간적인 요소는 보통 처리 시간을 기준으로 이루어진다. 
    1. 근사 결과에 대한 증명 가능한 오차 범위를 제공하는 알고리즘의 경우에 더 주용하다. 
    2. 오차 범위가 데이터 발생 순서대로 도착한다는 사실을 가정할 경우
        1. 가변적인 이벤트 시간 왜곡이 있는 비순서 데이터에 적용할 때의 오차 범위는 본질적으로 무의미하다. 
4. 본질적으로 시간 무시 처리의 다른 예이기 때문에 사용하기 쉽다. 

### 윈도우

![윈도우 사용 형태. 모든 데이터에 같은 시점으로 적용되는 정렬윈도우와 데이터마다 다른 시점에 적용되는 비정렬 윈도우 사이의 차이를 볼 수 있다. ](/resource/post-format/streaming-systems-01-%EC%8A%A4%ED%8A%B8%EB%A6%AC%EB%B0%8D-101/Untitled%207.png)

윈도우 사용 형태. 모든 데이터에 같은 시점으로 적용되는 정렬윈도우와 데이터마다 다른 시점에 적용되는 비정렬 윈도우 사이의 차이를 볼 수 있다. 

- 간단한 유한 또는 무한 데이터 소스를 가져와 처리를 위해 시간 경계를 따라 유한한 크기의 조각으로 자르는 개념
1. 고정 윈도우(fixed window), 텀블링 윈도우(tumbling window)
    1. 시간을 고정된 크기의 길이로 자른다. 
    2. 전체 데이터 셋에 균일하게 적용되어 정렬 윈도우의 형태를 갖는다. 
    3. 윈도우 구성이 완료되는 시점을 더욱 고르게 분산시키기 위해 데이터마다 시간이 것나도록 윈도우를 적용하기도 한다. 
    4. 결과적으로 비정렬 윈도우 형태를 갖는다. 
2. 슬라이딩 윈도우(sliding window), 호핑 윈도우(hopping window)
    1. 고정 윈도우의 일반화된 형태
    2. 일정한 길이와 일정한 주기로 정의된다. 
        1. 주기가 길이보다 작으면 윈도우 간에 겹치는 부분이 발생한다. 
        2. 주기가 길이와 같으면 고정 윈도우가 된다. 
        3. 주기가 윈도우보다 크다면 시간에 따라 데이터의 일정 비율을 샘플링하는 윈도우가 된다. 
    3. 슬라이딩 윈도우도 성능을 위해 비정렬 형태로 만들 수 있으나 보통 정렬 윈도우 형태를 갖는다. 
3. 세션 (session)
    1. 동적 윈도우(dynamic window) 의 한 예
    2. 일정 타임아웃 시간보다 긴 길이를 갖는 비활성화 간격으로 구분해 일련의 연속된 이벤트들을 묶어서 윈도우를 구성한다. 
    3. 시간적으로 관련되어 있는 이벤트를 묶어서 시간에 따른 사용자의 행동을 분석하기 위해 사용된다. 
    4. 세션은 길이가 미리 정해질 수 없다. 
        1. 세션의 길이는 실제 들어오는 데이터와 직접적으로 연관되어 있다. 
        2. 데이터의 서로 다른 부분에 동일한 윈도우를 적용할 수 없기 때문에 비정렬 윈도우의 전형적인 예이기도 하다. 
- 윈도우는 처리 시간과 이벤트 시간 두 가지 영역에서 모두 사용 가능하다.

![처리 시간에 따라 고정 윈도우 구성. 파이프라인에 도착하는 순서에 따라 데이터가 윈도우로 수집된다. ](/resource/post-format/streaming-systems-01-%EC%8A%A4%ED%8A%B8%EB%A6%AC%EB%B0%8D-101/Untitled%208.png)

처리 시간에 따라 고정 윈도우 구성. 파이프라인에 도착하는 순서에 따라 데이터가 윈도우로 수집된다. 

1. 처리 시간 윈도우(processing time window)
    1. 처리시간으로 윈도우를 구성할 때 시스템은 어느 정도의 처리 시간이 지날 때까지 들어오는 데이터를 버퍼링한다. 
    2. 장점
        1. 단순하다. 
            1. 시간을 기준으로 데이터를 셔플링할 필요가 없기 때문에 구현이 극도로 간단하다. 데이터가 도착하면 버퍼링했다가 윈도우가 마무리 될 때마다 전달하면 된다. 
        2. 윈도우가 완료되는 시점을 판단하기 쉽다.
            1. 시스템이 언제 특정 윈도우의 입력이 모두 도착하는지 알기 때문에 윈도우의 완료 시점을 완벽하게 결정할 수 있다. 
            2. ‘지연된’ 데이터를 다룰 필요가 없다. 
        3. 데이터 소스가 관측되는 시점을 기준으로 정보를 추론하고자 하는 경우에 적합하다. 
            1. 다수의 모니터링을 위한 시나리오의 경우 여기에 속한다. 
    3. 단점
        1. 데이터가 관련된 이벤트 시간을 가지고 있는 경우, 처리 시간 윈도우가 실제 이벤트가 발생한 순서를 제대로 반영해야 한다면, 데이터가 이벤트 시간 순서대로 들어와야 한다. 
            1. 분산된 입력 소스를 감안할 때 현실에서 이벤트 시간 순서가 보장돼 들어오는 데이터는 드물다. 
2. 이벤트 시간 윈도우(event time window) 
    1. 이벤트가 실제 발생한 시간을 반영해 유한한 크기의 조각으로 데이터 소스를 관찰하고자 할 때 사용한다. 
    2. 윈도우의 표준 방식
    3. 2016년 이전에는 대부분의 데이터 처리 시스템이 이에 대해 기본 적인 지원이 부족했다. 
    4. 현재는 프링크, 스파크, 스톰, 에이펙스에 이르기까지 여러 시스템이 기본적으로 이벤트 시간 윈도우를 지원하고 있다. 
    5. 이벤트 시간에 대한 정확성을 제공한다.

    6. 세션 같은 동적인 크기를 갖는 윈도우를 만들 수 있어 고정 윈도우에 걸쳐 있는 세션을 임의로 분할해야 하는 상황을 피할 수 있다.

        이벤트 시간 기준의 세션 윈도우. 데이터는 이벤트가 발생한 시간에 따라 활동이 보이는 구간을 잡는 세션 윈도우로 수집된다. 검은색 화살표는 데이터를 올바른 이벤트 시간 자리에 배치하기 위해 필요한 시간적인 셔플을 의미한다. 
        
    7. 이벤트 시간 윈도우는 실제 윈도우 자체의 길이보다 이를 수집해 구성하는 시간이 길어지는 경우가 많기 때문에 단점을 갖는다.
        1. 버퍼링(buffering)
            1. 윈도우의 수명이 길어짐에 따라 더 많은 데이터를 버퍼링할 필요가 생긴다. 
            2. 영구적인 스토리지는 대부분의 데이터 처리 시스템이 의존하는 리소스 유형 중 가장 저렴한 편에 속한다.
                1. 강한 일관성을 위해 영구적인 상태 보존과 적절한 메모리 캐싱을 제공하는 잘 설계된 데이터 처리 시스템을 사용할 때 큰 문제가 되지 않는다.
            3. 데이터를 집계하는 방식 중 다수는 입력 데이터 모두를 버퍼링할 필요가 없으며, 훨씬 작은 크기의 중간 데이터를 영구적인 상태로 유지해 점진적으로 결과를 낼 수 있다. 
        2. 완결성(completness)
            1. 데이터처리 시스템은 많은 입력 유형에 대해 윈도우 완료를 결정하는 비교적 정확한 휴리스틱을 제공할 수 있다. 
            2. 밀휠, 클라우드 데이터플로우, 플링크 등에서 제공하는 워터마크를 통해 사용이 가능하다. 
            3. 하지만 정확한 완결성이 필요한 경우(결제 등) 사용 가능한 유일한 방법은 파이프라인 생성자가 윈도우 결과를 언제 구체화 할지, 그 결과가 시간에 따라 어떻게 정제될지 표현할 수 있는 방법을 제공해야 한다.
