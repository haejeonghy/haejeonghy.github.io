# 언제/어디서: 처리시간 윈도우

- 처리 시간 윈도우가 중요한 이유
    - 모니터링을 하는 경우 인입되는 데이터스트림을 관측 시점을 기준으로 분석할 필요가 있다.
        - 웹 서비스 트래픽의 초당 쿼리수 등
    - 이벤트가 발생한 시점이 중요한 경우
        - 사용자의 행동 트렌드나 결제, 점수 분석 등
- 윈도우가 엄격히 이벤트 시간 기반으로 동작하는 모델 안에서 처리 시간 윈도우를 사용하는 방법
    - 트리거 trigger 사용
        - 이벤트 시간을 무시하고
            - 이벤트 시간 전범 위에 걸친 하나의 전역 윈도우를 사용하는 경우
        - 처리 시간 축에서 해당 윈도우의 스냅샷을 제공하는 트리거를 사용한다.
        - 다단 파이프라인의 각 단계마다 독립적으로 처리 시간 윈도우를 자르게 된다.
            - 한 단계에서 N번째 윈도우에 나왔던 데이터가 이후 단계에서는 N-1번째나 N+1번째 윈도우에 나올 수 있다.
    - 인입 시간 ingress time 사용
        - 데이터가 인입하는 시간을 이벤트 시간으로 설정
        - 일반적인 이벤트 시간 윈도우를 그대로 사용한다.
            - 스파크 스트리밍 1.x가 사용하는 방식
        - 데이터가 N번째 윈도우에 속한다면
            - 워터마크
                - 클라우드 데이터플로우의 경우
            - 마이크로배치 경계 microbatch boundary
                - 스파크 스트리밍의 경우
            - 혹은 엔진 레벨에서 쓰이는 조정 방식에 의해 단계 사이 동기화가 이루어져
            - 한 파이프라인 안에서 계속 같은 윈도우에 남아있음.
    - 거의 유사하지만 다단 파이프라인 multi-stage pipeline을 사용할 때 미묘한 차이를 만들 수 있음
    - 처리 시간 윈도우의 단점
        - 윈도우의 내용이 입력 데이터의 관측 순서에 따라 달라진다.
- 정확히 같은 이벤트를 다른 관측 순서로 배치한 입력셋

## 트리거를 사용한 처리 시간 윈도우

- 처리 시간 윈도우가 트리거를 사용하도록 할 때 고려해야 할 세 가지
    - 윈도우
        - 이벤트 시간 윈도우로 처리 시간 윈도우를 흉내내기 위해 전역 이벤트 시간 윈도우를 사용한다.
    - 트리거
        - 처리 시간 윈도우의 원하는 크기에 맞춰 처리 시간 도메인에서 주기적으로 트리거를 작동시킨다.
    - 누적
        - 무시 모드를 사용해 윈도우 패널 간의 독립성을 확보해 윈도우 패널이 마치 독립적인 처리 시간 윈도우처럼 동작하게 만든다.
- 서로 다른 순서를 갖는 경우에 이를 적용한 결과
    
    [같은 입력의 두 관측 순서에 트리거를 사용한 처리 시간 윈도우를 적용한 결과](/resource/post-format/streaming-systems-04-%EA%B3%A0%EA%B8%89-%EC%9C%88%EB%8F%84%EC%9A%B0/stsy_0403.mp4)
    
    같은 입력의 두 관측 순서에 트리거를 사용한 처리 시간 윈도우를 적용한 결과
    
    - 이벤트 시간 패널을 사용해 처리 시간 윈도우를 흉내내고 있기 때문에
        - 윈도우는 처리 시간 축을 따라 표시되고
        - 가로축 대신 세로축에서 그 너비를 측정할 수 있다.
    - 처리 시간 윈도우는 데이터의 관찰 순서에 영향을 받기 때문에
        - 이벤트가 사실상 같은 시간에 발생했더라도 윈도우의 각 결과는 관찰 순서에 따라 달라진다.
        - 왼쪽의 경우 12, 18, 18을 얻었지만 오른쪽에서는 7, 36, 5를 얻게 된다.

## 인입 시간을 사용한 처리 시간 윈도우

- 마지막으로 이벤트 시간을 인입 시간으로 설정해 처리 시간 윈도우를 구현하는 경우
    - 시간 변환
        - 데이터가 도착할 때 각 이벤트 시간을 도착 시간으로 덮어써야 한다.
        - 빔에서는 outputWithTimestamp 메소드를 사용해 입력 요소의 시간을 현재 시간으로 설정하는 새로운 DoFn을 제공함으로써 이를 표현할 수 있다.
    - 윈도우
        - 표준 이벤트 시간 고정 윈도우를 사용한다.
    - 트리거
        - 도착 시간을 사용한다는 것은 완벽한 워터마크를 사용할 수 있음을 의미
        - 워터마크가 윈도우의 끝을 통과할 때 암묵적으로 실행되는 기본 트리거를 사용한다.
    - 누적 모드
        - 윈도우당 하나의 결과만 얻게 되므로 누적 모두의 영향은 없다.
- 스트리밍 엔진에서 이를 실행한 결과
    
    [같은 입력의 두 처리 시간 순서에 대해 도착 시간을 사용한 처리 시간 윈도우를 적용한 결과](/resource/post-format/streaming-systems-04-%EA%B3%A0%EA%B8%89-%EC%9C%88%EB%8F%84%EC%9A%B0/stsy_0404.mp4)
    
    같은 입력의 두 처리 시간 순서에 대해 도착 시간을 사용한 처리 시간 윈도우를 적용한 결과
    
    - 데이터가 도착하면 이벤트 시간은 인입 시간으로 덮어쓰이고, 이상적인 워터마크를 따라 놓이게 된다.
    - 다른 처리 시간 윈도우와 마찬가지로 입력 순서가 변할 때 다른 결과를 얻게 된다.
    - 윈도우는 이벤트 시간 영역, 가로축을 따라 표시된다.
        - 실제로도 이벤트 시간 윈도우는 아니고, 데이터의 원래 시간을 지우고 파이프라인에 의해 관측된 시간으로 대체해
        - 이벤트 시간을 처리 시간으로 대체 사용하는 것
    - 워터마크 덕분에 트리거가 이전 처리 시간 윈도우 예제와 동일한 시간에 동작한다.
        - 출력 값도 이전 예와 동일
    - 도착 시간을 사용하는 경우 완벽한 워터마크를 사용할 수 있음
        - 실제 워터마크가 이상적인 워터마크와 일치하도록
- 이벤트 시간 윈도우는 순서에 독립적이며 처리 시간 윈도우는 그렇지 않다.
    - 이벤트가 실제로 일어난 시점이 중요하다면
    - 유의미한 결과를 얻기 위해 반드시 이벤트 시간 윈도우를 사용해야 한다.

# 어디서: 세션 윈도우

- 세션
    - 데이터 기반의 동적 윈도우
    - 데이터 상의 활동 기간 activity period 을 잡아내는 특수한 형태의 윈도우
- 활동 기간은 일정 길이의 비활동 간격 inactivity gap 이 보일 때 끝나게 된다.
- 사용자의 일정 행동과 관련된 시간 동안 해당 사용자의 활동을 관찰할 수 있는 방법을 제공한다.
    - 데이터 분석에 유용하다
    - 세션 내 활동의 상호 관련성을 볼 수 있다.
    - 세션 길이에 따른 참여도에 대한 추론 등을 가능하게 한다.
- 세션의 특성
    - 데이터 기반 윈도우 data-driven window 에 해당한다.
        - 윈도우의 위치와 크기가 고정이나 슬라이디 윈도우처럼 미리 정의된 일정 패턴에 기반하지 않고
        - 입력 데이터만으로 결정된다.
    - 비정렬 윈도우 unaligned window 의 특성을 보인다.
        - 윈도우가 모든 데이터에 같은 형태로 적용되지 않고
        - 데이터의 특정 부분 집합에 적용되는 형태를 보인다.
        - 모든 데이터에 같은 형태로 적용되는 고정, 혹은 슬라이딩 윈도우와 대조된다.
- 경우에 따라 한 세션 내 데이터마다 미리 공통 아이디 같은 걸로 표시해둘 수 있다.
    - 서비스 품질 정보를 위해 하트비트 핑 heartbeat ping 을 수행하는 비디오 재생기가 있다고 할 때
    - 재생 중인 동안 일정한 세션 아이디를 핑 정보에 미리 표시를 해두는 것이 가능하다.
        - 나중에 해당 아이디를 키로 사용해 세션을 구성하기가 훨씬 수월해진다.
- 일반적인 경우 실제 세션을 미리 알 수 없는 경우라면
    - 시간에 대한 정보만 가지고 데이터의 위치를 통해 세션을 구성해야 한다.
    - 비순서 out-of-order 데이터를 다룰 때 이 부분이 어렵다.
- 독립적인 데이터가 60분의 간격 타임아웃 gap timeout을 갖는 세션 윈도우로 묶이는 모습
    
    ![병합 전의 프로토 세션 윈도우와 병합 후의 결과](/resource/post-format/streaming-systems-04-%EA%B3%A0%EA%B8%89-%EC%9C%88%EB%8F%84%EC%9A%B0/stsy_0405.png)
    
    병합 전의 프로토 세션 윈도우와 병합 후의 결과
    
    - 각 데이터는 자신만의 60분짜리 윈도우에서 시작한다.
        - 데이터 하나마다 배정되는 윈도우를 proto-session이라고 부른다.
    - 겹치는 프로토 세션을 병합해 3개, 2개의 데이터를 갖는 더 큰 2개의 세션 윈도우를 생성할 수 있다.
    - 세션 지원의 핵심은 작고 서로 겹치는 윈도우를 병합해 최종 세션 윈도우를 생성한다는 것
        - 각각의 작은 윈도우는 단일 데이터를 담고 있고
        - 한 세션 안에서 데이터는 미리 약속된 타임아웃보다 짧은 비활동 간격으로 분리된다.
            - 비순서 out-of-order 형태로 들어오는 데이터를 보게 되더라도
            - 각 데이터를 위한 개별 윈도우를 구성한 후 병합해 최종 세션을 구성할 수 있다.
- 세션 타임아웃을 1분으로 볼 때
    
    ![계산하고자 하는 최종 세션의 모습](/resource/post-format/streaming-systems-04-%EA%B3%A0%EA%B8%89-%EC%9C%88%EB%8F%84%EC%9A%B0/stsy_0406.png)
    
    계산하고자 하는 최종 세션의 모습
    
    - 각 세션은 사용자 활동이 모여있는 부분을 포함
    - 1분 이내의 시간 차이를 보이는 이벤트가 세션 안에 모여 있게 된다.
- 이벤트를 보며 시간 흐름에 따라 윈도우가 병합되고 세션을 구성하는 모습
    - 철회를 지원하는 조기 지연 코드를 가져와 1분 간격의 타임아웃 대신 세션을 구성하는 형태로 수정한 결과
        
        [스트리밍 엔진에서 세션 윈도우와 철회를 지원하는 조기/지연 점화](/resource/post-format/streaming-systems-04-%EA%B3%A0%EA%B8%89-%EC%9C%88%EB%8F%84%EC%9A%B0/stsy_0407.mp4)
        
        스트리밍 엔진에서 세션 윈도우와 철회를 지원하는 조기/지연 점화
        
    - 과정은
        - 값으로 5를 갖는 첫 데이터가 도착하면
            - 해당 데이터의 이벤트 시간에 시작하는 단일 프로토 세션으로 배치되고
            - 세션 갭 지속 기간이 확장되기 시작한다.
                - 데이터가 발생한 시점으로 부터 1분 동안 확장되는 식
                - 이 윈도우와 겹치는 다른 윈도우를 만나게 되면 같은 세션으로 병합된다.
        - 두 번째 데이터인 7이 나오면
            - 처음의 5와 겹치지 않기에 자신만의 프로토 세션으로 들어간다.
        - 이 과정에서 첫 번째 윈도우의 워터마크를 통과하고
            - 그 결과 5가 구체화된다.
            - 이후 예측 결과 로서 두 번째 윈도우 역시 구체화된다.
        - 이후 3, 4가 오고 이들의 프로토 세션은 겹친다.
            - 이 둘은 병합되고 조기 트리거에 의해  7의 결과를 갖는 단일 윈도우가 생성된다.
        - 8이 도착하면
            - 이전의 7을 갖는 두 윈도우와 겹치고
            - 셋이 병합돼 22를 갖는 하나의 세션을 구성한다.
            - 워터마크가 해당 세션의 끝을 지나면
            - 이전의 7 값을 갖는 두 윈도우가 철회되고 22를 겨과로 갖는 새 세션을 구체화한다.
        - 9가 늦게 도착하면
            - 비슷한 과정이 이루어져서 5와 22를 갖는 세션과 병합돼 36이라는 세션이 구성된다.
            - 5, 22값의 두 윈도우가 철회되고
            - 지연 트리거에 의해 36이 생성된다.
- 세션은 강력한 도구이다.
    - 스트리밍 처리의 복잡성이 별개의 구성 가능한 조각으로 분리해 표현하는 모델을 통해 쉽게 설명될 수 있다.

# 어디서: 커스텀 윈도우

- 빔에서 커스텀 윈도우 전략은 두 가지로 구성된다.
    - 윈도우 배정 window assginment
        - 각 요소를 초기 윈도우로 배치하는 작업을 의미한다.
        - 극단적으로 각 요소를 개별 윈도우로 배치하는 것이 가능 - 매우 강력하다.
    - 생략 가능한 윈도우 병합 window merge
        - 윈도우가 그룹핑 시점에 병합될 수 있도록 한다.
        - 시간에 따라 윈도우가 특정 형태로 변화해 갈 수 있으며 이미 세션을 다룰 때 살펴볼 수 있다.

## 고정 윈도우의 변형

- 고정 윈도우 구현은 매우 단순하다
    - 배정
        - 요소가 타임스탬프와 윈도우 크기 및 오프셋 매개변수에 맞춰 적절한 고정 윈도우로 배치된다.
    - 병합
        - 해당 없음
- 비정렬 고정 윈도우
    - 기본 고정 윈도우 구현의 한 가지 특성은 윈도우가 모든 데이터에 일관되게 적용된다는 것
        - 팀 간의 비교 등을 하고자 하는 경우 이런 정렬 방식은 매우 유용항ㄹ다.
            - 눈에 잘 띄지 않는 비용을 치뤄야 한다.
    - 병렬로 들어오는 같은 데이터 셋에서 다른 2개의 키를 보게 되는 케이스
        
        [정렬된 고정 윈도우](/resource/post-format/streaming-systems-04-%EA%B3%A0%EA%B8%89-%EC%9C%88%EB%8F%84%EC%9A%B0/stsy_0408.mp4)
        
        정렬된 고정 윈도우
        
        - 모든 키에 대해 윈도우가 정렬되기에 두 키의 결과 역시 정렬된다.
        - 워터마크가 윈도우 끝을 지날 때마다 N개의 패널이 구체화된다.
        - N값이 2에 불과하지만 수백만개로 급격하게 증가하면 시스템에 부하가 걸린다.
    - 두 팀 간의 점수 비교처럼 윈도우 간의 비교가 필요한 경우가 아니라면
        - 윈도우가 완료 되는 시점을 분산시켜 부하를 낮출 수 있다.
            - 이를 통해 시스템은 더욱 예측 가능한 상태가 되고
            - 부하가 최대로 걸리는 경우를 대비하기 위한 프로비저닝 provisioning 요구 사항을 줄일 수 있다.
        - 대부분의 시스템에서 비정렬 고정 윈도우는 시스템이 지원해줘야 사용할 수 있다.
    - 커스텀 윈도우 지원이 있다면 기본 고정 윈도우 구현을 약간만 수정해 비정렬 고정 윈도우를 지원할 수 있도록 할 수 있다.
    - 같은 키를 갖는 데이터 요소를 묶어주는 윈도우는 계속 보장하되 다른 키 간의 정렬 제한을 완화시키고자 하는 것이 목적
    - 같은 키를 갖는 데이터 요소의 윈도우는 정렬된다.
        - 다른 키를 갖는 데이터 요소 윈도우는 정렬되지 않으며
        - 키 간의 비교를 의미없게 만드는 대신 윈도우가 완료될 때 발생하는 부하를 분산시킬 수 있다.
        - 이전과 동일한 데이터셋을 대상으로 고정 윈도우가 다르게 정렬되는 모습을 확인할 수 있다.
            
            [비정렬 고정 윈도우](/resource/post-format/streaming-systems-04-%EA%B3%A0%EA%B8%89-%EC%9C%88%EB%8F%84%EC%9A%B0/stsy_0409.mp4)
            
            비정렬 고정 윈도우
            
            - 여러 키에 대해 동시에 여러 패널이 생성되는 경우가 사라짐
                - 대신 패널이 좀 더 고르게 퍼져서 도착
                    - 적용하고자 하는 요구사항이 만족한다면 한 차원의 이득으로 교환하는 예
                    - 이런 유연함은 대량의 데이터를 최대한 효율적으로 다루고자 할 때 중요하다.
- 요소 / 키 단위 고정 윈도우
    - 클라우드 데이터플로우의 초기 사용자를 위해 나왔던 것
    - 회사는 고객을 위해 분석 데이터를 생성하는 데 개별 고객이 매트릭을 모을 때 사용할 윈도우 크기를 설정할 수 있도록 한다.
        - 각 사용자가 고정 윈도우의 크기를 정의할 수 있도록 해주려는 것
    - 허용해줄 윈도우 크기의 종류가 고정돼 있다면 이런 사례를 지원하는 것은 어려운 일이 아니다.
        - 30분, 60분, 90분 단위의 고정 윈도우를 선택할 수 있게 옵션으로 지원
        - 각 옵션마다 분리된 파이프라인을 돌리는 방법 가능
            - 제공하는 옵션 종류가 늘어날수록 금방 비현실적인 방법이 된다.
            - 진짜 임의의 윈도우 크기를 지원하는 것은 사실상 불가능하다.
            - 고객이 처리하는 각 데이터가 이미 필요로 하는 윈도우 크기를 메타데이터로 포함하고 있음
                - 기존 고정 윈도우 구현을 약간 수정해 임의의 사용자별 고정 윈도우 크기를 지원할 수 있다.
                - 각 요소는 포함된 메타데이터에 기술된 크기에 맞춰 적절한 크기의 고정 윈도우로 배치된다.
                    
                    [키 단위 커스텀 크기 고정 윈도우](/resource/post-format/streaming-systems-04-%EA%B3%A0%EA%B8%89-%EC%9C%88%EB%8F%84%EC%9A%B0/stsy_0410.mp4)
                    
                    키 단위 커스텀 크기 고정 윈도우
                    
                    - 키 A를 위한 요소는 2분 크기 윈도우를, 키 B의 경우에는 1분 크기의 윈도우를 갖게 된다.

## 세션 윈도우의 변형

- 커스텀 윈도우의 유용함 강조
- 세션 윈도우는 고정 윈도우보다는 복잡하고 그 구현은 다음과 같이 구성된다.
    - 배정
        - 각 요소는 요소의 타임스탬프에 시작해 비활동 간격이 보일 때까지 확장되는 프로토세션 윈도우로 위치된다.
    - 병합
        - 이후 묶는 과정에서 모든 윈도우가 정렬된 후 겹치는 윈도우끼리 병합된다.
- 유한 세션
    - 시간이나 요소의 개수 같은 조건을 정해 일정한 크기 이상으로 커지지 않는 세션을 의미한다.
    - 요구사항 충족을 위해 사용되거나 스팸 데이터 방지 등에 쓰일 수 있다.
    - 변형할 수 있는 형태의 한계를 생각하면 유한 세션을 위한 깔끔한 API를 제공하기가 쉽지 않다.
        - 더 실용적인 접근 방법은 사용자가 스스로 필요로 하는 커스텀 윈도우 로직을 구현할 수 있도록 해주는 것
    - 
        
        [길이가 3분으로 제한된 유한 세션](/resource/post-format/streaming-systems-04-%EA%B3%A0%EA%B8%89-%EC%9C%88%EB%8F%84%EC%9A%B0/stsy_0411.mp4)
        
        길이가 3분으로 제한된 유한 세션
        
    - 대부분의 시스템이 커스텀 윈도우를 지원하지 않는 상황에서 제약 없는 세션 구현만 지원하는 시스템상에서 이러한 내용을 구현하는 것은 쉽지 않다.
        - 유일한 수단은 생성된 세션을 지켜보다가 길이 제한을 초과하는 경우 잘라내도록 세션을 묶어내는 로직 작성 필요
        - 세션을 구성하면서가 아닌 사후에 분해할 방법이 필요하다.
            - 증분 취합 incremental aggregation의 이득을 포기하는 결과가 발생하게 된다.
            - 세션이 잘리기 전에 이미 완전한 크기로 구성되기 때문에 스팸 데이터 방지 시에 세션 크기를 제한함으로써 얻을 수 있는 이점이 사라진다.

## 한 크기로는 충분치 않다

- 현재까지는 커스텀 윈도우에 대한 지원이 충분하지 않지만
    - 대량의 데이터를 가능한 효율적으로 다뤄야 하는 복잡한 실제 사례에 쓰일 파이프라인을 구축할 때 상당한 유연성을 제공한다.
