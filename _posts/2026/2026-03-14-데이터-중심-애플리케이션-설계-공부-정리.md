## 들어가며

요즘 트렌드는 **계산 중심**보다 **데이터 중심** 애플리케이션으로 이동하고 있다.

---

## 좋은 시스템의 조건

- **신뢰성** — 기대하는 대로 동작한다. 여기서 중요한 개념이 **Systematic Error(체계적 오류)** 인데, 무작위로 발생하는 오류와 달리 **일관된 방향으로 반복 발생하는 오류**를 말한다. 예를 들어 잘못된 설계 가정이나 결함 있는 로직에서 비롯된 버그가 이에 해당한다. 신뢰성 있는 시스템은 이런 체계적 오류를 설계 단계에서부터 방지해야 한다.
- **확장성** — 증가한 부하에 대해 시스템이 감당할 수 있는 능력

### 소프트웨어 설계 3원칙

- **운용성** — 운영이 원활하게 이루어질 것 → 자동화
- **단순성** — 엔지니어가 이해하기 쉬울 것 → 추상화
- **발전성** — 쉽게 변경할 수 있을 것

## 데이터 모델과 질의 언어

데이터 모델은 그 위에서 동작하는 소프트웨어가 **할 수 있는 일과 없는 일**에 지대한 영향을 미친다.

### 관계형 모델 (SQL)

- 정규화된 구조, 데이터 저장·질의, 비즈니스 데이터 모델에 특화
- **쿼리 옵티마이저**가 실제 접근 경로를 결정해주므로, 애플리케이션에서 접근 경로를 따로 고민할 필요가 없음
- 외래키(Foreign Key)로 관계를 표현

### 문서형 모델 (NoSQL)

- 높은 쓰기 처리량, 뛰어난 확장성
- 스키마 유연성, 저장소 지역성, 질의가 간단한 경우에 유리
- JSON을 활용하면 애플리케이션 객체와 데이터 구조 간 불일치를 줄일 수 있음
- 문서 모델에서 관계는 **문서 참조(Document Reference)** 라고 부름

#### 객체-관계형 불일치 (Object-Relational Mismatch)

- 애플리케이션의 객체 모델과 데이터베이스 모델 사이에 불일치가 존재
- ORM이 보일러플레이트 코드를 줄여주지만, 근본적인 불일치를 완전히 숨길 수는 없음

#### 문서 모델이 적합한 경우

- 애플리케이션 데이터가 문서와 유사한 구조일 때 (일대다 트리 관계, 전체 트리를 한 번에 적재)
- 애플리케이션이 자주 전체 문서에 접근할 때 → **저장소 지역성(Storage Locality)** 으로 성능 이점

#### 문서 모델의 한계

- 중첩 항목을 직접 참조할 수 없음
- **다대일, 다대다 관계**에 취약 — 조인 지원이 미흡하기 때문
- 특정 시점에 발생한 이벤트를 기록하는 분석 애플리케이션에는 적합하지 않을 수 있음

### 스키마 유연성 — 암묵적 스키마

문서 데이터베이스는 스키마가 없는 것이 아니라, **암묵적 스키마**가 있지만 데이터베이스가 이를 강요하지 않는다.

- 임의의 키와 값을 문서에 추가할 수 있고, 읽을 때 특정 필드의 존재를 보장하지 않음
- 예시: 전체 이름을 하나의 필드에 저장하다가 성/이름으로 분리하고 싶다면
  - 문서 DB → 새 필드를 가진 문서를 작성하고, 이전 문서를 읽을 때 처리하는 코드만 추가하면 됨
  - 정적 타입 RDBMS → 스키마 마이그레이션 필요, MySQL의 경우 변경 시 수 분~수 시간의 서비스 중단이 발생할 수 있음
- 모든 레코드가 동일한 구조를 가져야 한다면 스키마는 문서화와 구조 강제를 위한 **유용한 메커니즘**이 됨

> Oracle은 **다중 테이블 색인 클러스터 테이블(Multi-table Index Cluster Table)** 기능으로 관계형 모델에서도 저장소 지역성과 유사한 특성을 제공한다. 이 기능은 서로 관련 있는 여러 테이블의 데이터를 물리적으로 같은 위치(같은 디스크 블록)에 저장해, 함께 조회되는 데이터를 한 번의 I/O로 읽을 수 있게 해준다. 즉, 문서 DB의 지역성 이점을 관계형 DB에서도 누릴 수 있게 하는 방법이다.

### 관계형 vs 문서형의 수렴

관계형 데이터베이스와 문서 데이터베이스는 시간이 지남에 따라 점점 더 비슷해지고 있다. 각자 부족한 부분을 서로 보완해나가고 있기 때문이다.

## 데이터를 위한 질의 언어

### 명령형 언어

특정 순서로 특정 연산을 수행하도록 컴퓨터에 직접 지시한다.

### 선언형 언어 (SQL, 관계 대수 등)

- 목표를 달성하는 방법이 아닌, **알고자 하는 데이터의 패턴**과 조건만 지정
- 더 간결하고 다루기 쉬움
- DB 엔진의 상세 구현이 숨겨져 있어, **질의를 변경하지 않고도 DB 성능을 향상**시킬 수 있음
- SQL은 순서가 달라져도 결과가 같음 → 기능적 제한이 오히려 **자동 최적화 여지**를 넓혀줌
- **병렬 실행에 적합**

### 맵리듀스 (MapReduce)

- 많은 컴퓨터에서 대량의 데이터를 처리하기 위한 프로그래밍 모델
- 질의 로직은 처리 프레임워크가 반복적으로 호출하는 조각 코드로 표현됨
- `map`과 `reduce` 함수는 **순수 함수**여야 하고, 사이드 이펙트가 없어야 함
- 클러스터 환경에서 분산 실행을 위한 저수준 프로그래밍 모델
- MongoDB는 이를 보완하기 위해 **집계 파이프라인(Aggregation Pipeline)** 이라는 선언형 질의 언어를 지원함. 집계 파이프라인은 데이터를 여러 단계(stage)에 걸쳐 순차적으로 변환·집계하는 방식으로, `$match`(필터링) → `$group`(그룹화) → `$sort`(정렬) 같은 단계를 파이프처럼 연결해 사용한다. MapReduce보다 훨씬 간결하고 최적화하기도 쉬움

## 그래프형 데이터 모델

다대다 관계가 복잡하게 얽혀 있을 때 적합한 모델.

그래프는 두 가지 객체로 이루어진다:

- **정점 (Vertex)** — 고유 식별자, 유출 간선 집합, 유입 간선 집합, 속성 컬렉션(키-값 쌍)
- **간선 (Edge)** — 고유 식별자, 꼬리 정점(시작), 머리 정점(끝), 관계 유형 레이블, 속성 컬렉션(키-값 쌍)

그래프 위에서 동작하는 다양한 알고리즘이 있다 (예: 최단 경로 알고리즘 등).

### 속성 그래프 모델과 Cypher

- **Cypher** 는 속성 그래프를 위한 **선언형 질의 언어**
- 수행 방법을 자세히 지정할 필요 없이, 질의 최적화기가 가장 효율적인 전략을 자동으로 선택
- SQL로도 그래프 질의가 가능하지만 **재귀 공통 테이블 식(Recursive CTE)** 을 사용해야 하며, Cypher에 비해 문법이 어렵고 길고 복잡함
- 그래프 모델은 데이터 모델링에 있어 **높은 유연성**을 제공함

### 트리플 저장소 모델

- 그래프 모델과 거의 동등하지만, 다른 용어를 사용해 같은 개념을 설명
- 모든 정보를 **(주어, 서술어, 목적어)** 세 부분 형식으로 저장

### 시맨틱 웹 (Semantic Web)

- 웹사이트가 사람이 읽을 수 있는 정보뿐 아니라, **컴퓨터가 읽을 수 있는 기계 판독 가능 데이터**도 제시하자는 개념
- 현재는 부정적인 평가가 많음

### SPARQL

- **RDF(Resource Description Framework) 데이터 모델**은 웹 자원 간의 관계를 표현하기 위한 W3C 표준으로, 모든 정보를 **(주어, 서술어, 목적어)** 트리플 형태로 나타낸다. 예를 들어 "서울은 대한민국에 위치한다"를 `(서울, 위치한다, 대한민국)` 으로 표현하는 식이다. 트리플 저장소는 이 RDF 모델을 기반으로 데이터를 저장한다.
- **SPARQL**은 이 RDF 데이터 모델을 사용하는 트리플 저장소의 질의 언어
- RDF는 속성과 간선을 구별하지 않고 **서술어만 사용**하여, 속성 매칭에도 동일한 구문을 사용할 수 있음

### 데이터로그 (Datalog)

- 일부 데이터 시스템에서 사용되는 보다 일반화된 질의 언어
- 서술어(predicate)로 작성
- 단계를 나눠 한 번에 조금씩 질의로 나아감
- 새로운 서술어를 정의하는 규칙을 데이터베이스에 전달
- 다른 질의의 규칙을 **결합하거나 재사용**할 수 있어 매우 강력한 접근 방식
